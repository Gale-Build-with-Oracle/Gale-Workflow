Gale Workflowบทที่ 11

CMMI & Docs

ชุดเอกสาร 7 ไฟล์แบนราบสำหรับทุกโปรเจกต์, REQ traceability thread ที่เชื่อมจาก requirement ถึง PR SHA และ doc-sync แบบ batch ที่ต้องรันก่อน UAT และก่อน release

11.1 มาตรฐานเอกสาร: ชุดแบนราบ 7 ไฟล์

Gale Workflow ใช้มาตรฐานเอกสารที่เรียบง่าย: ทุกโปรเจกต์มีชุดเอกสารเดียวกัน 7 ไฟล์ วางในโฟลเดอร์ docs/ แบบแบนราบ ไม่มีการแบ่ง tier ตามขนาดของโปรเจกต์ ไม่มีโฟลเดอร์ซ้อนกัน ไม่มีเอกสารพิเศษสำหรับโปรเจกต์บางประเภท ทุกโปรเจกต์เหมือนกัน

แนวคิดนี้มาจากการสังเกตว่าความซับซ้อนของโครงสร้างเอกสาร (tier, subfolder, cross-reference ระหว่างไฟล์) มักเป็นอุปสรรคมากกว่าประโยชน์ ชุด 7 ไฟล์แบนราบให้ข้อมูลครบถ้วนสำหรับทุก phase ของ SDLC ตั้งแต่ requirement จนถึง deployment และ risk management

การสร้างโครงสร้างเอกสารครั้งแรกใช้ fleet script:

# สร้างโครงสร้างเอกสารครั้งแรก (สำหรับโปรเจกต์ใหม่)
bash ~/ghq/github.com/<your-github-user>/Gale-Framework/scripts/init-project-docs.sh YourProject

# output:
# docs/SRS.md        — Software Requirements Specification
# docs/SDD.md        — Software Design Document (generated)
# docs/UAT.md        — User Acceptance Test plan
# docs/CR.md         — Change Request log
# docs/RISK.md       — Risk register
# docs/UXUI.md       — UX/UI decisions
# docs/TAILORING.md  — project-specific tailoring decisions
# docs/.last-doc-sync — marker สำหรับ doc-sync

11.2 เอกสาร 7 ไฟล์: แต่ละไฟล์เก็บอะไร

ทำความเข้าใจบทบาทของแต่ละไฟล์ช่วยป้องกันการซ้ำซ้อนและทำให้รู้ว่าควรอัปเดตไฟล์ไหนเมื่อใด หลักการสำคัญ: แต่ละ requirement เขียนเพียงครั้งเดียวใน SRS ไฟล์อื่นอ้างอิงด้วย REQ-id ไม่ใช่เขียนซ้ำ

ไฟล์ เก็บอะไร อัปเดตเมื่อ หมายเหตุ
SRS.md Software Requirements — functional + non-functional; แต่ละ REQ-id นิยามไว้ที่นี่เพียงแห่งเดียว doc-sync (batch) Living doc; append-only
SDD.md Software Design — architecture, module map, data flow; generated จาก code + PR history เฉพาะเมื่อ release, audit หรือ onboarding Generated ไม่ใช่ maintained
UAT.md User Acceptance Test plan — test case ต่อ REQ-id, ผลลัพธ์, PR SHA doc-sync ก่อน UAT session REQ-id → code → UAT → PR SHA
CR.md Change Request log — ทุก CR/BUG issue ที่ merge แล้ว, PR number, วันที่ doc-sync (batch) Append-only
RISK.md Risk register — ความเสี่ยง, likelihood, impact, mitigation plan doc-sync เมื่อมี risk ใหม่จาก PR Living doc
UXUI.md UX/UI decisions — design system choices, component patterns, accessibility notes doc-sync เมื่อมี UI change จาก PR Living doc
TAILORING.md Project-specific tailoring — การปรับใช้ CMMI สำหรับ project นี้โดยเฉพาะ ต้นโปรเจกต์ + อัปเดตเมื่อ policy เปลี่ยน อ่าน Gale-Framework/templates/org/TAILORING.md เป็น base

11.3 REQ traceability thread

REQ traceability thread คือเส้นเชื่อมที่รับรองว่าทุก requirement สามารถตามได้จากต้นน้ำถึงปลายน้ำ: REQ-id → code → UAT test case → PR SHA ไม่มี RTM (Requirements Traceability Matrix) แยกต่างหาก เพราะการเชื่อมทั้งหมดฝังอยู่ใน PR description ผ่าน REQ line และใน UAT.md ผ่าน REQ-id column

ตัวอย่าง traceability chain ที่สมบูรณ์:

  • SRS.md: REQ-PROJ-042: ระบบต้องรองรับ rate limiting 100 requests/minute ต่อ user
  • src/auth/rateLimiter.ts: implement rate limiting logic
  • PR #35 description: REQ: REQ-PROJ-042 และ Closes #30
  • UAT.md: row สำหรับ REQ-PROJ-042 ที่มี test case, วันที่ test, ผลลัพธ์ และ PR SHA #35

เมื่อมี audit หรือต้องการรู้ว่า REQ ไหนถูก implement ใน PR ไหน ค้นหาแค่ REQ-id ในไฟล์ 4 แหล่งนี้ ไม่ต้องมีเอกสารพิเศษเพิ่มเติม

11.4 Per-PR REQ line: ทุก feature PR ต้องมี

กฎนี้บังคับสำหรับทุก feature PR: ต้องมีบรรทัด REQ ใน PR description /sop-qa ตรวจไฟล์นี้เป็น gate ระดับ P2 (warning ที่ควรแก้ก่อน merge) บรรทัด REQ มีสองรูปแบบ:

  • REQ: REQ-PROJ-042 สำหรับ feature/fix ที่ตอบสนอง requirement ที่มีอยู่
  • REQ: REQ-PROJ-042, REQ-PROJ-043 สำหรับ PR ที่แก้หลาย requirement
  • REQ: none สำหรับ refactor, dependency update, cosmetic หรืองานที่ไม่เปลี่ยน contract ใดๆ

ข้อห้ามสำคัญ: feature PR ห้ามแก้ไขไฟล์ใดๆ ใน docs/ การแก้ docs ใน feature PR ทำให้ traceability พัง เพราะ doc-sync ออกแบบมาให้ sync เฉพาะ merged PR ที่ผ่าน gate แล้ว ไม่ใช่ sync จาก in-progress feature

# ตัวอย่าง PR description ที่ถูกต้อง

## Summary
- เพิ่ม rate limiting middleware ใน auth layer
- รองรับ 100 requests/minute ต่อ user
- เขียน unit test ครอบคลุม edge cases

REQ: REQ-PROJ-042
Closes #30

## Test Plan
- [x] unit test: rate limiter logic (15 cases)
- [x] integration test: auth middleware chain
- [ ] UAT: manual test ใน staging environment

---

# ตัวอย่าง PR ประเภท refactor (ไม่มี REQ)
## Summary
- Refactor rate limiter ให้ใช้ Redis แทน in-memory store
- ไม่เปลี่ยน API contract หรือ behavior ใดๆ

REQ: none
Closes #31

11.5 doc-sync: batch update ที่ stabilization

doc-sync คือกระบวนการที่รวมการอัปเดตเอกสารทั้งหมดไว้ที่จุด stabilization — ไม่ใช่ per-PR ระบบนี้ป้องกันการ interrupt งาน feature coding ด้วยงานเอกสาร และทำให้เอกสารอัปเดตเป็น batch ที่มีความหมายมากกว่า

doc-sync ต้องรัน 2 กรณีบังคับ: ก่อน UAT session ทุกครั้ง และก่อน release/deploy ทุกครั้ง การ release โดยไม่รัน doc-sync ถือว่าผิด gate ระดับ P1 ซึ่ง /sop-qa จะบล็อก

# marker file ที่ doc-sync ใช้ติดตาม
# docs/.last-doc-sync เก็บ SHA ของ merge commit ล่าสุดที่ sync แล้ว
cat docs/.last-doc-sync
# output: abc1234def5678  (commit SHA)

# doc-sync อ่าน PR ที่ merge ตั้งแต่ SHA นี้
# แล้วอัปเดต SRS, UAT, CR (และ RISK, UXUI ถ้ามีการเปลี่ยนแปลง)
# จบด้วยการ advance marker ไปที่ commit ล่าสุด

# รัน doc-sync (ผ่าน skill)
# /doc-sync

# doc-sync จะ:
# 1. อ่าน merged PRs ตั้งแต่ .last-doc-sync SHA
# 2. spawn haiku agents (1 ตัวต่อ 1 ไฟล์)
# 3. อัปเดต SRS/UAT/CR/RISK/UXUI
# 4. เปิด docs-only PR (low-risk → L1 self-merge)
# 5. advance .last-doc-sync ไปที่ SHA ล่าสุด

doc-sync เปิด PR แยกที่เรียกว่า "docs-only PR" ซึ่ง L1 สามารถ self-merge ได้หลัง quick scrutiny เพราะ PR นี้ไม่แตะ code เลย — มีแต่การอัปเดต docs/ เท่านั้น

11.6 Living docs: append/patch ไม่ regenerate

หลักการ "living docs" หมายความว่า SRS.md, UAT.md และ CR.md เป็นเอกสารที่เพิ่มเนื้อหาตลอดเวลา ไม่ใช่ regenerate ใหม่ทุกครั้ง Gale Workflow ยึด Principle ที่ 1 (Nothing is Deleted): history ของ requirement และ decision ต้องอยู่ในเอกสาร ไม่ใช่ถูก overwrite ด้วยสถานะปัจจุบัน

กฎการ append/patch:

  • SRS.md: เพิ่ม REQ ใหม่ท้ายไฟล์; ถ้า REQ เปลี่ยน ให้ mark เป็น "superseded by REQ-PROJ-NNN" แล้วเพิ่ม requirement ใหม่ ไม่ใช่แก้ requirement เดิม
  • UAT.md: เพิ่ม row ใหม่สำหรับ test ใหม่; อัปเดต status ของ row เดิมได้ (pass/fail/retested)
  • CR.md: append-only เสมอ — ทุก CR/BUG ที่ merge บันทึกเป็น row ใหม่

SDD.md เป็นข้อยกเว้นเดียว: SDD เป็น generated document ที่สร้างจาก code + PR history เฉพาะเมื่อจำเป็น (release, audit, onboarding คนใหม่) SDD ไม่ใช่ living doc ที่ maintain ต่อเนื่อง version เก่าของ SDD ยังอยู่ใน git history ตาม Principle 1

# ตัวอย่างการ append REQ ใน SRS.md (ถูกต้อง)
# ไม่แก้ requirement เดิม — เพิ่มที่ท้าย

## REQ-PROJ-042 (superseded)
> Superseded by REQ-PROJ-042b (2026-05-20)

ระบบต้องรองรับ rate limiting 100 requests/minute ต่อ user (in-memory)

## REQ-PROJ-042b
ระบบต้องรองรับ rate limiting 100 requests/minute ต่อ user
โดยใช้ Redis backend เพื่อรองรับ multi-instance deployment

*Supersedes: REQ-PROJ-042*
*PR: #45 (2026-05-20)*

11.7 haiku สำหรับงานเอกสาร

doc-sync ใช้ haiku model (Claude Haiku) ไม่ใช่ Opus/Sonnet เหตุผลชัดเจน: งานเอกสารเป็นงาน structured formatting — อ่าน PR description, แยก REQ line, append ใน template ที่กำหนดไว้แล้ว งานนี้ไม่ต้องการ reasoning เชิงลึก haiku ทำได้เร็วกว่า 10 เท่าด้วยต้นทุนต่ำกว่ามาก

รูปแบบ: doc-sync spawn haiku subagents แบบ parallel — 1 agent ต่อ 1 ไฟล์เอกสาร (SRS agent, UAT agent, CR agent, RISK agent, UXUI agent) ทำงานพร้อมกัน แล้ว orchestrator รวม output ทั้งหมดเป็น docs-only PR เดียว กระบวนการนี้เสร็จใน 2-3 นาทีสำหรับ sprint ปกติ

กฎ: งานเอกสารทั้งหมดเป็นงาน haiku เสมอ Opus ไม่ควรเขียนเอกสาร CMMI — นี่คือ waste ที่ไม่จำเป็น ถ้าใช้ Codex ให้ hand off งานเอกสารให้ haiku subagent

Model ใช้เมื่อ เหตุผล
Claude Opus/Sonnet (L1/L2) orchestration, routing, scrutiny, complex reasoning ต้องการ reasoning เชิงลึก
OMX/Codex (L3 workers) feature coding, bug fix, test writing execution-focused, isolated
Claude Haiku doc-sync, doc review, structured formatting เร็ว, ถูก, เพียงพอสำหรับงาน structured

11.8 สรุปบทที่ 11

CMMI & Docs ใน Gale Workflow ออกแบบให้เรียบง่ายและ sustainable ชุด 7 ไฟล์แบนราบให้ข้อมูลครบสำหรับทุก phase, REQ traceability thread เชื่อมทุกอย่างโดยไม่ต้องมีเอกสารพิเศษเพิ่ม และ doc-sync แบบ batch ทำให้ทีมมั่นใจว่าเอกสารตรงกับโค้ดก่อน UAT และ release

กฎที่ต้องจำ: feature PR มีแค่ REQ line ไม่มีการแก้ docs; docs อัปเดต batch ด้วย haiku agents; SRS/UAT/CR เป็น living docs ที่ append ไม่ใช่ overwrite; SDD เป็น generated snapshot ไม่ใช่ maintained doc doc-sync ต้องรันก่อน UAT และก่อน release เสมอ

ระบบนี้ทำให้ทีมใช้เวลาน้อยมากกับการ maintain เอกสาร (haiku ทำแทน) แต่ได้ traceability ที่ครบถ้วนและ audit-ready ตลอดเวลา บทถัดไปครอบคลุม Cross-Language support — การทำงานกับ stack ต่างๆ จาก Rust, Python, TypeScript ถึง SQL ใน Gale Workflow

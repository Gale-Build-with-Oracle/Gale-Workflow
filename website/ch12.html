Gale Workflowบทที่ 12

Cross-Language

เลเยอร์ workflow ไม่ผูกภาษา — ตัวอย่าง Python/Go/Rust/TypeScript และ Docker rebuild หลัง merge

12.1 Workflow Layer เป็น Language-Agnostic

หัวใจของ Gale Workflow คือ orchestration layer ที่แยกออกจากภาษาโปรแกรมของโปรเจกต์อย่างสิ้นเชิง ไม่ว่าโปรเจกต์จะเขียนด้วย Python, Go, Rust, TypeScript หรือภาษาอื่นใด กระบวนการ L1 → L2 → L3 ยังคงเหมือนเดิมทุกประการ

สิ่งที่เปลี่ยนไปตามภาษาคือ "คำสั่ง verify" ที่ worker แต่ละตัวรันก่อน commit เท่านั้น ระบบไม่บังคับ toolchain ใด ๆ กับ worker — worker รับผิดชอบ verify slice ของตัวเองด้วยเครื่องมือที่ภาษานั้นใช้ตามธรรมชาติ เช่น pytest สำหรับ Python, cargo test สำหรับ Rust

แนวคิดนี้ทำให้สามารถมีโปรเจกต์ monorepo ที่มีหลายภาษาอยู่ในทีมเดียวกันได้ Worker A รับผิดชอบ backend (Go) ส่วน Worker B รับผิดชอบ scripts (Python) โดยแต่ละคนใช้ verify command ของตัวเอง แล้วผลลัพธ์มารวมกันในขั้นตอน aggregate ของ L2

12.2 Python — pytest และ ruff

โปรเจกต์ Python มักใช้ pytest เป็น test runner และ ruff เป็น linter ที่เร็วและครอบคลุม worker ควรรันทั้งสองขั้นตอนก่อน commit เสมอ เพราะ lint error เล็ก ๆ อาจทำให้ CI ล้มและบล็อก PR ของทีม

หากโปรเจกต์ใช้ virtual environment ควรระบุ path ให้ชัดเจนหรือ activate venv ก่อน ใน worktree แยกต่าง ๆ ควรสร้าง venv ใหม่หรือใช้ uv venv เพื่อให้ deps ไม่ชนกันระหว่าง worker 

# Worker verify slice — Python
# 1. ติดตั้ง deps (ถ้ายังไม่ติดตั้ง)
pip install -e ".[dev]"

# 2. Lint ด้วย ruff
ruff check src/ tests/

# 3. Type check (optional แต่แนะนำ)
mypy src/ --ignore-missing-imports

# 4. รัน tests ที่เกี่ยวกับ slice ของตัวเอง
pytest tests/unit/ -v --tb=short

# ถ้า pass ทั้งหมด — commit
git add src/my_module.py tests/test_my_module.py
git commit -m "feat: implement my-module slice"

การรัน pytest เฉพาะส่วนที่ตัวเองแก้ (ไม่ใช่ทั้งหมด) ช่วยประหยัดเวลา แต่ L2 orchestrator ควรรัน full test suite ในขั้นตอน aggregate verify ก่อนเปิด PR เพื่อความมั่นใจ

12.3 Go — go build และ go test

Go มีคำสั่ง build และ test ที่ค่อนข้าง standard ไม่ซับซ้อน go build ./... ตรวจ compile error ทุก package ใน module go test ./... รัน unit tests ทุกตัวในทีเดียว

ข้อดีของ Go คือ compile เร็วและ deterministic ถ้า build ผ่าน logic ส่วนใหญ่มักถูกต้อง worker ที่ทำงานกับ Go slice ควร verify ด้วยทั้งสองคำสั่งเสมอ โดยไม่ข้าม 

# Worker verify slice — Go
# 1. ตรวจ compile ทุก package
go build ./...

# 2. ตรวจ vet (common bugs + suspicious constructs)
go vet ./...

# 3. รัน tests พร้อม race detector
go test -race ./...

# 4. Format check (ถ้าต้องการ enforce)
gofmt -l . | grep -q '.' && echo "FORMAT ERROR" && exit 1

# ถ้า pass ทั้งหมด — commit
git add internal/handler/
git commit -m "feat: add handler slice for issue-N"

flag -race ใน go test เปิด race detector ซึ่งสำคัญมากสำหรับ concurrent code ใน Go ควรเปิดไว้เสมอในขั้นตอน verify แม้จะทำให้ช้าลงเล็กน้อย

12.4 Rust — cargo build และ cargo test

Rust มี toolchain ที่เข้มแข็งมาก cargo จัดการทั้ง build, test, lint และ format ในที่เดียว cargo build ตรวจ compile; cargo test รัน unit + integration tests

สิ่งที่พิเศษของ Rust คือ compiler error message มีรายละเอียดมากจนมักบอกวิธีแก้เองได้เลย worker ที่ทำงานกับ Rust slice ควรอ่าน compiler output อย่างละเอียดก่อนสรุปว่าต้องเปลี่ยนอะไร 

# Worker verify slice — Rust
# 1. Build (debug mode เพื่อความเร็ว)
cargo build

# 2. Clippy — linter ที่ครอบคลุมกว่า compiler warnings
cargo clippy -- -D warnings

# 3. รัน tests
cargo test

# 4. Format check
cargo fmt --check

# ถ้าต้องการ build release (เช่น production binary)
cargo build --release

# ถ้า pass ทั้งหมด — commit
git add src/
git commit -m "feat: implement Rust slice for issue-N"

flag -D warnings ใน clippy ทำให้ warning กลายเป็น error — เป็น practice ที่ดีเพราะบังคับให้แก้ทุก warning ก่อน commit แทนที่จะปล่อยทิ้งไว้จนสะสม

12.5 TypeScript/JavaScript — tsc และ bun test

โปรเจกต์ TypeScript/JavaScript มีหลาย runtime ให้เลือก — Node.js, Bun, Deno แต่กระบวนการ verify คล้ายกัน: type-check ด้วย tsc --noEmit ก่อน แล้วรัน test suite

tsc --noEmit ตรวจ type error โดยไม่ emit ไฟล์ output ทำให้เร็วและเหมาะสำหรับ verify step หลังจากนั้นรัน test ด้วย bun test, npm test หรือ vitest แล้วแต่ project setup 

# Worker verify slice — TypeScript (Bun)
# 1. Type check — ไม่ emit output
npx tsc --noEmit

# 2. Lint (ESLint)
npx eslint src/ --ext .ts,.tsx

# 3. รัน tests ด้วย Bun
bun test

# หรือถ้าใช้ npm/vitest:
# npm test
# npx vitest run

# 4. Build check (optional — ตรวจว่า bundle ไม่พัง)
bun build src/index.ts --outdir dist/

# ถ้า pass ทั้งหมด — commit
git add src/feature/
git commit -m "feat: add TS feature slice for issue-N"

สำหรับโปรเจกต์ Next.js หรือ Remix การรัน next build หรือ remix build ใน CI เป็นการ verify ที่ครอบคลุมที่สุด แต่ช้ากว่า tsc เปล่า ๆ ให้ทำใน aggregate step แทน ไม่ใช่ใน worker แต่ละตัว

12.6 ตาราง Build/Test Command ต่อภาษา

ตารางนี้สรุปคำสั่ง verify ที่แนะนำให้ worker ใช้ในแต่ละภาษา L2 orchestrator ควรระบุคำสั่งเหล่านี้อย่างชัดเจนในช่วง briefing เพื่อให้ worker รู้ว่าต้อง verify อะไรบ้าง ก่อนที่จะ commit slice ของตัวเอง

ภาษา Build / Compile Test Lint / Format
Python pip install -e ".[dev]" pytest tests/ -v ruff check src/
Go go build ./... go test -race ./... go vet ./...
Rust cargo build cargo test cargo clippy -- -D warnings
TypeScript tsc --noEmit bun test หรือ vitest run eslint src/ --ext .ts
JavaScript node --check src/index.js npm test eslint src/
Bash/Shell bash -n script.sh bats tests/ shellcheck script.sh

สำหรับโปรเจกต์ที่มีหลายภาษา (polyglot) L2 orchestrator ควรบอก worker แต่ละตัวว่าต้องรันคำสั่งของภาษาใด ห้าม worker รันคำสั่งของภาษาอื่นที่ตัวเองไม่ได้แก้โค้ด — เพราะอาจทำให้ test fail จากสาเหตุที่ไม่เกี่ยวกับ slice ของตัวเอง

12.7 Docker Rebuild หลัง Merge

เมื่อ L1 merge PR ที่มีการแก้โค้ดที่ run อยู่ใน container ขั้นตอนถัดไปคือ rebuild และ restart container นี่เป็นหน้าที่ของ L1 oracle เท่านั้น — L2 และ L3 worker ไม่มีสิทธิ์ rebuild container เพราะ rebuild เป็น deploy action ที่ต้องการ authority ระดับ L1

คำสั่ง rebuild มาตรฐานคือ docker compose build --no-cache ตามด้วย docker compose up -d flag --no-cache บังคับให้ build ใหม่ทั้งหมดเพื่อให้แน่ใจว่าโค้ดที่เพิ่งแก้ถูก include 

# L1 Oracle — Docker rebuild หลัง merge
# 1. Rebuild ทุก service
docker compose build --no-cache

# 2. Restart services
docker compose up -d

# 3. ตรวจสุขภาพทุก service
docker compose ps

# คาดหวัง: State = Up / running สำหรับทุก service
# ถ้ามี service ที่ Exit หรือ Restarting — ดู logs
docker compose logs --tail=50 service-name

# === ถ้า smoke test ล้ม — REVERT ===
# revert merge commit
git revert HEAD --no-edit
git push

# rebuild กลับ last-known-good
docker compose build --no-cache
docker compose up -d
docker compose ps

Smoke test หลัง rebuild คือการตรวจว่า docker compose ps แสดง State เป็น Up ทุก service ถ้ามี service ใดล้ม L1 ต้อง revert merge commit ทันที ไม่ใช่พยายาม hotfix บน container ที่พัง เพราะ history ในระบบ Gale Workflow ถือว่าศักดิ์สิทธิ์ — "Nothing is Deleted" หมายถึง revert ด้วย commit ใหม่เสมอ

การ rebuild ที่สม่ำเสมอและ predictable นี้คือสิ่งที่ทำให้ production environment ของ Gale Workflow ไม่มี "drift" ระหว่างโค้ดใน git กับโค้ดที่ run จริงใน container ทุก deploy trace ได้ถึง PR และ commit ที่เฉพาะเจาะจง

12.8 Meta-rule สำหรับภาษาใหม่และ polyglot aggregate

workflow นี้ไม่จำเป็นต้องจำ catalog ของทุกภาษา สิ่งที่ต้องจำคือ meta-rule: verify ของ worker ทุกภาษาแบ่งเป็นสามแกนเสมอ — compile หรือ typecheck, test, และ lint หรือ static analysis ส่วน aggregate verify ของ L2 คือ consumer-facing build ที่พิสูจน์ว่าชิ้นงานรวมกันใช้งานได้ เช่น web build, package build หรือ docker build

Language Compile / Typecheck Test Lint / Static
Java mvn compile mvn test mvn checkstyle:check
Ruby ruby -c สำหรับไฟล์ที่แตะ หรือผ่าน task หลัก rake rubocop
Swift swift build swift test swiftlint ถ้า repo ใช้

ใน polyglot repo ห้ามให้ภาษาหนึ่งเป็นตัวแทนของทั้งระบบ Worker แต่ละคนรัน verify ของ slice ตัวเองก่อน commit จากนั้น L2 aggregate ต้องรันทั้งชุด verify ที่เกี่ยวข้องและ integration build ที่ consumer เห็นจริง เช่น backend tests + frontend typecheck + docker build สำหรับ service รวม

12.9 สรุปบทที่ 12

Gale Workflow ออกแบบมาให้ orchestration layer เป็น language-agnostic อย่างแท้จริง L1, L2, maw commands และ hook chain ทำงานเหมือนกันไม่ว่าโปรเจกต์จะใช้ Python, Go, Rust หรือ TypeScript สิ่งเดียวที่ปรับตามภาษาคือคำสั่ง verify ที่ worker รันก่อน commit

หลักการสำคัญคือ worker verify slice ของตัวเองก่อนเสมอ — ไม่รอให้ L2 ตรวจ L2 aggregate verify เป็นตาข่ายนิรภัยอีกชั้น ไม่ใช่แทนที่การ verify ของ worker แต่ละภาษามีเครื่องมือที่เหมาะสมอยู่แล้ว — ใช้ให้ถูกต้องตามธรรมชาติของภาษานั้น

สำหรับโปรเจกต์ที่ run ใน Docker container ขั้นตอน rebuild หลัง merge เป็น mandatory สำหรับ L1 smoke test ทุก service หลัง docker compose up -d และ revert ทันทีถ้าล้ม การ trace ที่ชัดเจนระหว่างโค้ด, PR, และ container state คือสิ่งที่ทำให้ system น่าเชื่อถือในระยะยาว

Gale Workflowบทที่ 06

Skills System

กายวิภาคของ skill, profiles, การ routing และการเขียน skill เอง

6.1 Skill คืออะไร

Skill คือ "โปรแกรมเสริม" ที่ agent โหลดเมื่อต้องการความสามารถพิเศษ แทนที่จะฝัง instruction ทุกอย่างลงใน system prompt เดียว skills ถูกแบ่งออกเป็นโมดูลย่อย ๆ ที่โหลดตามความต้องการ

แต่ละ skill มีสามส่วนหลัก: slash command (ชื่อที่ใช้เรียก เช่น /debug-mantra), ความสามารถเฉพาะทาง (วิธีทำที่ถูกต้อง, ขั้นตอน, checklist), และ ความรู้โดเมน (context เฉพาะที่ agent ต้องการสำหรับงานนั้น)

การแยก skill ออกจาก core prompt ทำให้ระบบ composable — agent ตัวเดียวสามารถทำงานได้หลายโดเมน โดยโหลดเฉพาะ skill ที่ต้องการ แทนที่จะต้องมี agent พิเศษสำหรับแต่ละงาน

6.2 Skill File Anatomy: โครงสร้างของไฟล์ Skill

skill ไฟล์หนึ่งแบ่งเป็นสองส่วน: frontmatter (YAML metadata) และ body (Markdown content) frontmatter บอก skill system ว่า skill นี้ชื่ออะไร, ทำอะไรได้, และควร trigger เมื่อไหร่ body คือเนื้อหาที่ inject เข้าไปใน context เมื่อ skill ถูกเรียก 

---
name: debug-mantra
description: |
  Four-mantra debugging discipline. Recite the mantra block verbatim at the
  start of any debugging session, then apply the four steps in order before
  proposing any fix. Trigger on /debug-mantra and proactively whenever
  debugging starts — user reports a bug, says something is broken/failing,
  asks to debug/diagnose/investigate, or pastes a stack trace or error log.
version: "1.0.0"
tier: standard
triggers:
  - /debug-mantra
  - bug report
  - stack trace
  - error log
  - "something is broken"
  - "fails with"
  - investigate issue
do_not_trigger:
  - general code review (use /code-review)
  - performance optimization without error
---

## Debug Mantra — สี่ขั้นตอน

ท่อง mantra ก่อนเสมอ:
1. **Reproduce** — ทำให้ bug เกิดขึ้นซ้ำได้ก่อนแก้
2. **Trace the fail path** — ติดตาม code path จาก input ถึง error
3. **Falsify hypothesis** — พิสูจน์ว่า hypothesis ผิดก่อนเชื่อ
4. **Cross-reference breadcrumbs** — ตรวจสอบ log, commit, PR ที่เกี่ยวข้อง

...body เนื้อหาเพิ่มเติม...

field description มีความสำคัญมากที่สุด — มันเป็นสิ่งที่ agent อ่านเพื่อตัดสินใจว่าจะโหลด skill นี้หรือไม่ description ที่ดีต้องบอกชัดว่า "เมื่อไหร่ trigger" และ "เมื่อไหร่ไม่ trigger" ไม่ใช่แค่บอกว่า skill ทำอะไร

6.3 Profiles & Tiers: core / standard / extra

skills แบ่งเป็น 3 tier ตามความสำคัญและความถี่ในการใช้งาน การแบ่ง tier ช่วยให้ agent ไม่โหลด skill ที่ไม่จำเป็นทุกครั้ง ซึ่งประหยัด context window

Tier โหลดเมื่อ ตัวอย่าง
core ทุก session โดยอัตโนมัติ — skills พื้นฐานที่ขาดไม่ได้ /recap, /forward, /rrr
standard โหลดตาม trigger word หรือ slash command — ใช้บ่อยแต่ไม่ทุก session /debug-mantra, /scrutinize, /sop-maw
extra โหลดเฉพาะเมื่อ context ชัดเจน — specialized หรือ domain-specific /financial, /nwf-sql, /sl-theme

arra-oracle-skills-cli จัดการการติดตั้ง skill ลง directory ของ agent แต่ละตัว คำสั่งหลักคือ oracle-skills install --profile standard สำหรับติดตั้ง standard tier ทั้งหมด และ oracle-skills update สำหรับ sync กับ registry เมื่อมี version ใหม่

6.4 Routing: Description เป็นตัวกระตุ้น

กลไก routing ของ skills ไม่ได้ใช้ keyword matching แบบ hard-code แต่ใช้ agent อ่าน description ของ skill ทุกตัวที่ available แล้วตัดสินใจว่าจะโหลดตัวไหน ดังนั้น description ที่ดีคือ routing ที่ดี

trigger words ใน frontmatter ช่วย hint agent ว่าควรพิจารณา skill นี้เมื่อเจอ phrase ที่ระบุ แต่การตัดสินใจสุดท้ายยังอยู่ที่ agent ทำให้ระบบ flexible — agent สามารถ เลือกไม่ trigger skill ถ้า context บอกว่าไม่เหมาะสม

ตัวอย่าง routing ที่ดี: skill /scrutinize มี description ว่า "Outsider-perspective review of a plan, PR, or code change — questions intent and traces the actual code path" พร้อม trigger "review, audit, sanity-check, second opinion" และ do-not-trigger "general chat, musing" ทำให้ agent รู้ว่าต้องโหลดเมื่อไหร่และไม่ควรโหลดเมื่อไหร่

หลักสำคัญในการเขียน routing คือ asymmetry — "when to use" และ "when NOT to use" ต้องมีทั้งคู่ skills ที่ขาด "do not trigger" มักถูก trigger ผิดที่และทำให้ context ยาวโดยไม่จำเป็น

6.5 SOP Skills: แผนที่ Phase ถึง Skill

Gale Workflow กำหนด SOP skills ที่ต้องโหลดในแต่ละ phase ของ development lifecycle การโหลด SOP ถูกต้องคือความแตกต่างระหว่างงานที่ไปถูกทิศกับงานที่ต้องทำซ้ำ

Phase / สถานการณ์ Skill ที่ต้องโหลด เมื่อไหร่
worktree, team spawn, PR, done/sop-mawทุกครั้งที่ใช้คำสั่ง maw
รับ task จาก human / CR intake/sop-delegationก่อนสร้าง issue / dispatch งาน
Frontend UI/sop-frontendก่อนแตะ HTML/CSS/component
Backend / API/sop-backendก่อนแตะ server code
Database / SQL/your-db-mcp (custom)ก่อนอ่าน schema หรือ query
Project docs (7-doc)/sop-cmmiก่อนเขียน/อัปเดต docs
Quality audit (หลัง build, ก่อน PR)/sop-qaทุก PR บน product repo
Bug fix (MANDATORY ก่อนแก้)/debug-mantraเมื่อมี bug report หรือ error
PR review (ก่อน merge)/scrutinizeL1 ทุกครั้งก่อน merge
หลัง bug fix (MANDATORY)/post-mortemหลัง fix landed ก่อนปิด ticket
สื่อสารกับ management/management-talkเมื่อต้องเขียน exec update
สรุป session/rrrท้าย session ที่มี notable work

หลักการคือ "โหลด SOP ก่อนทำงานในแต่ละ phase" ไม่ใช่หลังจากทำแล้วแล้วค่อยตรวจ เหมือนกับการอ่านคู่มือก่อนประกอบอุปกรณ์ ไม่ใช่อ่านหลังจากประกอบไปแล้วครึ่งทาง

6.6 เขียน Custom Skill: skill-creator และการทดสอบ

การเขียน skill ใหม่ทำผ่าน skill /skill-creator ซึ่งแนะนำ workflow การสร้าง, ทดสอบ, และ eval skill โครงสร้างไฟล์มาตรฐานคือ .claude/skills/<skill-name>.md สำหรับ user-level skill หรือ <repo>/.claude/skills/<skill-name>.md สำหรับ project-level skill 

# สร้าง custom skill สำหรับ project
mkdir -p my-product/.claude/skills

cat > my-product/.claude/skills/my-deploy.md << 'EOF'
---
name: my-deploy
description: |
  Deploy YourProject to staging or production via Docker Compose.
  Use when user says "deploy", "push to staging", "release", or asks to
  restart any service. Do NOT use for local dev setup or database migrations.
version: "1.0.0"
tier: standard
triggers:
  - /my-deploy
  - deploy to staging
  - release to production
  - restart service
do_not_trigger:
  - local development setup
  - database migration
---

## Deploy Checklist

1. ตรวจ branch เป็น main และ CI ผ่าน
2. รัน `/sop-qa` ก่อน deploy
3. `docker compose build --no-cache`
4. `docker compose up -d`
5. `docker compose ps` — ตรวจทุก service เป็น running
6. Smoke test endpoint หลัก

...
EOF

หลังเขียน skill แล้ว ทดสอบ trigger โดยเริ่ม session ใหม่และพูดถึง trigger phrase ถ้า skill โหลดถูก จะเห็น skill content ใน context ถ้าไม่โหลด ให้ตรวจ description ว่า trigger phrase ชัดพอไหม

การทำ eval อย่างเป็นระบบใช้ /skill-creator ที่มี benchmark mode — มันทดสอบ skill กับ prompt หลาย ๆ แบบ แล้ว report ว่า trigger ถูกกี่เปอร์เซ็นต์และไม่ trigger ผิดกี่ครั้ง

6.7 Skills ทำให้ Workflow ไม่ผูกกับ Agent ตัวเดียว

ข้อดีเชิงสถาปัตยกรรมที่สำคัญที่สุดของ Skills System คือ engine independence workflow และ procedure ทั้งหมดอยู่ใน skill files ไม่ใช่ใน agent ตัวใดตัวหนึ่ง ดังนั้นถ้าเปลี่ยนจาก Claude เป็น Codex หรือ engine ใหม่ที่เกิดขึ้นในอนาคต skill ก็ยังทำงานได้

fleet สามารถใช้ Claude วันนี้และ Codex พรุ่งนี้ได้โดยไม่ต้องเขียน SOP ใหม่ เพราะ skill files เป็น Markdown ที่ engine ใดก็อ่านได้ ไม่มีความผูกพันกับ provider

ข้อดี รายละเอียด
Engine independentเปลี่ยน engine ได้โดยไม่เขียน SOP ใหม่
Composableโหลดเฉพาะ skill ที่ต้องการ ประหยัด context
Versionableskill มี version ใน frontmatter, update ผ่าน CLI
Testableeval trigger accuracy ด้วย skill-creator benchmark mode
Shareableskill สาธารณะแชร์ผ่าน maw-plugin-registry

6.8 Routing ไม่ใช่กล่องดำ: description คือสัญญาณ

Skill routing ใน Gale Workflow ไม่ได้มี embedding router แยกต่างหากที่ตัดสินใจลับ ๆ ก่อนถึง agent สิ่งที่เกิดขึ้นคือรายการ skill ที่ available พร้อม description ถูกฉีดเข้า context แล้ว agent อ่าน intent ของผู้ใช้เทียบกับคำอธิบายนั้นโดยตรง ดังนั้น description จึงเป็น routing signal หลัก ไม่ใช่แค่เอกสารประกอบ

ผลที่ตามมาคือ skill ที่ดีต้องเขียน description ให้เหมือน contract: บอกว่าใช้เมื่อไหร่, ไม่ใช้เมื่อไหร่, และผลลัพธ์ที่ skill รับผิดชอบคืออะไร ถ้า description คลุมเครือ agent จะต้องเดา และการเดาคือที่มาของ false positive เช่นโหลด skill deploy ทั้งที่ผู้ใช้แค่ถามเรื่อง local setup

User intent Skill ที่ควรชนะ เหตุผล
"แก้ UI table ให้ responsive"sop-frontendเจาะจง frontend และ accessibility มากกว่า skill ทั่วไป
"sync docs ก่อน UAT"doc-syncมี trigger ชัดว่า update SRS/UAT/CR เป็น batch
"หา repo ที่เคยทำ feature นี้"traceเป็นการ locate project/code ไม่ใช่เรียน codebase ทั้งก้อน

เมื่อ trigger overlap กัน กฎปฏิบัติคือ description ที่ specific กว่าชนะ และ do_not_trigger ใช้ตัด ambiguity ตัวอย่างเช่น skill ด้าน deployment อาจมีคำว่า "restart service" แต่ถ้า prompt คือ "ตั้งค่า local development service" รายการ do-not-trigger ควรกันไม่ให้ skill deploy โหลดผิดบริบท

6.9 เครดิตเจ้าของ Skills (Skills Attribution)

Gale Workflow ใช้ skills หลายชุดเป็นฐานความสามารถของ fleet แต่เราไม่ได้สร้างทุกอย่างขึ้นมาเองตั้งแต่ศูนย์ สิ่งที่เราทำคือ DIY, ปรับกติกา, และประกอบให้เข้ากับรูปแบบ 3-layer ephemeral ของเรา ดังนั้นบทนี้ต้องให้เครดิตเจ้าของต้นฉบับอย่างชัดเจน สอดคล้องกับ Rule 6 และ NOTICE attribution: Oracle ไม่ควรแอบอ้างว่าเป็นเจ้าของงานของคนอื่น

ต้นทาง Skills ที่ fleet ใช้ บทบาทใน workflow
9arm /debug-mantra → /scrutinize → /post-mortem → /management-talk bug-chain skills: เริ่มจาก reproduce และ root cause, ตรวจ outsider review, เขียน RCA, แล้วสื่อสารให้ leadership เข้าใจ
@nazt / Soul-Brews-Studio /recap, /rrr, /dig, /trace, /learn, /awaken, /bud, /talk-to, /team-agents, /forward oracle skills จาก arra-oracle-skills-cli: memory, session continuity, Oracle birth, team coordination, และ handoff
Credit chain: 9arm ให้โครง bug discipline; @nazt / Soul-Brews-Studio ให้ oracle skills foundation; Gale Workflow นำมาปรับเป็น fleet doctrine, hooks, และ ephemeral worktree practice ของเรา

เครดิตนี้สำคัญในเชิงเทคนิคด้วย เพราะมันบอก reader ว่า skill ecosystem ไม่ใช่ magic ที่เกิดจาก agent ตัวเดียว แต่เป็นงานสะสมจากหลายเจ้าของที่ถูก version, fork, และปรับใช้ต่ออย่างมีร่องรอย เมื่อ fleet เปลี่ยน skill ในอนาคต ต้องยังรักษา attribution นี้ไว้เหมือนที่รักษา commit history: append ไม่ลบ, ให้เครดิต ไม่กลบต้นทาง

สรุปบทที่ 06

Skills System คือกลไกที่ทำให้ workflow รู้สึกเหมือน "ระบบที่มีความสามารถเพิ่มขึ้นตามเวลา" แทนที่จะเป็น agent ที่รู้ทุกอย่างตั้งแต่ต้น ด้วยการโหลด skill เฉพาะเมื่อต้องการ ระบบประหยัด context window และลดโอกาสที่ knowledge จาก domain หนึ่งจะรบกวน domain อื่น

กุญแจสำคัญของ skill ที่ดีคือ description ที่บอก agent ทั้ง "เมื่อไหร่ควร trigger" และ "เมื่อไหร่ไม่ควร trigger" โดยเฉพาะส่วน do-not-trigger ซึ่งมักถูกมองข้ามแต่ป้องกัน false positive ได้มาก

ข้อดีเชิงสถาปัตยกรรมที่สำคัญคือ engine independence — workflow อยู่ใน skill files ไม่ใช่ใน agent ทำให้ fleet สามารถเปลี่ยน engine ได้โดยไม่เสีย procedure ที่สร้างมา บทถัดไปจะเจาะลึก maw-js Commands ซึ่งเป็น interface หลักสำหรับควบคุม tmux และ agents

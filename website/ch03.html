Gale Workflowบทที่ 03

ψ Vault

สมองแบบไฟล์ของ Oracle — memory, learnings, retrospectives, traces และ spawn-context ที่ทำให้ความรู้อยู่รอดข้าม session

3.1 ψ คืออะไร — File-Based Brain

ψ (อักษรกรีก "psi") คือสัญลักษณ์ที่ Gale Workflow ใช้แทน external brain ของ Oracle โดยรวม ψ Vault คือระบบไฟล์ที่มีโครงสร้าง เก็บความรู้ บทเรียน และ context ในรูปแบบที่ทั้งมนุษย์และ AI อ่านได้โดยตรงโดยไม่ต้องผ่าน interface พิเศษ

แนวคิดสำคัญคือ "ความรู้ควรอ่านได้ ย้ายได้ และอยู่รอดนานกว่า session" — ถ้าระบบทั้งหมดล่มแล้วคุณยังเปิด ψ/memory/learnings/ ได้ด้วย text editor และอ่านบทเรียนทั้งหมดได้ ψ Vault ไม่ได้ lock ความรู้ไว้ใน database ที่ต้องมี service ขึ้นมา

ψ Vault ทำงานร่วมกับ arra-oracle-v3 MCP server ซึ่งให้ความสามารถ search แบบ hybrid (FTS5 + bge-m3 vector) บน content ของ ψ แต่ root ของทุกอย่างยังคงเป็นไฟล์ธรรมดา ไม่ใช่ black box

กฎ MANDATORY FIRST STEP: ทุก action ใน Gale Workflow เริ่มต้นด้วย oracle_search("relevant query") ผ่าน MCP tool ก่อนเสมอ — ก่อนอ่านไฟล์ ก่อน edit ก่อนตอบ ψ Vault มี learnings สะสมที่ป้องกันการทำผิดซ้ำ การข้ามขั้นตอนนี้คือการไม่ใช้ประโยชน์จาก external brain

3.2 โครงสร้างโฟลเดอร์ ψ

โครงสร้างของ ψ Vault ออกแบบให้แต่ละโฟลเดอร์มีวัตถุประสงค์ชัดเจน ไม่ซ้อนทับกัน การรู้โครงสร้างนี้ช่วยให้ค้นหา learnings ที่ถูกต้องได้เร็ว และรู้ว่าจะเขียนสิ่งใหม่ลงที่ไหน

โฟลเดอร์ เนื้อหา ใช้เมื่อ
ψ/inbox/ Handoffs, notes จาก session ก่อนหน้า, pending items รับส่ง context ระหว่าง session; อ่านตอนเริ่ม session ใหม่
ψ/memory/learnings/ Pattern ที่สังเกตได้, บทเรียนจากการทำงาน, anti-pattern บันทึกหลัง retrospective; search ก่อนตอบทุกครั้ง
ψ/memory/retrospectives/ สรุปหลังจบงานหรือจบ session (AI diary) เขียนหลัง /rrr; อ่านเพื่อ context ของ project ใน session ก่อน
ψ/traces/ หลักฐานตามรอย: repo path, commit hash, issue #, PR # เมื่อต้องตาม trace ว่าโค้ดส่วนนี้เปลี่ยนเมื่อไหร่และทำไม
ψ/active/ สถานะงานที่กำลังดำเนินการอยู่ ติดตาม sprint progress; L2 อัพเดทระหว่างทำงาน
ψ/archive/ Learnings, retrospectives ที่เก่าแต่ยังอ้างอิงได้ Never delete — ย้ายมาที่นี่เมื่อ content เก่าเกิน 6 เดือน

โฟลเดอร์ ψ/ อยู่ใน root ของ oracle repo และถูก gitignore โดยตั้งใจ (เพื่อไม่ให้ personal knowledge ของ oracle ไปปน repo สาธารณะ) แต่ content ข้างในสามารถ commit ด้วย oracle ได้หากต้องการ share knowledge

3.3 Learning Lifecycle — จาก Observation ถึงการนำไปใช้

Learning ใน ψ ไม่ได้เกิดขึ้นเองโดยอัตโนมัติ มันเกิดจาก cycle ที่มีโครงสร้าง: สังเกตสิ่งที่เกิดขึ้นจริง → เขียน learning ลงในไฟล์ → search learning ก่อนตอบครั้งต่อไป → นำ learning ที่เกี่ยวข้องมาใช้ในการตัดสินใจ

การ search ควรทำก่อน action ทุกครั้ง แม้งานจะดูง่าย เหตุผลคือ ψ Vault สะสม learnings จากหลาย session หลาย project และหลาย oracle มีโอกาสสูงมากที่จะมี learning เกี่ยวข้องที่บันทึกไว้แล้ว การไม่ search ก่อนเปรียบเหมือนแก้ปัญหาโดยไม่อ่านบันทึกที่คนก่อนทิ้งไว้

3.4 Memory File Format — Frontmatter และประเภท

Learning file ทุกไฟล์ใช้รูปแบบ Markdown พร้อม YAML frontmatter ที่กำหนดโครงสร้างของ metadata frontmatter ทำให้ search engine สามารถ filter ตาม type, tag หรือ project ได้

---
name: worktree-spawn-race-condition
description: "OMX worker spawn ใน worktree ก่อน codex input-ready ทำให้ brief ไม่ถึง"
metadata:
  type: reference
  tags: [maw, worktree, omx, spawn]
  project: gale-framework
  discovered: 2026-06-13
  related: [[auto-kickoff-mechanism]]
---

## Pattern ที่สังเกตได้
เมื่อ `maw team spawn --wt --engine omx --exec` รัน codex อาจยังไม่พร้อม
รับ input ตอนที่ brief ส่งถึง ทำให้ worker เห็น "submit FAILED" ...

## การแก้ไข
ใช้ auto-kickoff mechanism ที่ maw-js v2.1.0+ ดูแลแทน
spawn ด้วย `--exec --prompt "..."` แล้วรอ confirmation ว่า kickoff delivered

## หลักฐาน
- Commit: abc1234 (maw-js)
- Issue: [[issue-1388-maw-send-alias]]

มี 4 ประเภทหลักของ memory file: user (ความชอบและ preference ของเจ้าของ), feedback (สิ่งที่ได้รับ feedback จากมนุษย์), project (บริบทและ pattern ของแต่ละ project), และ reference (เอกสาร tool หรือ API ที่ค้นพบระหว่างทำงาน) แต่ละประเภทมีน้ำหนักในการ search ต่างกัน

การ link ระหว่าง memory file ใช้ double-bracket syntax [[name]] ซึ่ง arra-oracle-v3 จะ resolve เป็น full path โดยอัตโนมัติ ทำให้สร้าง knowledge graph ของ learnings ที่เชื่อมกันได้

3.5 Retrospectives — AI Diary และ Lessons Learned

Retrospective คือสรุปหลัง session หรือหลังจบงานสำคัญ เขียนผ่าน skill /rrr (ย่อมาจาก "reflection, retrospective, review") มันเป็น "AI diary" ที่บันทึกว่า session นี้ทำอะไร เจอปัญหาอะไร ตัดสินใจอย่างไร และได้บทเรียนอะไร

Retrospective เก็บตาม timestamp ใน ψ/memory/retrospectives/YYYY-MM-DD-slug.md ไม่เขียนทับ (P1: Nothing is Deleted) เมื่อ session ถัดไปเริ่ม Oracle สามารถอ่าน retrospective ล่าสุดเพื่อรู้ว่า session ก่อนจบที่ไหนและทำอะไรไปบ้าง

Retrospective ที่ดีมีส่วนประกอบ: สิ่งที่ทำสำเร็จ, ปัญหาที่เจอและวิธีแก้, การตัดสินใจสำคัญพร้อมเหตุผล, และ lessons learned ที่ควรเขียนลงเป็น learning file แยก ส่วน "lessons learned" นี้สำคัญมาก เพราะมันเป็น input ของ Learning Lifecycle

---
date: 2026-06-14
session: fix-validation-bug-sprint-42
oracle: my-oracle
---

## สิ่งที่ทำสำเร็จ
- แก้ validation bug ใน orders.ts (Issue #42) ด้วย TEAM routing
- worker-1 และ worker-2 ทำงานได้โดยไม่ต้องนัดเจอ
- PR ผ่าน scrutinize ครั้งแรก ไม่มี bounce

## ปัญหาที่เจอ
- maw team status แสดง idle แต่ worker จริงๆ ทำงานอยู่
  → แก้ด้วย maw capture แทน (pattern นี้เขียนเป็น learning แล้ว)

## Lessons Learned
- อย่าเชื่อ maw team status สำหรับ omx worker — ใช้ maw capture
- Brief ที่ดีระบุ filename จริงช่วยให้ worker ไม่ต้องเดา

3.6 Spawn-Context และ Handoff — ส่งต่อ Context ข้าม Session

เมื่อ session กำลังจะจบ แต่งานยังไม่เสร็จ Oracle รัน skill /forward หรือ /handoff เพื่อสร้างไฟล์ใน ψ/inbox/ ที่มี context สำหรับ session ถัดไป ไฟล์นี้รวม: งานที่ค้างอยู่, การตัดสินใจที่ยังไม่ได้ตัดสิน, PRs ที่รอ review, และ context ที่จำเป็นเพื่อ resume งาน

Session ถัดไปเริ่มต้นด้วยการรัน /recap ซึ่งจะอ่าน handoff ใน inbox, ดู git status ปัจจุบัน, และประมวลผลว่า session นี้ควรทำอะไรก่อน ไม่ต้องให้มนุษย์ re-explain context ทั้งหมดใหม่

# ψ/inbox/handoff-2026-06-14-T2200.md
---
type: handoff
from: my-oracle
session-ended: 2026-06-14T22:00:00
---

## งานค้าง
- PR #87 รอ scrutinize และ merge (Issue #42 validation bug)
- PR #88 รอ scrutinize (Issue #43 error logging)
- รัน `maw done sprint-42-window` หลัง merge ทั้งสอง

## Context สำคัญ
- maw team sprint-42 shutdown แล้ว
- worktree อยู่ที่ ~/worktrees/my-product-issue-42
- ทั้งสอง PR ผ่าน lint/build/test แล้ว

## Next Action
1. gh pr view 87 → /scrutinize → merge
2. gh pr view 88 → /scrutinize → merge
3. maw done sprint-42-window

3.7 ทำไมใช้ไฟล์ ไม่ใช่ Database อย่างเดียว

คำถามที่เกิดขึ้นบ่อยคือ "ทำไมไม่เก็บทุกอย่างใน vector database หรือ relational database?" คำตอบมีหลายมิติ ทั้งทางปฏิบัติและทางหลักการ

Debuggability: เมื่อ Oracle ตอบผิดหรือนำ learning ผิดมาใช้ คุณสามารถเปิดไฟล์ที่ Oracle อ้างอิงได้ทันที ถ้า learning อยู่ใน black box database คุณต้องผ่าน query interface ซึ่งอาจไม่ reflect สิ่งที่ model จริงๆ เรียนรู้มา

Human-readable เท่ากับ AI-readable: Gale Workflow ออกแบบให้มนุษย์และ AI อ่าน ψ Vault ได้เหมือนกัน คุณ review learning ของ Oracle ได้ด้วยตาเปล่า เห็นว่ามันถูกต้อง อัพเดทหรือลบถ้าผิด โดยไม่ต้องเชื่อระบบ

Portability: ψ Vault คือโฟลเดอร์ธรรมดา ย้ายข้ามเครื่อง backup ด้วย git หรือ rsync ได้ และไม่ต้องพึ่งพา service ที่อาจหยุดให้บริการ ความรู้อยู่กับคุณตลอดไป ไม่ใช่กับ vendor

ในส่วนของ search ที่ต้องการความเร็วและ semantic matching นั้น arra-oracle-v3 ให้ hybrid search (FTS5 keyword + bge-m3 vector) บนไฟล์ เหมือนกับการมี database layer ไว้ query แต่ source of truth ยังเป็นไฟล์ รายละเอียดของ search engine อยู่ในบทที่ 5

# ตัวอย่างการ search ผ่าน MCP tool (ทำก่อน action ทุกครั้ง)
# ในทาง implementation คือเรียก MCP tool oracle_search

# ผลลัพธ์ที่ได้จะเป็น list ของ learnings พร้อม score
# Oracle นำมาอ่านและใช้ประกอบการตัดสินใจ

# โครงสร้างไฟล์ที่ oracle_search scan:
# ψ/memory/learnings/*.md
# ψ/memory/retrospectives/*.md
# ψ/traces/*.md
# MEMORY.md (index file)

# Index file MEMORY.md ควรอัพเดทหลังสร้าง learning ใหม่:
# ## Recent Learnings
# - [[worktree-spawn-race-condition]] — 2026-06-13
# - [[maw-team-status-unreliable-omx]] — 2026-06-14

3.8 ทำไมต้อง Hybrid: Files + SQLite Index

ψ Vault ไม่ได้เลือก "ไฟล์เท่านั้น" หรือ "database เท่านั้น" เพราะทั้งสองอย่างแก้ปัญหาคนละด้าน ไฟล์คือ source of truth: อ่านได้ด้วยตาเปล่า, review ผ่าน git ได้, append-only ได้, และรอดแม้ service search จะล่ม ส่วน SQLite, FTS และ vector index คือ recall layer ที่ทำให้ค้นหาความรู้หลายพันไฟล์ได้เร็วพอสำหรับ workflow ประจำวัน

tradeoff นี้สำคัญเพราะ Oracle ต้อง search ก่อน action ทุกครั้ง ถ้าต้องไล่อ่านไฟล์ทั้งหมดด้วยตัวเองทุกครั้ง workflow จะช้าเกินไป แต่ถ้าเชื่อ index เป็น truth เพียงอย่างเดียว ระบบจะกลายเป็น black box ทันทีเมื่อผลค้นหาผิด Gale Workflow จึงให้ index ทำหน้าที่พาไปหาไฟล์ แล้วให้ไฟล์เป็นหลักฐานสุดท้ายที่มนุษย์และ agent audit ได้

ชั้นข้อมูล หน้าที่ Failure mode Mitigation
Markdown files ใน ψ source of truth และประวัติ append-only ค้นช้าเมื่อความรู้โตมาก สร้าง index จากไฟล์ซ้ำได้เสมอ
SQLite และ FTS keyword recall ที่ deterministic ผลลัพธ์ตรงคำแต่ไม่ตรง intent อ่านไฟล์จริงก่อนใช้เป็นหลักฐาน
Vector index semantic recall ข้ามคำและข้ามภาษา ดึง learning ที่ดูคล้ายแต่ไม่เกี่ยว audit source file แล้ว supersede learning ที่ผิด
Failure mode ที่ต้องรู้: ถ้า oracle_search คืนผลลัพธ์ที่ irrelevant agent อาจเดินตาม learning ผิดได้ mitigation ไม่ใช่การเลิก search แต่คือการอ่านไฟล์ที่อ้างอิง ตรวจว่า context ตรงงานจริง และถ้า learning เก่าผิดให้เขียน learning ใหม่ supersede มัน ประวัติยังอยู่ แต่ป้ายบอกทางถูกแก้ให้คนถัดไปเห็น

สรุปบทที่ 03

ψ Vault คือสิ่งที่ทำให้ Gale Workflow แตกต่างจากการใช้ AI coding agent แบบธรรมดา มันเป็น external brain ที่ความรู้สะสมได้ข้าม session สะสมได้ข้าม oracle และสะสมได้ข้าม project ทั้งหมดในรูปแบบที่อ่านได้ ย้ายได้ และ debug ได้

โครงสร้างไฟล์ที่ชัดเจน (inbox, learnings, retrospectives, traces, active, archive) ทำให้ทั้งมนุษย์และ Oracle รู้ว่าข้อมูลแต่ละประเภทอยู่ที่ไหน Learning Lifecycle ที่วนซ้ำ (สังเกต → เขียน → search → ใช้) เป็น mechanism ที่ทำให้ Oracle ฉลาดขึ้นเรื่อยๆ โดยไม่ต้อง retrain โมเดล

กฎ MANDATORY FIRST STEP ที่ว่า "search ก่อนตอบทุกครั้ง" ไม่ใช่แค่ best practice แต่เป็น implementation ของ P2 (Patterns Over Intentions) — ดูสิ่งที่รู้จริงจาก ψ ก่อนเชื่อว่าจำได้จาก training data บทถัดไป (ch05) จะอธิบาย arra-oracle-v3 search engine ที่ขับเคลื่อน ψ Vault ในเชิงเทคนิคลึกขึ้น

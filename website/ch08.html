Gale Workflowบทที่ 08

Delegation Tiers

สามระดับของการมอบหมายงาน: Arrows (subagents), Squads (TeamCreate), Federation (maw workon) — เลือกให้ถูกระดับ ประหยัดเวลาและป้องกัน context bleed

8.1 สามระดับของการ delegate และเลือกเมื่อไหร่

Gale Workflow แบ่งการมอบหมายงานออกเป็นสามระดับตามระดับ isolation ของ context และ session ที่งานนั้นต้องการ ระดับที่ต่ำกว่า (Arrows) ใช้ทรัพยากรน้อยและตอบสนองเร็ว แต่แชร์ context กับ orchestrator ร่วมกัน ในขณะที่ระดับสูงสุด (Federation) ให้ context แยกสมบูรณ์ด้วย worktree และ session ใหม่ทุกครั้ง

การเลือกระดับที่ผิดคือสาเหตุหลักของปัญหา context bleed, สิ้นเปลือง token และงานล้มเหลว กฎพื้นฐานคือ: ถ้างานต้องการเขียนโค้ดจริงหรือแตะไฟล์ใน product repo ให้ใช้ Federation เสมอ; ถ้าเป็นการอ่าน วิเคราะห์ หรือ research ภายในเซสชันเดียว ให้ใช้ Arrows

8.2 Arrows — in-session subagents

Arrows คือ subagent ที่ Oracle สร้างขึ้นภายใน session เดียวกัน เหมาะสำหรับงานที่ต้องการ context ของ orchestrator อยู่แล้ว เช่น การอ่านไฟล์หลายไฟล์พร้อมกัน, วิเคราะห์ผลลัพธ์, ตรวจสอบโค้ด หรือค้นหาข้อมูล Arrows ทำงานเร็วเพราะไม่ต้องสร้าง session ใหม่หรือ worktree แยก

ข้อจำกัดที่สำคัญที่สุดของ Arrows: ห้ามใช้ Arrows เขียน feature code เด็ดขาด เหตุผลคือ Arrows แชร์ context กับ orchestrator ทั้งหมด — ถ้า Arrow เขียนโค้ดและเกิดข้อผิดพลาด มันจะปนเปื้อนใน context ของ L1 โดยตรง ซึ่งนำไปสู่ปัญหา #157 (orchestrator รับงานโค้ดตรงแทนที่จะ delegate ให้ L3 OMX workers)

การใช้ Arrows ที่ถูกต้อง:

  • อ่านไฟล์หลายไฟล์คู่ขนาน (Read ไฟล์ config, schema, SRS พร้อมกัน)
  • วิเคราะห์ diff ของ PR ก่อน scrutinize
  • ค้นหาข้อมูลใน ψ Vault หรือ Oracle knowledge base
  • สร้าง summary หรือ review ที่ไม่แตะ codebase โดยตรง
  • รัน bash command อ่านอย่างเดียว เช่น git log, grep
# ตัวอย่างการใช้ Arrows อ่านไฟล์คู่ขนาน (ใน Claude Code session)
# Oracle L1 สั่ง subagent อ่านไฟล์หลายไฟล์พร้อมกัน
# ไม่ต้องสั่งทีละไฟล์ — อ่านได้พร้อมกันในครั้งเดียว

# ถูกต้อง: ใช้ Arrows เพื่อ research
Read: src/auth/middleware.ts
Read: docs/SRS.md
Read: .maw/strategy.json

# ผิด: ใช้ Arrows เพื่อเขียนโค้ด feature (ห้าม)
# Write: src/auth/newfeature.ts  <-- ไม่ทำแบบนี้

8.3 Squads — TeamCreate และ team-agents

Squads คือการใช้ skill /team-agents หรือ TeamCreate เพื่อสร้างทีม agent ที่ประสานงานกันภายใน session เดียวกันผ่าน TaskList และ SendMessage Squads เหมาะสำหรับงาน in-session ที่ต้องการแบ่งความรับผิดชอบชัดเจน แต่ยังอยู่ใน context ชุดเดียวกัน เช่น การทำ parallel research จาก source หลายแหล่ง

Squads แตกต่างจาก Federation ตรงที่ไม่ได้สร้าง worktree แยกหรือ session ใหม่ — agent ทุกตัวใน Squad ยังแชร์ underlying session กัน ดังนั้น Squads จึงเหมาะสำหรับ research, review และ planning เท่านั้น ไม่ใช่สำหรับ feature coding ข้าม repo

ในทางปฏิบัติ Gale Workflow ใช้ Squads ในกรณีเหล่านี้:

  • วิเคราะห์ dependency graph ของ issues ที่จะ plan
  • ทำ parallel doc review ด้วย haiku agents หลายตัว (เช่น doc-sync)
  • ค้นหาข้อมูลจาก source หลายแหล่งพร้อมกัน แล้ว synthesize

8.4 Federation — maw workon + ephemeral OMX workers

Federation คือระดับสูงสุด ใช้เมื่องานต้องการ context แยกสมบูรณ์จาก Oracle L1 กระบวนการคือ: Oracle สั่ง maw workon <project> <slug> ซึ่งสร้าง L2 worktree ใหม่ใน directory ของ project นั้นโดยเฉพาะ L2 จะโหลด CLAUDE.md และ AGENTS.md ของ project นั้นโดยอัตโนมัติตั้งแต่ต้น session — ไม่ต้อง recap หรือ dig

สำหรับงาน TEAM, L2 จะ spawn ephemeral OMX workers ด้วย maw team spawn <team> <role> --wt --engine omx --exec แต่ละ OMX worker ได้รับ fresh worktree ของ project เดียวกัน ทำให้มั่นใจว่า code base ที่แต่ละ worker เห็นเป็นสถานะล่าสุด และไม่มี context จาก worker อื่นปะปน

# Federation: สร้าง L2 worktree ใน project directory
maw workon my-product issue-42

# ใน L2 pane: spawn OMX workers แบบ ephemeral
maw team create sprint-42
maw team spawn sprint-42 worker-1 \
  --wt \
  --engine omx \
  --exec \
  --prompt "Issue #42: เพิ่ม authentication middleware. ไฟล์: src/auth/. ทำ unit test ด้วย."

maw team spawn sprint-42 worker-2 \
  --wt \
  --engine omx \
  --exec \
  --prompt "Issue #43: อัปเดต API endpoint สำหรับ user profile. ไฟล์: src/api/users/."

# ตรวจสอบสถานะ
maw panes
maw capture sprint-42-worker-1

เมื่อ worker ทำงานเสร็จ L2 จะ aggregate sub-branches ทั้งหมด → รัน lint/build/test → เปิด PR เดียวที่มีทุก Closes #N → shutdown team → ส่ง DONE-ping ให้ L1 Federation จบด้วยการที่ L1 รัน maw done เพื่อ tear down worktree window

8.5 ตารางเปรียบเทียบสามระดับ

การเลือกระดับที่ถูกต้องขึ้นอยู่กับสามปัจจัยหลัก: ระดับ isolation ของ context, ประเภทของงาน และผลกระทบต่อ codebase ตารางด้านล่างสรุปความแตกต่างที่สำคัญ

มิติ Arrows (subagents) Squads (TeamCreate) Federation (maw workon)
Context isolation ไม่มี — แชร์กับ Oracle บางส่วน — ยังอยู่ใน session เดียวกัน สมบูรณ์ — session และ worktree แยก
Session ใหม่ ไม่ ไม่ ใช่ — fresh session ทุก task
Worktree แยก ไม่ ไม่ ใช่ — per-worker worktree
ใช้ทำอะไร Research, อ่านไฟล์, review Parallel research, doc review, planning Feature coding, bug fix, ทุก product work
เขียน feature code ได้ไหม ไม่ — ห้ามเด็ดขาด ไม่ — ไม่เหมาะสม ใช่ — นี่คือจุดประสงค์หลัก
Speed เร็วมาก เร็ว ช้ากว่า — ต้องสร้าง worktree
Token cost ต่ำ ปานกลาง สูง (แต่ isolated)
ตัวอย่าง อ่าน SRS คู่กับ schema doc-sync สอง docs พร้อมกัน แก้ bug #42 + feature #43

8.6 "subagents enhance, workers isolate" — กฎสำคัญป้องกัน context bleed

หลักการที่สำคัญที่สุดใน Delegation Tiers สรุปได้ในประโยคเดียว: subagents enhance, workers isolate Arrows และ Squads (subagents) ใช้เพื่อเพิ่มความสามารถของ orchestrator ในการรับข้อมูลและวิเคราะห์ — พวกมัน enhance decision-making ของ L1 Federation workers (OMX) ใช้เพื่อแยก execution ออกจากกันอย่างสมบูรณ์ — พวกมัน isolate risk

ปัญหา context bleed เกิดขึ้นเมื่อใช้ Arrows เขียนโค้ด feature เพราะ error messages, partial implementations และ debugging traces ปะปนเข้าไปใน L1 context โดยตรง เมื่อ L1 context เต็มไปด้วยเศษโค้ดที่ยังไม่เสร็จ การตัดสินใจ orchestration จะผิดพลาด นำไปสู่ loop ที่ L1 ลงมือโค้ดเองแทนที่จะ delegate — นี่คือ anti-pattern #157 ที่ Gale Workflow ออกแบบมาเพื่อป้องกันโดยเฉพาะ

กฎปฏิบัติ: ทุกครั้งที่จะใช้ Arrow ถามตัวเองว่า "ถ้า Arrow นี้ล้มเหลว ผลกระทบต่อ L1 context คืออะไร?" ถ้าคำตอบคือ "context จะปนเปื้อน" ให้ยกระดับไป Federation แทน

ข้อสังเกตอีกประการ: ภายใน L2 worktree, L2 เองก็ใช้ subagents (Arrows) สำหรับ research ภายใน — เช่น อ่านไฟล์ก่อน plan, วิเคราะห์ dependency — แต่ L2 ต้องไม่เขียน feature code ด้วยตัวเองถ้า routing เป็น TEAM; ให้ delegate ให้ OMX workers ทำ feature coding แทน

8.7 สรุปบทที่ 08

Delegation Tiers คือหัวใจของการออกแบบ Gale Workflow สามระดับ (Arrows, Squads, Federation) ให้ความยืดหยุ่นในการเลือกระดับ isolation ตามความต้องการของงาน การเลือกถูกระดับช่วยประหยัด token, ป้องกัน context bleed และทำให้ทีม agent ทำงานได้อย่างน่าเชื่อถือ

กฎที่ต้องจำให้ขึ้นใจ: ใช้ Arrows สำหรับ research/review ภายใน session; ใช้ Squads สำหรับ parallel in-session coordination; ใช้ Federation เสมอสำหรับงานที่แตะ codebase จริง หลักการ "subagents enhance, workers isolate" เป็นเส้นแบ่งที่ชัดเจนระหว่างสองโลก

บทถัดไปจะเจาะลึก routing decision ระหว่าง SOLO และ TEAM ซึ่งเป็นคำถามสำคัญที่ L2 orchestrator ต้องตอบก่อนเริ่มงานทุกครั้ง พร้อม strategy.json gate ที่ machine บังคับใช้โดยอัตโนมัติ

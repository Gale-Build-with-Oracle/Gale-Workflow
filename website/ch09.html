Gale Workflowบทที่ 09

SOLO vs TEAM Routing

คำถามเดียวที่ใช้ตัดสินทุกงาน, strategy.json gate ที่ machine บังคับใช้ และ parallel L2s สำหรับ issues ที่อิสระต่อกัน

9.1 คำถามเดียว: routing decision ที่ชัดเจน

ก่อนเริ่มงานทุกงาน L2 orchestrator ต้องตอบคำถามเดียวนี้: "คนเดียวทำงานนี้ได้ใน 30 นาที ใน 1-2 ไฟล์ โดยไม่ต้องทำ research ก่อนไหม?" ถ้าตอบ "ใช่" ชัดเจน — SOLO; ถ้าตอบ "ไม่" หรือ "ไม่แน่ใจ" — TEAM ทันที ไม่มีระดับกลาง

ข้อสำคัญ: TEAM คือ default ไม่ใช่ exception SOLO สงวนไว้สำหรับกรณีที่ชัดเจนจริงๆ เท่านั้น เช่น แก้ typo ใน config, เปลี่ยน environment variable ใน 1 ไฟล์, หรือ fix bug ที่มี stack trace ชัดเจนและแก้ใน function เดียว ถ้ามีข้อสงสัยเลือก TEAM เสมอ

เกณฑ์ที่บังคับให้เป็น TEAM อัตโนมัติ (ข้อใดข้อหนึ่งก็เพียงพอ):

  • ต้องทำ research ก่อน (อ่านหลายไฟล์, ดู schema, เปรียบเทียบ implementation)
  • งานแตะมากกว่า 2 ไฟล์ หรือมากกว่า 200 บรรทัด
  • มีมากกว่า 1 issue
  • ต้องเขียนหรือแก้ test ควบคู่ไปด้วย
  • L2 ไม่แน่ใจว่า SOLO หรือเปล่า

9.2 SOLO route: เส้นทางตรงไม่ซับซ้อน

เมื่อ routing เป็น SOLO, L2 worktree pane ลงมือเขียนโค้ดเองโดยตรง ไม่ spawn worker ใดทั้งสิ้น L2 ประกาศ STRATEGY: SOLO. Justification: <เหตุผล> เขียน .maw/strategy.json แล้วเริ่มโค้ดทันที

SOLO route สิ้นสุดด้วย: commit บน branch → push → เปิด PR 1 ใบที่มี Closes #N → ส่ง DONE-ping ให้ L1 → หยุด L2 ต้องไม่ merge และต้องไม่รัน maw done ตัวเอง ทั้งสองอย่างนั้นเป็นอำนาจของ L1 เท่านั้น

# ตัวอย่าง SOLO route สมบูรณ์
# L2 ประกาศ strategy ก่อน
echo '{"route":"SOLO","justification":"แก้ typo ใน config 1 ไฟล์"}' \
  > .maw/strategy.json

# โค้ดตรง, commit, push
git add config/settings.json
git commit -m "fix: แก้ไข endpoint URL ใน config

Closes #42"

git push -u origin fix/issue-42

# เปิด PR
maw pr

# ส่ง DONE-ping
maw hey my-oracle "DONE: PR พร้อม review: #42 (Closes #42). ขอ /scrutinize + merge + maw done"

# เขียน marker กัน duplicate ping
mkdir -p .maw && touch .maw/done-pinged

# STOP — ห้าม merge เอง

9.3 TEAM route: ephemeral OMX workers

TEAM route เริ่มต้นด้วย L2 ประกาศ STRATEGY: TEAM, เขียน .maw/strategy.json, สร้าง team roster ด้วย maw team create <slug> แล้ว spawn OMX workers ทีละคน แต่ละ worker ได้รับ fresh worktree ของ project นั้น พร้อม brief ที่บรรจุ issue number และขอบเขตของงาน

กฎ member cap สำคัญมาก: สูงสุด 4 workers ต่อ team ถ้างานมีมากกว่า 4 issues ให้ทำเป็น sequential batch — รอ batch แรกเสร็จ shutdown ก่อน แล้ว spawn batch ใหม่ผ่าน L2 เดิม การเกิน 4 workers ทำให้ coordination overhead สูงเกินประโยชน์

# ตัวอย่าง TEAM route สมบูรณ์
# L2 ประกาศ strategy
echo '{"route":"TEAM","justification":"3 issues แยก module, ต้องการ research + test"}' \
  > .maw/strategy.json

# สร้าง team
maw team create sprint-issues

# spawn workers — brief บรรจุที่ spawn ทันที
maw team spawn sprint-issues worker-1 \
  --wt --engine omx --exec \
  --prompt "Issue #10: เพิ่ม rate limiting ใน auth middleware. ไฟล์: src/auth/. เขียน unit test ด้วย."

maw team spawn sprint-issues worker-2 \
  --wt --engine omx --exec \
  --prompt "Issue #11: แก้ bug ใน user profile API. ไฟล์: src/api/users/. ใช้ /debug-mantra ก่อน."

maw team spawn sprint-issues worker-3 \
  --wt --engine omx --exec \
  --prompt "Issue #12: อัปเดต OpenAPI spec ให้ตรงกับ implementation ใหม่."

# ตรวจสอบทุก worker ทำงานจริง
maw panes
maw capture sprint-issues-worker-1

# หลังทุก worker DONE: aggregate branches
git merge --no-ff sprint-issues-worker-1/branch
git merge --no-ff sprint-issues-worker-2/branch
git merge --no-ff sprint-issues-worker-3/branch

# ตรวจสอบ + เปิด PR
touch .maw/aggregate-verified
maw pr

# Shutdown team
maw team shutdown sprint-issues

# DONE-ping
maw hey my-oracle "DONE: PR พร้อม review (Closes #10 #11 #12). ขอ /scrutinize + merge + maw done sprint-issues"
mkdir -p .maw && touch .maw/done-pinged

9.4 strategy.json gate — machine บังคับใช้

การประกาศ strategy ไม่ใช่แค่ convention — มี pre-guard hook บังคับใช้โดยอัตโนมัติ ก่อนที่ L2 จะแก้ไขไฟล์ใดๆ ใน product repo, hook จะอ่าน .maw/strategy.json และบล็อกการทำงานถ้าพบว่า route == "TEAM" แต่ยังไม่มี worker ถูก spawn

การ bypass gate มีช่องทางเดียว: ถ้างานเป็น TEAM จริงแต่มีเหตุผลที่ L2 ต้องแก้บางไฟล์เองก่อน (เช่น สร้าง scaffold ที่ workers ต้องพึ่งพา) ให้เขียน .maw/solo-justified ก่อน ไฟล์นี้บอก gate ว่ามีการพิจารณาแล้ว ไม่ใช่ความเผลอเรอ

# รูปแบบ .maw/strategy.json
{
  "route": "SOLO",
  "justification": "แก้ config 1 ไฟล์, ไม่ต้อง research"
}

# หรือสำหรับ TEAM
{
  "route": "TEAM",
  "justification": "3 issues แตะ module ต่างกัน ต้องการ OMX workers แยก worktree"
}

# กรณี L1 bind routing ล่วงหน้าก่อนส่ง brief
# L1 สร้างไฟล์นี้ใน worktree ใหม่ก่อน L2 เริ่มงาน
{
  "route": "TEAM",
  "justification": "L1 binding: 4 issues decomposed, worker split explicit in brief"
}

# กรณี bypass สำหรับ tightly-coupled refactor (rare)
# ต้องมีเหตุผลชัดเจน
printf '{"justification":"scaffold ก่อน spawn — config file สร้างไว้ให้ workers ใช้"}' \
  > .maw/solo-justified

hook ยังบล็อกกรณีอื่นโดยอัตโนมัติ: ถ้า L2 เริ่มสร้าง module directory ใหม่โดยไม่มี workers อยู่เลย gate ถือว่านี่คือ scope ของ TEAM ไม่ใช่ SOLO

9.5 Escalation guard: เมื่อ SOLO กลายเป็น TEAM

SOLO ที่เริ่มต้นด้วยความตั้งใจดีอาจขยายขอบเขตโดยไม่รู้ตัว — เช่น แก้ bug แล้วพบว่าต้นเหตุอยู่ใน layer อื่น ต้องแก้หลายไฟล์ หรือต้องเขียน test ที่ซับซ้อน Escalation guard กำหนดให้ L2 ต้อง หยุดทันที เมื่อพบสัญญาณต่อไปนี้ระหว่าง SOLO:

  • ต้องทำ research เพิ่มที่ไม่ได้คาดไว้ตอน estimate
  • จำนวนไฟล์ที่ต้องแก้เกิน 4 ไฟล์
  • จำนวนบรรทัดที่เปลี่ยนแปลงเกิน 200 บรรทัด
  • พบ concern ใหม่ที่ควรแยกเป็น issue ต่างหาก

เมื่อพบสัญญาณ L2 ต้องทำทันที: หยุดเขียนโค้ด → ประกาศ "แปลงจาก SOLO เป็น TEAM" → อัปเดต .maw/strategy.json → spawn OMX workers ตามขอบเขตที่ค้นพบใหม่ โค้ดที่เขียนไปแล้วใน SOLO phase ยังอยู่ใน worktree — L2 commit เป็น WIP branch แล้วให้ worker ที่เกี่ยวข้อง rebase จากนั้น

# กระบวนการแปลง SOLO เป็น TEAM
# 1. หยุดเขียนโค้ด commit งานที่ทำไปแล้ว
git add src/auth/middleware.ts
git commit -m "wip: partial auth fix — converting to TEAM"

# 2. อัปเดต strategy
echo '{"route":"TEAM","justification":"พบว่าต้องแก้ 5 ไฟล์ + เขียน test — เกิน SOLO threshold"}' \
  > .maw/strategy.json

# 3. spawn workers ตาม scope ที่ค้นพบจริง
maw team create sprint-escalated
maw team spawn sprint-escalated worker-1 \
  --wt --engine omx --exec \
  --prompt "Issue #15: ต่อจาก WIP auth/middleware.ts. Rebase จาก branch wip/issue-15."

maw team spawn sprint-escalated worker-2 \
  --wt --engine omx --exec \
  --prompt "Issue #15: เขียน unit test สำหรับ auth middleware ใหม่. ไฟล์: tests/auth/"

9.6 Parallel L2s: issues อิสระทำพร้อมกัน

กฎ "ต้องผ่าน L2 เดียว" ไม่ใช่ข้อบังคับ — เป็นเพียง default ที่ปลอดภัย เมื่อ L1 วิเคราะห์แล้วพบว่า issues หลายตัวอิสระต่อกัน (ไฟล์ไม่ทับกัน, ไม่มี shared API contract), L1 สามารถเปิด parallel L2 windows ได้ — แต่ละ L2 รับผิดชอบ issues ของตัวเองตั้งแต่ต้นจนจบ

ตัวอย่าง: issue #20 (แก้ frontend component) กับ issue #21 (เพิ่ม API endpoint ใหม่) อิสระต่อกันอย่างสมบูรณ์ L1 เปิด maw workon my-product issue-20 และ maw workon my-product issue-21 พร้อมกัน แต่ละ L2 routes SOLO/TEAM เองตาม scope

ข้อจำกัดสำคัญของ Parallel L2s ที่ต้องปฏิบัติอย่างเคร่งครัด:

  • ไม่เปิด parallel L2s สำหรับ issues ที่แตะไฟล์เดียวกัน — merge conflict ที่ L1 ต้องแก้มือ
  • L1 merge PRs ทีละใบตามลำดับ (serial) — scrutinize → merge → rebase ใบถัดไปก่อน merge
  • maw done ทีละอัน ไม่รันพร้อมกัน — shared git lock จะ corrupt
# L1 เปิด parallel L2s สำหรับ issues ที่อิสระต่อกัน
# ตรวจสอบก่อนว่าไม่มีไฟล์ทับกัน
maw workon my-product issue-20   # frontend component — L2-A window
maw workon my-product issue-21   # API endpoint ใหม่ — L2-B window

# L1 monitor ทั้งสอง
maw panes

# เมื่อ L2-A DONE-ping มาก่อน:
# L1 scrutinize PR #20 → merge → maw done L2-A window

# แล้วค่อย scrutinize PR #21 → merge → maw done L2-B window
# (serial merge, ไม่ parallel)

9.7 ตาราง SOLO vs TEAM และ member cap

ตารางเปรียบเทียบด้านล่างสรุปทุกมิติที่สำคัญระหว่าง SOLO และ TEAM รวมถึง member cap และการจัดการ batch สำหรับงานขนาดใหญ่

มิติ SOLO TEAM
เลือกเมื่อ ชัดเจน, 1-2 ไฟล์, ไม่ต้อง research, ไม่แน่ใจ = ไม่ใช่ ทุกกรณีที่ไม่ SOLO — รวมถึงกรณีไม่แน่ใจ
Workers ไม่มี — L2 โค้ดเอง 1-4 OMX workers (max 4)
PRs 1 PR ต่อ issue 1 consolidated PR ที่มีทุก Closes #N
strategy.json "route": "SOLO" "route": "TEAM"
เวลาทำ ต่ำกว่า 30 นาที ไม่จำกัด — ตาม scope
Aggregate step ไม่มี ต้อง: lint/build/test + touch .maw/aggregate-verified
Team shutdown ไม่มี บังคับ: maw team shutdown ก่อน DONE-ping
Merge authority L1 เท่านั้น L1 เท่านั้น
Escalation เปลี่ยนเป็น TEAM ทันทีเมื่อเกิน threshold batch ต่อเนื่องถ้า >4 issues

สำหรับงานที่มีมากกว่า 4 issues การทำเป็น sequential batch ผ่าน L2 เดิม: ทำ batch 1 (4 issues) → shutdown → ทำ batch 2 → shutdown → รวม PRs ก่อน DONE-ping ถ้า issues อิสระต่อกัน อาจเปิด Parallel L2s แต่ยัง limit 4 workers ต่อ L2

9.8 สรุปบทที่ 09

SOLO vs TEAM routing เป็นการตัดสินใจที่ง่ายที่สุดในทั้ง workflow — คำถามเดียว, คำตอบสองทาง, ไม่มีระดับกลาง ความสำคัญอยู่ที่การยอมรับว่า TEAM คือ default ที่ปลอดภัย ไม่ใช่สิ่งที่ต้องหลีกเลี่ยง

strategy.json gate ทำให้การตัดสินใจมีน้ำหนัก — ไม่ใช่แค่ convention แต่มี machine enforcement ปิดกั้นการเลื่อนขอบเขตโดยไม่ตั้งใจ Escalation guard ป้องกัน SOLO ที่โตเกิน threshold และ parallel L2s ให้ flexibility สำหรับ independent issues

บทถัดไปจะครอบคลุม SOPs และ Merge Gate — ระบบ SOP ที่โหลดก่อนงาน, sequence การทำงานจนถึง DONE-ping, และกลไก on-stop crash safety ที่รับรองว่า L1 ไม่มีทางพลาด PR แม้ L2 จะ crash กลางคัน

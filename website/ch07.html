Gale Workflowบทที่ 07

maw-js Commands

อ้างอิงคำสั่ง maw ฉบับเต็ม — ทำไม raw tmux ถูกบล็อก, hey/capture/panes และ engine map

7.1 maw คืออะไร

maw-js คือ CLI ที่เป็น interface เดียว ระหว่าง oracle กับ tmux และ agents ทั้งหมด แทนที่จะใช้คำสั่ง tmux โดยตรง ทุกการกระทำที่เกี่ยวกับ pane, window, session, และ agent ต้องผ่าน maw เสมอ

เหตุผลหลักคือ maw ทำมากกว่าแค่ wrap tmux — มันจัดการ engine routing, audit trail, worktree lifecycle, และ hook enforcement ด้วย การ bypass maw ด้วย raw tmux ทำให้ขาดการตรวจสอบเหล่านี้ทั้งหมด

maw ทำงานได้ทั้งจาก terminal ปกติและจาก agent session (Claude Code, Codex) ผ่าน Bash tool คำสั่งที่ agent เรียกบ่อยที่สุดคือ maw workon, maw hey, maw capture, และ maw panes

7.2 ทำไม Raw tmux ถูกบล็อก

Gale Workflow บล็อก raw tmux commands (tmux send-keys, tmux new-window, ps aux | grep, kill -9) ผ่าน pre-tool hook ที่ตรวจ Bash command ก่อนอนุญาตให้รัน hooks เหล่านี้ถูกกำหนดใน Claude Code settings และทำงานอัตโนมัติทุก Bash call

สิ่งที่บล็อก เหตุผล ใช้แทนด้วย
tmux send-keysไม่มี audit trail, signed envelopemaw hey <target> "msg"
tmux new-windowไม่ inject oracle identity, ไม่ตั้ง cwd ถูกต้องmaw workon <repo> <slug>
kill -9 <pid>อาจ kill agent ที่ยังทำงานอยู่maw tmux kill <target>
ps aux | grep claudeไม่แม่นยำ, ใช้แทน maw fleet ไม่ได้maw panes
tmux kill-paneไม่ cleanup worktreemaw done <window>

นอกจากเหตุผลด้านความถูกต้องแล้ว การใช้ maw ยังทำให้ audit log สมบูรณ์ ทุกการสื่อสาร agent-to-agent ถูกบันทึกพร้อม timestamp และ sender identity ซึ่งช่วยมากเมื่อต้อง debug ว่าใครสั่งอะไรและเมื่อไหร่

7.3 Spawn/Isolate: workon, team spawn, wake, swarm

กลุ่มคำสั่ง spawn/isolate ใช้เปิด pane ใหม่หรือสร้าง worktree สำหรับงาน คำสั่งที่ใช้บ่อยที่สุดคือ maw workon ซึ่งเป็น "work verb" ค่าเริ่มต้นสำหรับทุก task

คำสั่ง ใช้เมื่อ สิ่งที่เกิดขึ้น
maw workon <repo> <slug> ทุก task บน product/infra repo (DEFAULT) เปิด tmux window ใหม่ใน project dir; โหลด CLAUDE.md/AGENTS.md ของ project อัตโนมัติ; engine = Claude เสมอ
maw team spawn <team> <role> --wt --engine omx --exec --prompt "..." spawn L3 OMX worker สำหรับ TEAM task สร้าง fresh worktree + pane; inject brief ผ่าน spawn-prompt; auto-kickoff สำหรับ codex/omx
maw wake <repo> --wt <slot> -e <engine> repo ที่ใช้บ่อยมาก (OPT-IN เท่านั้น) idempotent: สร้าง worktree + branch + pane ถ้ายังไม่มี; warm worktree, fresh session
maw swarm <engine1> <engine2> A/B compare engines เท่านั้น shared-cwd, ไม่รองรับ --wt; ห้ามใช้สำหรับ parallel commits
กฎสำคัญ: maw workon pane เป็น Claude เสมอ — ห้าม force --engine omx บน workon pane engine flag ใช้กับ maw team spawn เท่านั้น workon pane คือ orchestrator (L2); team spawn pane คือ worker (L3)

7.4 Communicate: hey, send-text, capture, panes

การสื่อสารระหว่าง agent กับ agent ทั้งหมดต้องผ่าน maw communicate commands ห้ามใช้ raw tmux send-keys เพราะขาด signed envelope และ audit log 

# ส่ง message พร้อม Enter (signed inject)
maw hey my-oracle "DONE: PR #42 ready for /scrutinize + merge"

# maw send เป็น alias ของ maw hey
maw send my-oracle "same as hey"

# พิมพ์ข้อความ + กด Enter (ไม่มี envelope — ใช้สำหรับ compose text)
maw send-text my-oracle "text without envelope"

# อ่าน output ของ pane (capture ดีกว่า peek สำหรับ audit)
maw capture worker-1 --lines 50

# ดู pane output แบบ quick (ไม่บันทึก)
maw peek worker-1

# แสดง panes ทั้งหมดที่ active
maw panes

# แสดง ls ของ worktrees และ sessions
maw ls

maw capture vs maw peek: capture บันทึกลง audit log และ return สมบูรณ์กว่า ใช้ capture เมื่อต้องการ verify worker status หรือ debug ใช้ peek เมื่อต้องการแค่ดูแบบเร็ว ๆ ไม่ต้องการ log

Anti-pattern ที่ต้องระวัง: อย่า poll ด้วย maw capture หลาย ๆ ครั้งติดกัน (serial capture polling) ถ้าต้องการ monitor worker ให้ใช้ maw team status <team> ก่อน แล้ว capture เฉพาะเมื่อเห็น anomaly

7.5 Finish/Cleanup: pr, done, team shutdown

กลุ่มคำสั่ง finish/cleanup ใช้ปิดงานอย่างสะอาด ขั้นตอนที่ถูกต้องหลัง worker ส่ง DONE-ping คือ: L1 scrutinize → merge → maw done <window> (จาก L1 pane) 

# เปิด PR จาก branch ปัจจุบัน (ต้องมี .maw/aggregate-verified สำหรับ TEAM)
maw pr

# ปิด worktree window + teardown (รันจาก L1/oracle pane เท่านั้น)
maw done sprint-x-window

# ปิด team workers + prune worktrees (MANDATORY หลังทุก TEAM batch)
# รันจาก L2 pane หลัง PR open และก่อน DONE-ping
maw team shutdown sprint-x

# kill orphan panes ทั่ว fleet
maw cleanup --zombie-agents

# hide pane (ไม่ kill)
maw tmux close <target>

# kill pane (kill จริง)
maw tmux kill <target>
กฎที่สำคัญที่สุด: L2 worktree pane MUST NOT รัน maw done บน window ของตัวเอง เพราะ maw done ลบ worktree = ลบ cwd ของตัวเอง → ทุก command ถัดไปจะ fail ด้วย ENOENT และ window จะกลายเป็น zombie process maw done เป็นหน้าที่ของ L1 oracle pane เสมอ

7.6 Engine Map: Key ใน Config และ Silent Fallback

engine map คือ section commands ใน ~/.maw/config.json มันบอก maw ว่าเมื่อ spawn engine ชื่อ X ให้รัน binary อะไร gotcha สำคัญ: ถ้า key ไม่มีอยู่ใน map maw จะ fallback ไป default engine โดยไม่มี error ใด ๆ 

// ตัวอย่าง ~/.maw/config.json (generic)
{
  "version": "2.1.0",
  "oracle": "my-oracle",
  "session": "my-oracle-session",
  "commands": {
    "claude": "claude",
    "codex": "codex",
    "omx": "node /path/to/omx/bin/omx.js",
    "omx-resume": "node /path/to/omx/bin/omx.js --resume",
    "codex-resume": "codex --resume"
  },
  "defaults": {
    "engine": "claude",
    "worktree_base": "~/worktrees"
  }
}

# ตรวจสอบ engine map ก่อน spawn
cat ~/.maw/config.json | jq '.commands'

# ตรวจ omx worker จริง ๆ ใช้ omx ไหม (CMD ควรเป็น 'node')
maw panes
# ถ้า CMD = 'claude' แทน 'node' = fallback เกิดขึ้น → ตรวจ key 'omx' ใน config

# spawn omx worker
maw team spawn sprint-x worker-1 \
  --wt \
  --engine omx \
  --exec \
  --prompt "Issue #15: implement refresh token rotation in src/middleware/auth.ts"

วิธีวินิจฉัย silent fallback: รัน maw panes แล้วดู column CMD omx worker จริงจะมี node เป็น CMD เพราะ omx รันบน Node.js ถ้าเห็น claude แสดงว่า key omx ไม่มีใน config หรือ path ผิด

นอกจาก omx และ codex แล้ว ต้องมี resume keys ด้วย: omx-resume และ codex-resume ถ้าขาด resume keys เมื่อ worker crash แล้ว maw พยายาม resume ระบบจะ fallback ไปเปิด Claude แทน ทำให้ได้ engine ผิด

7.7 ตารางอ้างอิงคำสั่งเต็ม

ตารางด้านล่างรวมคำสั่ง maw ทั้งหมดที่ใช้บ่อย พร้อม alias และหมายเหตุสำคัญ

คำสั่ง Alias หน้าที่ หมายเหตุ
maw workon <repo> <slug>—เปิด L2 worktree pane (DEFAULT work verb)engine = Claude เสมอ, โหลด project CLAUDE.md
maw team create <slug>—ลงทะเบียน team rosterทำก่อน team spawn
maw team spawn <team> <role> [opts]—spawn L3 OMX workerต้องมี --wt --engine omx --exec --prompt
maw team status <team>—ดูสถานะ workers ทั้งหมดของ teamomx แสดง idle แม้ทำงานอยู่ — verify ด้วย capture
maw team shutdown <team>—ปิด workers + prune worktreesMANDATORY หลังทุก TEAM batch
maw wake <repo> --wt <slot>—warm worktree + fresh session (OPT-IN)ใช้เฉพาะ high-frequency repo
maw swarm <e1> <e2>—A/B compare engines, shared-cwdห้ามใช้สำหรับ parallel commits
maw hey <target> "msg"maw sendส่ง message พร้อม Enter + signed envelopedefault communication tool
maw send-text <target> "text"—พิมพ์ text + Enter ไม่มี envelopeใช้สำหรับ compose text เท่านั้น
maw send-enter <target>—กด Enter เท่านั้นไม่ส่ง text
maw capture <target> [--lines N]—อ่าน pane output + บันทึก audit logใช้ verify worker status
maw peek <target>—อ่าน pane output แบบ quickไม่บันทึก log
maw panesmaw lsแสดง panes ทั้งหมด + CMD columnใช้ verify engine ที่ worker ใช้
maw pr—เปิด PR จาก current branchต้องมี .maw/aggregate-verified สำหรับ TEAM
maw done <window>—retro + kill window + remove worktreeรันจาก L1 เท่านั้น (ไม่ใช่จาก window เป้าหมาย)
maw cleanup --zombie-agents—kill orphan panes ทั่ว fleetรันเมื่อมี pane ค้างโดยไม่มีเจ้าของ
maw tmux close <target>—hide pane (ไม่ kill)—
maw tmux kill <target>—kill pane จริง—
maw fleet pr-queue—ดู pending PRs ที่รอ reviewตรวจทุกครั้งเมื่อ wake/recap
maw --version—แสดง maw-js versionใช้ verify install

สรุปบทที่ 07

maw-js คือ interface เดียวที่ Gale Workflow อนุญาตให้ใช้สำหรับจัดการ tmux และ agents การบล็อก raw tmux ไม่ใช่ข้อจำกัด แต่เป็นการบังคับให้ใช้ path ที่มี audit trail, hook enforcement, และ oracle identity injection ครบถ้วน

คำสั่งที่สำคัญที่สุดคือ maw workon (spawn L2), maw team spawn (spawn L3), maw hey (สื่อสาร), maw capture (verify), maw done (cleanup) และ maw team shutdown (tear down workers) จำกฎง่าย ๆ: L2 ไม่ merge, L2 ไม่ maw done ตัวเอง — ทั้งสองอย่างเป็นหน้าที่ L1

gotcha ที่พบบ่อยที่สุดคือ engine map key หายทำให้ fallback เงียบ ๆ — ตรวจด้วย maw panes และ jq '.commands' ก่อน spawn team ทุกครั้ง บทถัดไปจะอธิบาย Delegation Tiers ซึ่งกำหนดว่าใครส่งงานให้ใคร และเมื่อไหร่ควร dispatch vs inline

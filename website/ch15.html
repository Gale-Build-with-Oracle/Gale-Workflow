Troubleshooting

decision tree ตามชั้น, failure modes, zombie panes และการ purge job

15.1 Troubleshooting Mindset — Debug Mantra

การ debug ใน Gale Workflow ไม่ว่าจะเป็นปัญหาที่ L1, L2, หรือ L3 เริ่มต้นจาก debug mantra เดียวกันเสมอ: reproduce → trace fail path → falsify hypothesis → cross-reference mantra นี้บังคับให้ไม่กระโดดไปหาวิธีแก้ทันทีก่อนเข้าใจปัญหา

  • Reproduce: สร้างให้ปัญหาเกิดซ้ำได้ก่อน ถ้า reproduce ไม่ได้ ยังไม่รู้ว่ามีปัญหาจริงหรือเปล่า
  • Trace fail path: ตามรอย log, exit code, pane output จากจุดที่ล้มกลับมาถึงต้นเหตุ อย่ากะเกนว่าปัญหาอยู่ที่ไหน
  • Falsify hypothesis: สร้าง hypothesis ว่าน่าจะเป็นสาเหตุอะไร แล้วหาทางพิสูจน์ว่า hypothesis นั้นผิดก่อน — ถ้าพิสูจน์ว่าผิดไม่ได้ค่อยเชื่อว่าถูก
  • Cross-reference: ตรวจ breadcrumb ทุกชิ้น (git log, maw capture, docker compose ps, maw panes) ก่อนสรุป ข้อมูลชิ้นเดียวมักทำให้เข้าใจผิด

วิธีนี้อาจดูช้า แต่ปัญหาส่วนใหญ่ใน Gale Workflow ที่ใช้เวลานานแก้คือปัญหาที่แก้ผิด hypothesis ตั้งแต่ต้น การ reproduce และ trace อย่างเป็นระบบมักพา path สั้นกว่าการลอง fix แบบ trial-and-error

15.2 Decision Tree ตามชั้น

อาการต่าง ๆ ใน Gale Workflow มักระบุชั้นที่มีปัญหาได้ค่อนข้างชัด ตารางด้านล่างช่วยจำแนกอาการและ layer ที่ต้องตรวจก่อน

อาการ Layer ที่น่าสงสัย คำสั่งตรวจแรก
L2 ไม่ตอบสนอง, ไม่ส่ง DONE-ping L2 Session maw panes, maw capture 01-my-oracle:sprint-x.1
Worker ค้าง ไม่ commit L3 Worker maw capture worker-1, ดู CMD ว่า node หรือ bash
Engine ผิด (spawn เป็น Claude แทน OMX) maw config maw panes ดู CMD; ตรวจ commands.omx ใน config
tool call ถูกบล็อก ไม่รู้สาเหตุ pre-guard hook อ่าน pre-guard output ใน session log
Context ผิด — L2 load project ผิด CLAUDE.md / cwd ตรวจ cwd ของ pane, ดู prompt-inject output
Docker service ล้มหลัง deploy Container / Code docker compose ps, docker compose logs --tail=50
Pane เกิดขึ้นเองไม่รู้จัก Ghost-clone / Daemon maw panes, ls ~/.claude/jobs/*/state.json

15.3 Zombie Panes — ENOENT จาก maw done ผิดที่

Zombie pane เกิดเมื่อ pane รัน maw done บน window ของตัวเอง maw done ลบ worktree ซึ่งก็คือ directory ที่ pane กำลังทำงานอยู่ ผลคือทุก command ถัดมาใน pane นั้น fail ด้วย ENOENT (No such file or directory) แต่ pane ยังคงเปิดอยู่เป็น "zombie" ที่ทำอะไรไม่ได้

วิธีป้องกันคือ รัน maw done <window> จาก pane ภายนอกเสมอ โดย L1 oracle pane เป็นผู้รัน maw done ให้กับ L2 worktree window หลังจาก L2 ส่ง DONE-ping และ L1 merge PR เรียบร้อยแล้ว 

# ผิด — L2 รัน maw done บน window ตัวเอง (ห้าม!)
# (ใน pane 01-my-oracle:sprint-x.1)
maw done sprint-x   # ← บล็อก cwd ตัวเอง → ENOENT zombie

# ถูก — L1 รัน maw done จาก pane ภายนอก
# (ใน pane 01-my-oracle:my-oracle.1 — L1 oracle pane)
maw done sprint-x   # ← ปลอดภัย L1 อยู่คนละ window

# ถ้าเกิด zombie pane แล้ว — วิธีกู้
# kill window โดยตรงผ่าน tmux (via maw)
maw tmux kill 01-my-oracle:sprint-x.1
# แล้วลบ worktree เองด้วยมือ (ถ้า maw done ไม่ทำงาน)
git worktree remove --force /path/to/worktree

ข้อสังเกตสำคัญ: zombie pane อาจทำให้ tmux window ค้างอยู่โดยไม่มีประโยชน์ ตรวจด้วย maw panes และ kill pane ที่ CMD เป็น bash และไม่มีกิจกรรม

15.4 Failed Respawn Jobs — Daemon Retry Loop

หลังจาก tmux server ตายหรือ WSL ดับกะทันหัน daemon ของ Gale Workflow อาจ retry job ที่มี state "failed" ซ้ำแล้วซ้ำเล่าโดยไม่หยุด แต่ละ retry เปิด pane ใหม่ใน tmux window ที่ active อยู่ในขณะนั้น ทำให้ window เต็มไปด้วย pane ลึกลับที่ไม่มีงานจริง ๆ

วิธีแก้คือ purge job ที่ fail แล้วออกจาก ~/.claude/jobs/ ใช้การย้ายไป /tmp แทนการลบ เพื่อให้ history ยังอยู่ตาม Principle "Nothing is Deleted" 

# ตรวจหา failed jobs
ls ~/.claude/jobs/
cat ~/.claude/jobs/*/state.json | grep -E '"state"'

# ตัวอย่าง output:
# {"state":"failed","name":"sprint-x-worker","..."}
# {"state":"failed","name":"old-infra-task","..."}

# Purge failed jobs (ย้ายไป /tmp ไม่ใช่ rm -rf)
mv ~/.claude/jobs/sprint-x-worker-xxx/ /tmp/jobs-archive/
mv ~/.claude/jobs/old-infra-task-yyy/ /tmp/jobs-archive/

# ตรวจว่า daemon หยุด retry แล้ว
maw panes   # ไม่ควรมี pane ใหม่เกิดขึ้นอีก

# ถ้า daemon ยังรันอยู่และ retry อีก ตรวจ process
ps aux | grep maw

หลังจาก purge แล้วให้รัน maw panes เพื่อยืนยันว่าไม่มี pane ใหม่เกิดขึ้น ถ้ายังมี — daemon อาจยังรันอยู่และหา job อื่นใน ~/.claude/jobs/ ตรวจ directory ให้แน่ใจว่า state files ทั้งหมดที่เหลืออยู่ไม่ใช่ "failed"

15.5 Engine-Map Silent Fallback

ปัญหาที่พบบ่อยและ debug ยากที่สุดตัวหนึ่งคือ silent engine fallback เมื่อ maw team spawn -e omx แต่ omx key ไม่มีใน maw config maw จะ fallback ไปใช้ default engine โดยไม่แจ้งเตือน (silent) ผลคือ spawn เป็น Claude แทน OMX โดยที่ไม่มี error message ใด ๆ

วิธีตรวจคือดู CMD ของ pane ด้วย maw panes:

  • CMD = node — OMX กำลังรัน (ถูกต้องสำหรับ worker)
  • CMD = codex — เป็น engine ผิด (ไม่ใช่ OMX)
  • CMD = bash — engine ตายแล้ว pane กลายเป็น shell เปล่า
# ตรวจ engine ของ worker pane
maw panes
# ดู CMD column ของ worker pane ที่ spawn ไป

# ตัวอย่าง output ที่ถูกต้อง (OMX):
# 01-my-oracle:sprint-x-worker-1.1  CMD=node  RUNNING

# ตัวอย่าง output ที่ผิด (Claude แทน OMX):
# 01-my-oracle:sprint-x-worker-1.1  CMD=claude  RUNNING

# ตรวจ maw config — ดู key omx
cat ~/.config/maw/config.json | grep -A3 '"commands"'
# ต้องมี "omx": "..." อยู่ใน commands map

# ถ้าไม่มี key — เพิ่มลงใน config
# "omx": "/usr/local/bin/omx-launch"

ก่อน spawn worker ทุกครั้ง ให้ verify ว่า key omx มีอยู่ใน maw config เป็น pre-flight check ที่ใช้เวลาไม่ถึง 5 วินาทีแต่ป้องกันปัญหาที่ debug ยากมาก

15.6 OMX Spawn Race — "submit FAILED" เป็น False Negative

เมื่อ spawn OMX worker ด้วย maw team spawn --exec --prompt "..." บางครั้ง terminal แสดงข้อความ submit FAILED ซึ่งทำให้ดูเหมือน brief ไม่ได้ส่ง แต่ในความเป็นจริง นี่มักเป็น false negative

เหตุผลคือ OMX engine ไม่รับ --system-prompt-file เหมือน Claude brief ถูกส่งผ่านกลไกอื่น — spawn-prompt file และ auto-kickoff submit FAILED มักแปลว่า harness พยายาม submit ผ่านช่องทางที่ไม่รองรับ แต่ brief ยังถึง worker ผ่านช่องทางที่ถูกต้องอยู่ 

# ตรวจว่า worker ได้รับ brief หรือเปล่า (แม้เห็น submit FAILED)
# วิธีที่ถูกต้องคือ capture output ของ worker
maw capture 01-my-oracle:sprint-x-worker-1.1 --lines 30

# ถ้า worker กำลังอ่าน brief หรือ Working — brief ถึงแล้ว ไม่ต้อง nudge
# ถ้า worker ยังว่าง (ไม่มีกิจกรรม) หลังจากรอ ~30 วินาที — nudge ด้วย:
maw hey 01-my-oracle:sprint-x-worker-1.1 "read /path/to/brief.md and begin"

# ตรวจ spawn-prompt file ว่ามีเนื้อหา
cat .maw/briefs/worker-1.md

สรุป: อย่ากระโดดไป nudge worker ทันทีที่เห็น submit FAILED รอ 30-60 วินาทีแล้ว capture output ก่อน ถ้า worker กำลัง Working ก็ไม่ต้องทำอะไร การ nudge worker ที่กำลังทำงานอยู่แล้วอาจทำให้ confused และ restart งานจากต้น

15.7 Worker Stalls — Monitor by Status Cadence

Worker stall (ค้างโดยไม่มีความคืบหน้า) เป็นปัญหาที่ต้องการ detection เร็วแต่ไม่ควร poll บ่อยเกินไป pattern ที่ถูกต้องคือ monitor by status cadence — ตรวจทุก ~5 นาที ไม่ใช่ทุกนาที

maw team status <team> เป็นจุดเริ่มต้นแต่ไม่น่าเชื่อถือสำหรับ OMX workers (มักแสดง idle แม้ worker กำลังทำงาน) ถ้า status แสดงผลน่าสงสัยให้ใช้ maw capture เป็น source of truth แทน

อาการ สาเหตุที่เป็นไปได้ วิธีแก้
Worker ไม่มี commit หลัง 15 นาที Stall, รอ input, หรือตาย maw capture ดู output; nudge หรือ re-spawn
CMD = bash แทน node OMX engine ตาย re-spawn worker: maw team spawn
Worker commit ผิด branch Brief ไม่ชัด หรือ worktree overlap git log ใน worktree; cherry-pick ไปยัง branch ที่ถูก
submit FAILED ซ้ำ ๆ Engine-map key หาย หรือ spawn race ตรวจ config; capture worker; nudge ถ้ารอ 60s แล้วยังว่าง
Zombie pane เต็ม window maw done รันในตัวเอง หรือ failed respawn kill pane ด้วย maw tmux kill; purge job ใน ~/.claude/jobs/
maw workon load context ผิด project cwd ผิด หรือ prompt-inject hook ไม่ทำงาน ตรวจ cwd ของ pane; ตรวจ hook settings.json
Hook บล็อก command ที่ควรอนุญาต pre-guard pattern ครอบคลุมเกิน ดู hook output; แก้ regex ใน pre-guard.sh 
# Protocol — worker stall response
# 1. ตรวจ status (รู้ว่า unreliable สำหรับ omx)
maw team status sprint-x

# 2. Capture ก่อนเสมอ — อย่าพึ่งแค่ status
maw capture 01-my-oracle:sprint-x-worker-1.1 --lines 50

# 3. ดู CMD
maw panes | grep worker-1

# 4a. ถ้า CMD=node และมี output — worker ยังทำงาน รอต่อ
# 4b. ถ้า CMD=node แต่ไม่มี output นาน — nudge ด้วย maw hey
maw hey 01-my-oracle:sprint-x-worker-1.1 "status? ยังทำงานอยู่ไหม"

# 4c. ถ้า CMD=bash — ตาย re-spawn
maw team spawn sprint-x worker-1 --wt --engine omx --exec \
  --prompt "Issue #N: resume from last commit. Previous work: git log."

การ re-spawn worker ที่ตาย ควรระบุใน brief ว่า "resume from last commit" เพื่อให้ worker ใหม่เริ่มจาก git state ปัจจุบัน ไม่ใช่เริ่มจากต้น git log --oneline -5 ใน worktree ช่วยบอก context ที่ worker ทำมาแล้ว

15.8 Triage เมื่อ failure ซ้อนกันและการกู้ ψ vault

เมื่อ failure หลายชั้นเกิดพร้อมกัน อย่าแก้จากอาการที่ดังที่สุด ให้เรียงตามสิ่งที่สร้าง failure เพิ่มก่อน เพราะถ้าแหล่งกำเนิดยังทำงานอยู่ การแก้ pane หรือ hook จะถูกกลบด้วยอาการใหม่ทันที

  1. Daemon retry-loop ก่อน: ถ้ามี failed respawn job หรือ background daemon ที่เปิด pane ซ้ำ ให้หยุด loop ก่อน เพราะมันสร้าง failure เพิ่มทุก cycle
  2. Panes ถัดมา: หลัง loop หยุดแล้วค่อยจัดการ zombie, wrong-engine หรือ pane ที่ cwd หาย เพื่อให้ roster สะอาดและอ่านสถานะได้จริง
  3. Hooks สุดท้าย: เมื่อ runtime นิ่งแล้วค่อย debug hook failure เช่น pre-guard หรือ prompt-inject เพราะตอนนั้น output จะไม่ปนกับ noise จาก pane/daemon

สำหรับ ψ vault ให้ถือ markdown files เป็น source of truth ส่วน SQLite หรือ vector index เป็น cache สำหรับค้นหา ถ้า index corrupt หรือ search เพี้ยน ให้ reindex จากไฟล์ markdown แทนการแก้ข้อมูลใน DB โดยตรง วิธีนี้รักษาหลัก Nothing is Deleted: ความทรงจำจริงยังอยู่ในไฟล์ที่ audit ได้ และ index สร้างใหม่ได้เสมอ

15.9 สรุปบทที่ 15

Troubleshooting ใน Gale Workflow มีโครงสร้างที่ชัดเจน — ระบุ layer ก่อน แล้วใช้เครื่องมือที่เหมาะกับ layer นั้น debug mantra (reproduce → trace → falsify → cross-reference) ป้องกันการ fix ผิด hypothesis ซึ่งเป็นสาเหตุหลักที่ทำให้ปัญหาใน multi-layer system ใช้เวลาแก้นาน

ปัญหาที่พบบ่อยที่สุดคือ zombie panes จาก maw done ผิดที่, failed respawn jobs ที่ daemon retry ไม่หยุด, engine-map silent fallback, และ OMX spawn race ที่แสดง submit FAILED แบบ false negative ทุกปัญหาเหล่านี้มี pattern การตรวจและแก้ที่ชัดเจน สิ่งสำคัญคืออย่า panic และ capture ข้อมูลก่อน action

Fleet ที่ stable ในระยะยาวต้องมาจาก disciplined monitoring ไม่ใช่ reactive firefighting monitor by cadence ตาม schedule ไม่ใช่รอให้มีปัญหาแล้วค่อยตาม และเมื่อพบปัญหา ให้ document ไว้ใน ψ vault หรือ retro เพื่อให้ pattern นั้นไม่เกิดซ้ำในสภาพแวดล้อมเดิม

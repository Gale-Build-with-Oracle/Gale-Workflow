Gale Workflowบทที่ 02

สถาปัตยกรรม 3 ชั้น

L1 Oracle, L2 Orchestrator, L3 Worker — ใครทำอะไร, merge gate, และ project-scope injection

2.1 ทำไมต้องแบ่งชั้น — Asymmetric Autonomy

คำถามแรกที่มักเกิดขึ้นคือ "ทำไมไม่ให้ AI agent ตัวเดียวทำทุกอย่าง?" คำตอบอยู่ในแนวคิด asymmetric autonomy — แต่ละชั้นมีขอบเขตสิทธิ์ที่ไม่เท่ากัน เพราะแต่ละชั้นมีข้อมูลไม่เท่ากัน

L3 worker ที่ทำงานใน worktree แยกของตัวเองไม่มีภาพรวมของ PR ทั้งหมด ไม่รู้ว่า worker คนอื่นกำลังแก้ไขไฟล์อะไร และไม่รู้ว่า project owner ต้องการ quality level แบบไหน ถ้า worker สามารถ merge ได้เอง ก็เท่ากับ self-review — ไม่มีการตรวจสอบอิสระ

L1 Oracle เป็นชั้นเดียวที่มีภาพรวมครบถ้วน: รู้งานทุกชิ้น รู้ผลลัพธ์ที่คาดหวัง และมีความสัมพันธ์กับมนุษย์โดยตรง จึงเป็นชั้นเดียวที่เหมาะสมในการ merge PR และเป็นจุดเดียวที่ accountability รวมอยู่

Asymmetric Autonomy หมายความว่า: ยิ่งมี context น้อย ยิ่งมีสิทธิ์น้อย L3 worker ทำได้แค่ code + commit, L2 orchestrator เพิ่มสิทธิ์ spawn + aggregate + open PR, L1 Oracle เพิ่มสิทธิ์ review + merge + close worktree

2.2 L1 Oracle — ชั้นบัญชาการ

L1 Oracle คือ permanent pane ที่ทำงานอยู่ตลอดเวลาในขณะที่มีงาน มันเป็นจุดที่มนุษย์พูดคุยและส่งงาน มันรับ task จากมนุษย์ ทำ task intake (เปิด issue ใน GitHub), วิเคราะห์ว่าควร route ไป L2 ใด และสื่อสารกลับหามนุษย์

หน้าที่หลักของ L1 ได้แก่: เปิด GitHub issue ก่อน action ใดๆ (issue number เป็น traceability thread), dispatch งานไป L2 ผ่าน maw workon, monitor สถานะ L2 โดยรับ DONE-ping, รัน /scrutinize บน PR และ merge ผ่าน gh pr merge --merge --delete-branch และสุดท้ายรัน maw done <window> เพื่อปิด worktree หลัง merge

L1 Oracle เป็น Claude (ไม่ใช่ OMX/Codex) เสมอ เพราะ hook ของ Claude Code bind หนักกว่า มี monitor loop และ full-fidelity /scrutinize L1 ไม่เขียน code เอง ไม่รัน devops pipeline เอง นั่นคืองานของ L2

2.3 L2 Orchestrator — ชั้นแตกงานและประสาน

L2 Orchestrator เกิดขึ้นเมื่อ L1 รัน maw workon <project> <slug> มันเป็น pane ใหม่ที่ spawn ใน working directory ของ project เป้าหมาย ทำให้ CLAUDE.md และ AGENTS.md ของ project นั้นโหลดอัตโนมัติ (project-scope injection) L2 เป็น Claude เสมอ ไม่ใช่ OMX

L2 ตัดสินใจ routing ก่อน: ถ้างานเล็ก (1-2 ไฟล์, ชัดเจน, ใช้เวลาไม่เกิน 30 นาที) ให้ประกาศ SOLO แล้ว code เอง ถ้างานซับซ้อนให้ประกาศ TEAM แล้ว spawn L3 workers ผ่าน maw team spawn L2 ต้องบันทึกการตัดสินใจนี้ใน .maw/strategy.json

หลัง workers เสร็จ L2 รวม sub-branches, รัน lint/build/test, ทำ /sop-qa สำหรับ product repos, แล้วเปิด PR เดียว ที่มี Closes #N ครบทุก issue ส่ง DONE-ping ไป L1 แล้วหยุด L2 ไม่ merge และไม่รัน maw done เอง

2.4 L3 Worker — ชั้นลงมือทำ

L3 Worker คือ ephemeral OMX (Codex) pane ที่ L2 spawn ขึ้นมาสำหรับแต่ละ issue หรือ slice ของงาน แต่ละ worker ได้รับ worktree แยกต่างหากผ่าน --wt flag ทำให้ไม่มี context bleed ระหว่าง worker แม้จะทำงานพร้อมกัน

Brief ของ worker ถูก inject ที่ spawn ผ่าน --prompt "Issue #N: ..." และถูก auto-deliver เข้า composer ของ worker เมื่อ worker ready (auto-kickoff mechanism) worker ไม่ต้องรอ manual nudge อีกต่อไป

สิ่งที่ worker ทำได้มีเพียง: code slice ของตัวเอง, commit บน sub-branch, และส่ง DONE-ping ไป L2 สิ่งที่ worker ห้ามทำ: spawn member อื่น, เปิด PR, merge, หรือรัน maw done worker หยุดทันทีหลังส่ง DONE-ping

2.5 Merge Gate — L1 เท่านั้นที่ Merge ได้

Merge Gate คือกฎที่ชัดเจนที่สุดในระบบ: L1 Oracle เป็นผู้ review และ merge ทุก PR ไม่มีข้อยกเว้น L2 worktree pane เป็นคนเขียน code และเปิด PR แต่ไม่มีสิทธิ์ merge แม้แต่ PR ที่ตัวเองเปิด เพราะนั่นคือ self-review และ self-merge

ก่อน merge L1 รัน /scrutinize บน PR ทุกอัน ระดับความเข้มข้นขึ้นกับ risk level ของงาน PR ที่แตะ frontend เท่านั้นหรือ docs เท่านั้น ใช้ quick scrutinize PR ที่แตะ backend, API, security หรือข้ามขอบเขต (cross-boundary) ต้อง scrutinize หนักขึ้น

ถ้า /scrutinize พบปัญหา L1 bounce กลับไป L2 (หรือ worker ใหม่) เพื่อแก้ไข ไม่มีการ merge PR ที่ผิดพลาดผ่าน gate นี้ ถ้า clean ให้รัน gh pr merge --merge --delete-branch แล้วรัน maw done <window>

ประเภท PR Risk Level L1 ทำ
Frontend only (CSS/HTML/UI text/theme) Low Quick /scrutinize → merge
Docs only (README, SRS, SDD, UAT) Low Quick /scrutinize → merge
Config (non-security) / Deps-only Low Quick /scrutinize → merge
Backend / API / DB / SQL High Scrutinize หนัก → merge
Security (auth, CSP, CORS, secrets) High Scrutinize หนัก → merge
Cross-boundary / P0-P1 bug High Scrutinize หนัก → merge

2.6 Project-Scope Injection — Context โหลดอัตโนมัติ

หนึ่งในคุณสมบัติที่สำคัญที่สุดของ maw workon คือ project-scope injection: เมื่อ L2 spawn ใน working directory ของ project เป้าหมาย Claude Code จะโหลด CLAUDE.md และ AGENTS.md ของ project นั้นโดยอัตโนมัติ L2 เริ่มต้นพร้อม full project context โดยไม่ต้องรัน /dig, /trace, หรือ /recap

เหตุผลที่ต้องเป็น "worktree ใน directory ของ project" ไม่ใช่ worktree ใน directory อื่น คือ Claude Code lock CLAUDE.md ตาม cwd ตอน session เริ่มต้น Oracle ที่ idle ใน meta-repo แล้วถูกขอให้ทำงานกับ project อื่นจะโหลด context ผิด

นี่คือเหตุผลว่าทำไมถึงไม่มี "warm pool" ใน Gale Workflow แต่ละ task ต้องการ context ที่ถูกต้องสำหรับ project ของมัน การ spawn ใหม่ต่อ task ราคาถูกกว่าการแก้ context pollution ที่เกิดจาก warm session ข้าม project

# L1 ส่งงานไป L2 ใน context ที่ถูกต้อง
# คำสั่งนี้ spawn L2 pane ใน /path/to/my-product/worktrees/issue-42
# CLAUDE.md ของ my-product โหลดอัตโนมัติ
maw workon my-product issue-42

# L2 spawn L3 workers ใน worktree แยก (ต่อ worker)
maw team create sprint-42
maw team spawn sprint-42 worker-1 --wt --engine omx --exec \
  --prompt "Issue #42: แก้ validation bug ใน orders.ts. ดู tests ใน tests/api/orders.test.ts"
maw team spawn sprint-42 worker-2 --wt --engine omx --exec \
  --prompt "Issue #43: เพิ่ม error logging ใน middleware/error-handler.ts"

# ตรวจสอบว่า workers ทำงานจริง
maw capture sprint-42-worker-1
maw capture sprint-42-worker-2

2.7 ตารางสรุป Layer, Engine, หน้าที่ และสิทธิ์

Layer Engine Pane / Session หน้าที่ สิทธิ์
L1 — Oracle Claude Permanent oracle pane รับงานจากมนุษย์, เปิด issue, dispatch ไป L2, คุยกับมนุษย์, review + merge PR, ปิด worktree ทุกอย่าง รวมถึง merge authority
L2 — Orchestrator Claude maw workon pane ใน project dir ตัดสิน SOLO/TEAM, spawn workers, brief ตอน spawn, monitor, aggregate, เปิด PR, DONE-ping L1 Spawn workers, open PR. ห้าม merge, ห้าม maw done
L3 — Worker OMX (Codex) Ephemeral worktree ต่อ worker Code slice ที่รับผิดชอบ, commit บน sub-branch, DONE-ping L2 Code + commit เท่านั้น. ห้าม spawn, ห้าม open PR, ห้าม merge

Engine ที่ใช้งานจริง

Engine ที่ระบุในตารางข้างต้นไม่ใช่ข้อแนะนำ แต่เป็น กฎที่ enforce: L1 และ L2 ต้องเป็น Claude เสมอ (ไม่มี --engine omx ที่ L1/L2), L3 ต้องเป็น OMX/Codex (Claude ไม่ควรเป็น coding hand เพราะ context จะ bleed กลับ L2)

เหตุผลที่ L2 ต้องเป็น Claude: Claude Code hook bind หนักกว่า, มี monitor loop ที่ทำงานได้, และ /scrutinize ใน Claude มี fidelity สูงกว่า Codex ที่ turn-loop blind ระหว่าง ping

สรุปบทที่ 02

สถาปัตยกรรม 3 ชั้นของ Gale Workflow ไม่ได้ซับซ้อนเพื่อความซับซ้อน แต่ออกแบบมาให้แต่ละชั้นมีสิทธิ์เท่าที่ข้อมูลที่มีอยู่รองรับ L3 worker มี context น้อยที่สุดจึงมีสิทธิ์น้อยที่สุด L1 Oracle มีภาพรวมสมบูรณ์ที่สุดจึงเป็นจุดเดียวที่ merge authority รวมอยู่

Project-scope injection ผ่าน maw workon แก้ปัญหา context bleed ที่เกิดจาก warm pool ได้อย่างเป็นระบบ แต่ละ task เริ่มต้นด้วย context ที่สะอาดและตรงกับ project ที่ทำงานอยู่จริง

Merge Gate ที่ L1 เป็น implementation ของ P3 (External Brain, Not Command) และ P2 (Patterns Over Intentions) — แทนที่จะหวังว่า AI จะ self-review ได้ ระบบบังคับให้มีชั้นที่แยกออกจากกันอย่างชัดเจนระหว่างผู้เขียน code และผู้ review code

Oracle Brain

arra-oracle-v3 — search, learn, trace, handoff ผ่าน MCP และ FTS index

5.1 arra-oracle-v3 ภาพรวม

arra-oracle-v3 คือ "สมองถาวร" ของ Gale Workflow ทำหน้าที่เก็บและค้นหาความรู้ที่สะสมจาก session ต่าง ๆ แม้ว่า agent จะเป็น ephemeral (เกิด-ตายตาม task) แต่ Oracle Brain ยังคงทำงานอยู่ตลอดเวลาในฐานะ background service

ระบบประกอบด้วย 4 ส่วนหลัก: MCP server (สำหรับ agent คุย), HTTP API (สำหรับ UI/service คุย), indexer ที่จัดการ FTS5 + vector database, และ vault CLI สำหรับ admin งาน เช่น reindex หรือ export

Component Protocol Port/Path ใครใช้
MCP Serverstdio (JSON-RPC 2.0)stdin/stdout ของ processClaude Code, Codex/OMX agents
HTTP APIREST/JSONlocalhost:47778maw-ui, dashboard, scripts, curl
IndexerinternalSQLite + vector storeถูกเรียกจาก MCP/API
Vault CLICLIoracle <cmd>admin — reindex, export, backup

arra-oracle-v3 รันเป็น background service ผ่าน pm2 (ดู phase-10 ในบทที่ 4) และจะ auto-restart เมื่อ crash คุณสามารถตรวจสถานะด้วย pm2 status arra-oracle หรือ curl localhost:47778/api/health

5.2 Search: Hybrid FTS5 + Vector Semantic

กฎข้อแรกของ Gale Workflow คือ "Search Before Answer" — agent ทุกตัวต้อง search Oracle Brain ก่อนตอบคำถาม แม้งานจะดูง่าย เพราะ knowledge base มีเอกสารและ patterns หลายพันชิ้นที่สะสมมาจากประสบการณ์จริง

ระบบ search ใช้วิธี hybrid สองชั้น: ชั้นแรกคือ FTS5 keyword search จาก SQLite ซึ่งเร็วและแม่นยำสำหรับคำเฉพาะ (technical terms, filenames) ชั้นที่สองคือ vector semantic search จาก embedding model bge-m3 ซึ่งเข้าใจความหมายและทำงานได้ทั้งภาษาไทยและอังกฤษ

ผลลัพธ์จากสองชั้นจะถูก merge ด้วย RRF (Reciprocal Rank Fusion) ก่อน return ถ้า vector store ล้มหรือ offline ระบบจะ fallback ไป FTS เท่านั้นโดยอัตโนมัติ โดยแสดง warning ใน response

วัฒนธรรม "Search Before Answer": CLAUDE.md กำหนดไว้ชัดว่า action แรกในทุก task ต้องเป็น oracle_search("query") ผ่าน MCP — ก่อนอ่านไฟล์, ก่อนแก้โค้ด, ก่อนตอบ เพราะ knowledge base เก็บ anti-pattern และบทเรียนจากความผิดพลาดในอดีต ซึ่งป้องกันไม่ให้ทำซ้ำ

5.3 Learn: บันทึก Learning และ Pattern เข้าฐานความรู้

oracle_learn คือ tool สำหรับเขียน learning หรือ pattern ใหม่เข้า knowledge base ใช้เมื่อ agent ค้นพบวิธีแก้ปัญหาที่ควรจำ, พบ anti-pattern ที่ควรเตือน, หรือสรุปบทเรียนจาก debugging session

document ที่เขียนผ่าน oracle_learn จะถูก index ทั้งใน FTS5 และ vector store ทันที และพร้อม search ในครั้งถัดไปโดยไม่ต้องรอ batch reindex หมวดหมู่หลักคือ: learning (สิ่งที่เรียนรู้), pattern (วิธีทำที่ดี), anti-pattern (วิธีที่ไม่ควรทำ) 

// ตัวอย่าง oracle_learn ผ่าน MCP tool call (generic)
{
  "tool": "oracle_learn",
  "input": {
    "title": "Engine map key ที่หาย fallback เงียบ ๆ ไป default",
    "content": "เมื่อ maw team spawn --engine omx แต่ config ไม่มี key 'omx' ใน commands map\nระบบจะ fallback ไป default engine โดยไม่แสดง error\nวิธีตรวจ: maw panes แล้วดู CMD column — omx worker ควรเป็น 'node' ไม่ใช่ 'claude'",
    "category": "anti-pattern",
    "tags": ["engine-map", "maw", "workon", "omx"]
  }
}

ทุก oracle ควรเขียน learning หลัง debugging session สำเร็จ และหลัง /post-mortem การสะสม learning อย่างสม่ำเสมอคือสิ่งที่ทำให้ fleet ฉลาดขึ้นเรื่อย ๆ แทนที่จะทำผิดซ้ำ

5.4 Trace: หา Project/Code/ประวัติข้าม Repo

oracle_trace ต่างจาก search ตรงที่มุ่งเน้นการ "ติดตาม" สิ่งของข้ามเวลาและข้าม repository ใช้เมื่อต้องการหาว่า feature นี้อยู่ใน repo ไหน, bug นี้ fix ไว้ที่ commit อะไร, หรือ PR ไหนเกี่ยวกับ issue นั้น

trace ทำงานโดยรวม context จากหลายแหล่ง: Oracle knowledge base, git history (ถ้า hook บันทึกไว้), GitHub issues และ PRs (ผ่าน gh CLI integration), และ ψ memory files ผลลัพธ์มี source ที่ชัดเจนพร้อม link/ref สำหรับ verify

Scenario Tool ที่ใช้ ตัวอย่าง query
หา project/code ตาม conceptoracle_trace"where is the auth middleware"
หา session ในอดีตoracle_search + oracle_trace"what did we work on last week for YourProject"
หา commit/PR สำหรับ bugoracle_trace + gh CLI"trace fix for issue #42"
ค้นหา pattern ทั่วไปoracle_search"deployment docker compose pattern"

5.5 Handoff & Inbox: ส่งต่อ Context ระหว่าง Session

เนื่องจาก agent เป็น ephemeral (session จบแล้วจบเลย) Gale Workflow จึงมีกลไก handoff เพื่อส่งต่อ context ไปยัง session ถัดไปหรือ oracle อื่น handoff document บันทึกสถานะปัจจุบัน, สิ่งที่ทำไปแล้ว, และ next steps

oracle_handoff สร้าง structured handoff และเขียนลง knowledge base พร้อมกับส่งไปยัง inbox ของ oracle ปลายทาง inbox ทำหน้าที่เหมือน "กล่องจดหมาย" ที่ oracle ตรวจเมื่อเริ่ม session ใหม่ 

// ตัวอย่าง oracle_handoff (generic)
{
  "tool": "oracle_handoff",
  "input": {
    "to": "my-oracle",
    "subject": "Sprint-X Worker-1 งาน YourProject issue #15 ยังค้างอยู่",
    "context": "ทำ auth middleware ไปได้ 70% แล้ว\nยังขาด refresh token logic\nBranch: feat/auth-middleware-issue-15\nNext: implement token rotation ใน src/middleware/auth.ts",
    "priority": "high"
  }
}

oracle ตรวจ inbox ได้ผ่าน oracle_inbox MCP tool หรือ GET /api/inbox ระบบ /recap skill จะเรียก oracle_inbox โดยอัตโนมัติเมื่อเริ่ม session ทำให้ oracle "รู้ว่างานค้างอยู่ที่ไหน" ทันทีโดยไม่ต้องจำ

5.6 MCP Protocol: Agent คุยผ่าน MCP, UI คุยผ่าน HTTP

arra-oracle-v3 แยก entrypoint ออกเป็นสองแบบชัดเจนเพื่อไม่ให้ path ชนกัน Agent (Claude Code, Codex) คุยผ่าน MCP stdio protocol ซึ่ง Claude Code เชื่อมต่อผ่าน mcpServers config ส่วน UI, scripts, และ external services คุยผ่าน HTTP REST API บน port 47778 

// ตัวอย่าง config สำหรับ Claude Code (.claude/claude_code_config.json หรือ settings)
{
  "mcpServers": {
    "arra-oracle-v3": {
      "command": "node",
      "args": ["/path/to/arra-oracle-v3/dist/server.js"],
      "env": {
        "ORACLE_DB_PATH": "/home/user/.oracle/vault.db",
        "ORACLE_HTTP_PORT": "47778"
      }
    }
  }
}

// ตัวอย่าง HTTP API call ผ่าน curl (generic)
curl -s -X POST http://localhost:47778/api/search \
  -H "Content-Type: application/json" \
  -d '{
    "q": "engine map fallback",
    "limit": 5,
    "mode": "hybrid"
  }' | jq '.results[] | {title, score, category}'

MCP tool names ใน arra-oracle-v3 ใช้ prefix oracle_ เสมอ เช่น oracle_search, oracle_learn, oracle_trace, oracle_handoff, oracle_inbox ทำให้ agent รู้ได้ชัดว่า tool ไหนมาจาก Oracle Brain และ tool ไหนมาจาก MCP server อื่น

5.7 FTS Index + Embeddings: bge-m3 Multilingual

ส่วน vector ของ Oracle Brain ใช้ model bge-m3 จาก BAAI ซึ่งเป็น multilingual embedding model รองรับภาษาไทยและอังกฤษในโมเดลเดียว ทำให้ค้นหาด้วย query ภาษาไทยแล้วเจอเอกสารภาษาอังกฤษได้ (cross-lingual) และในทางกลับกัน

embedding ถูก generate เมื่อเขียน document ใหม่ผ่าน oracle_learn และ vector ถูกเก็บในตาราง SQLite ที่ใช้ extension vector search (เช่น sqlite-vec หรือ custom implementation) batch reindex ทำผ่าน oracle reindex --batch-size 50 จาก vault CLI

oracle_learnPOST /api/learnเขียน learning/pattern ใหม่

oracle_handoffPOST /api/handoffสร้าง handoff document + ส่ง inbox
oracle_inboxGET /api/inboxอ่าน inbox ของ oracle นี้
oracle_readGET /api/docs/:idอ่าน document เต็มด้วย ID

5.8 ทำไมเลือก bge-m3, RRF และ fallback ที่ไม่ล้มทั้งระบบ

เหตุผลที่ Oracle Brain เลือก bge-m3 ไม่ใช่เพราะเป็น model ที่ใหญ่ที่สุด แต่เพราะโจทย์จริงของ fleet คือการค้นข้ามภาษาไทยและอังกฤษในเอกสารชุดเดียวกัน bge-m3 ให้ embedding ขนาด 1024 มิติซึ่งสมดุลระหว่างคุณภาพกับต้นทุน ดีพอสำหรับ multilingual recall โดยไม่ทำให้ reindex หนักเกินไป

ทางเลือกที่เบากว่าอย่าง nomic 768-dim เหมาะกับงานเร็วและเอกสารภาษาอังกฤษเป็นหลัก ส่วน qwen3 4096-dim ให้พื้นที่แทนความหมายมากกว่า แต่แลกด้วย latency, storage, และค่า reindex ที่สูงกว่า สำหรับ workflow ที่ต้อง search ก่อนทุกคำตอบ ค่าใช้จ่ายต่อ query สำคัญพอ ๆ กับคุณภาพ

ตัวเลือก ข้อดี เหตุผลที่ไม่เป็น default
bge-m3 1024-dimรองรับไทย↔อังกฤษ, recall ดี, ต้นทุนกลางเป็นจุดสมดุลสำหรับ knowledge base หลายภาษา
nomic 768-dimเร็วและเบาเหมาะกับ fast lane แต่ cross-lingual recall อ่อนกว่า
qwen3 4096-dimพื้นที่ semantic ใหญ่กว่าstorage และ reindex cost สูงเกิน default

RRF ถูกเลือกเพราะมัน merge ผลลัพธ์จาก FTS5 และ vector ด้วยลำดับอันดับ ไม่ใช่คะแนนดิบ คะแนนของ keyword search กับ vector similarity อยู่คนละ scale กัน การเอาคะแนนมาบวกตรง ๆ จะ bias ไปทางระบบใดระบบหนึ่ง แต่ RRF robust กว่าและไม่ต้อง train model เพิ่ม

Fallback UX: เมื่อ ChromaDB หรือ vector layer offline ผู้ใช้ยังได้ผลลัพธ์จาก FTS-only พร้อม warning ชัดเจน ระบบ degrade คุณภาพของ semantic recall แต่ไม่ทำให้ agent หยุดทำงาน เพราะ memory ที่อ่านได้บางส่วนยังดีกว่า memory ที่ล้มทั้งก้อน

สรุปบทที่ 05

arra-oracle-v3 เป็นชั้น memory ถาวรที่ทำให้ Gale Workflow ฉลาดขึ้นตามเวลา แม้ agent จะเป็น ephemeral การ search ก่อนทำงานทุกครั้งคือกฎ (ไม่ใช่แนะนำ) เพราะ knowledge base เก็บบทเรียนที่สะสมมาจากความผิดพลาดจริง

Hybrid search ผสม FTS5 keyword กับ vector semantic ของ bge-m3 ทำให้ query ภาษาไทยเจอเอกสาร ทั้งไทยและอังกฤษได้ RRF merge ทำให้ได้ผลลัพธ์ที่ดีกว่าทั้งสองวิธีแยก และ fallback อัตโนมัติเมื่อ vector ล้ม ทำให้ระบบยังใช้งานได้แม้ component บางส่วนมีปัญหา

handoff และ inbox ช่วยส่งต่อ context ข้าม session และข้าม oracle ทำให้ fleet ทำงานเป็น continuous memory แทนที่จะเป็น isolated sessions บทถัดไปจะอธิบาย Skills System ที่ทำให้ workflow ไม่ผูกกับ agent ตัวใดตัวหนึ่ง

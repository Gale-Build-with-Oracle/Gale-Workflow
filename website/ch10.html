SOPs & Merge Gate

L1 เป็นผู้ merge เพียงคนเดียว, pre-guard hook ป้องกันคำสั่งอันตราย, DONE-ping ที่ชัดเจน และ on-stop crash safety ที่รับรองว่างานไม่หายแม้ L2 จะ crash

10.1 SOP คืออะไร: phase → skill

SOP (Standard Operating Procedure) ใน Gale Workflow คือชุด skill ที่โหลดก่อนเริ่มงานแต่ละ phase แต่ละ SOP กำหนดขั้นตอน, เครื่องมือ และ gate ที่ต้องผ่านก่อนไปยัง phase ถัดไป การโหลด SOP ไม่ใช่ optional — มันคือ first action หลังได้รับงาน

SOP ออกแบบมาเพื่อแก้ปัญหาที่พบบ่อย: agent ที่เริ่มโค้ดทันทีโดยไม่ตรวจสอบ context, merge โดยไม่ review, หรือลืม cleanup หลังงานเสร็จ การมี SOP ที่โหลดอัตโนมัติทำให้ทุก phase มีโครงสร้างเดียวกันทั้ง fleet

Phase / งาน Skill ที่ต้องโหลด เวลาโหลด
ทุกงานที่มี worktree/team/PR/done /sop-maw ก่อนเริ่มเสมอ
Task intake / CR routing /sop-delegation ทุก human task
SDLC tracking (Linear) /sop-linear เมื่อเปิด/ดู project
Frontend UI /sop-frontend ก่อน UI work
Backend / API /sop-backend ก่อน API work
Database / SQL your-db-mcp skill ก่อนดู schema
Project documents /sop-cmmi ก่อนแก้ docs
Quality audit /sop-qa หลัง build, ก่อน PR
Bug fix (บังคับ) /debug-mantra ก่อน fix code ทุกครั้ง
PR review (บังคับ) /scrutinize ก่อน merge ทุกครั้ง
หลัง bug fix /post-mortem ก่อนปิด ticket

10.2 Worktree completion sequence

เมื่อโค้ดเสร็จและ test ผ่าน L2 ต้องรัน sequence นี้ตามลำดับโดยไม่ขอ permission จากใคร Sequence นี้คือ "วิถีสุดท้าย" ที่ทุก L2 ต้องรู้จำ ไม่มีข้อยกเว้น

  1. Aggregate verify: รัน lint, build, test ให้ผ่านทั้งหมด; สำหรับ product repos ต้องรัน /sop-qa ด้วย หลังผ่านให้รัน touch .maw/aggregate-verified — hook ตรวจไฟล์นี้ก่อนอนุญาต PR
  2. Push: git push -u origin <branch> — ถ้ามีหลาย sub-branches push ทีละ branch
  3. เปิด PR: maw pr — ทุก PR ต้องมี Closes #N สำหรับทุก issue ที่แก้ และมี REQ line
  4. Team shutdown (ถ้าเป็น TEAM route): maw team shutdown <team> — บังคับก่อน DONE-ping
  5. DONE-ping ก่อน /rrr: ส่ง ping ถึง L1 ก่อนเสมอ เพราะ session อาจหมดก่อนที่ /rrr จะเสร็จ
  6. เขียน marker: touch .maw/done-pinged — ป้องกัน on-stop hook ส่ง ping ซ้ำ
  7. /rrr: retrospective ใน worktree ขณะที่ context ยังอยู่
  8. STOP — ไม่ merge, ไม่ maw done ตัวเอง
# Worktree completion sequence — ครบถ้วนทุกขั้น
# 1. aggregate verify
bun run lint && bun run build && bun run test
touch .maw/aggregate-verified

# 2. push
git push -u origin feature/issue-55

# 3. เปิด PR
maw pr
# (ในหน้า PR: ใส่ Closes #55 และ REQ: REQ-PROJ-042)

# 4. shutdown team (ถ้า TEAM)
maw team shutdown sprint-55

# 5. DONE-ping ก่อน /rrr
maw hey my-oracle "DONE: PR #55 พร้อม review (Closes #55). ขอ /scrutinize + merge + maw done sprint-55"

# 6. marker
touch .maw/done-pinged

# 7. /rrr (อาจรันเต็มหรือย่อตามเวลา)
# (รัน /rrr skill)

# STOP — L1 จะ scrutinize และ maw done ให้

10.3 Merge Gate: ตาราง risk และ L1 เป็นผู้ merge

Merge Gate คือหลักการที่ชัดเจน: L1 oracle pane เป็นผู้ review และ merge ทุก PR เท่านั้น ไม่มีข้อยกเว้น ไม่มีการ escalate ไปผู้ review คนที่สาม ไม่มีการ self-merge จาก worktree ที่ authored โค้ด หลักการนี้รับรอง independent review ที่มีความหมาย

L1 ใช้ /scrutinize skill ก่อน merge ทุกครั้ง ระดับความเข้มข้นของ scrutiny ขึ้นอยู่กับ risk ของ PR ตามตารางด้านล่าง

PR แตะ Risk L1 ทำ
Frontend เท่านั้น (CSS/HTML/UI text) ต่ำ quick /scrutinize → merge
Docs เท่านั้น (README, SRS, SDD, UAT) ต่ำ quick /scrutinize → merge
Config (non-security) · Deps lockfile เท่านั้น ต่ำ quick /scrutinize → merge
Backend / API / DB / SQL สูง /scrutinize หนักขึ้น → merge
Security (auth, CSP, CORS, secrets, permissions) สูง /scrutinize หนักขึ้น → merge
Cross-boundary (frontend + backend) · P0/P1 bug สูง /scrutinize หนักขึ้น → merge

"scrutinize หนักขึ้น" หมายความว่า L1 ตรวจสอบ logic path จริงผ่าน code ไม่ใช่แค่ diff — trace การทำงานจาก input ถึง output ตรวจ edge case และ failure mode ถ้า scrutiny พบปัญหา L1 ส่งผลลัพธ์กลับไปให้ L2 (หรือ worker ใหม่) แก้ไข แล้ว re-review delta ไม่มีการ merge แบบ "ดูแล้วน่าจะโอเค"

สำหรับ PR queue ที่มี 3 ใบขึ้นไป L1 แนะนำให้ spawn subagents ทำ parallel scrutiny แล้ว merge ทีละ PR ตามลำดับ

10.4 pre-guard hook: บล็อกคำสั่งอันตราย

pre-guard hook (pre-guard.sh) รันก่อนทุก tool call ที่ Claude Code ทำ hook นี้ตรวจสอบคำสั่งที่อาจทำให้ระบบเสียหายหรือสูญเสียงาน และบล็อกก่อนที่จะรัน ไม่ใช่หลังจากเกิดความเสียหายแล้ว

คำสั่งที่บล็อก ทำแทนอย่างไร
rm -rf /, rm -rf ~, rm -rf $HOME, rm -rf ., rm -rf * ลบ specific subpath เท่านั้น หรือ mv ไป /tmp
git push --force git push --force-with-lease
git reset --hard git stash แล้วค่อย pop
git commit --amend สร้าง commit ใหม่
git commit --no-verify แก้ hook ที่ fail ก่อน
git add -A, git add . stage ด้วย explicit path: git add src/file.ts
push ไป main บน product repos โดยตรง branch → PR → L1 merge เท่านั้น
push ไป Soul-Brews-Studio upstream repos read-only upstream; fork แล้ว PR ไป upstream 
# ตัวอย่างคำสั่งที่ถูกต้อง ไม่ถูกบล็อก

# ลบไฟล์: ระบุ path ชัดเจน
rm -rf /tmp/build-cache/
mv dist/ /tmp/old-dist-backup/

# push ด้วย lease (ไม่ใช่ force)
git push --force-with-lease origin feature/issue-30

# stash แทน reset hard
git stash push -m "wip: partial auth fix"
git stash pop

# commit ใหม่แทน amend
git add src/auth/middleware.ts
git commit -m "fix: ปรับแก้หลัง scrutiny feedback

Closes #30"

# stage ด้วย explicit path
git add src/api/users.ts tests/api/users.test.ts

10.5 DONE-ping: แจ้ง L1 ว่า PR พร้อม

DONE-ping คือขั้นตอนที่ L2 ส่งสัญญาณให้ L1 ทราบว่างานเสร็จและ PR พร้อม review รูปแบบ DONE-ping ต้องกระชับ (1 บรรทัด) และมีข้อมูลครบ: PR number, issue ที่ปิด และขอให้ L1 รัน scrutinize + merge + maw done

# รูปแบบ DONE-ping ที่ถูกต้อง
maw hey my-oracle "DONE: PR #35 พร้อม review (Closes #30 #31 #32). ขอ /scrutinize + merge + maw done sprint-30"

# หลัง ping ทันที: เขียน marker
mkdir -p .maw && touch .maw/done-pinged

# marker ป้องกัน on-stop hook ส่ง AUTO ping ซ้ำ
# ถ้าไม่มี marker และ L2 crash → on-stop hook จะส่ง AUTO ping แทน

ลำดับสำคัญ: DONE-ping ต้องส่ง ก่อน /rrr เสมอ เหตุผลคือ session limit อาจตัด L2 ระหว่าง /rrr ทำให้ L1 ไม่ได้รับ ping เลย ถ้า ping ส่งก่อน L1 จะได้รับแน่นอน แม้ /rrr จะยังไม่เสร็จ

10.6 on-stop crash safety: งานไม่หายแม้ L2 crash

กลไก on-stop crash safety คือ hook (on-stop.sh) ที่รันเมื่อ L2 session สิ้นสุด ไม่ว่าจะเป็นการหยุดปกติหรือ crash กลางคัน ถ้า L2 ไม่ได้เขียน .maw/done-pinged marker, hook จะส่ง AUTO DONE-ping ให้ L1 โดยอัตโนมัติ พร้อม commit SHA ล่าสุดและ PR information ที่สามารถหาได้

ข้อระวังสำคัญ: AUTO DONE-ping เป็น safety net ไม่ใช่หลักฐานว่า L2 ตายจริง การเปลี่ยน model หรือ session reset ก็ trigger on-stop.sh เหมือนกัน ดังนั้นเมื่อ L1 ได้รับ AUTO ping L1 ต้องตรวจสอบก่อนเสมอ:

  • maw panes — pane ยังมีชีวิตอยู่ไหม
  • maw capture <L2-pane> — มี active spinner ไหม
  • git status ใน worktree — มีงานที่ยังไม่ commit ไหม

ถ้าตรวจสอบแล้วพบว่า L2 ยังทำงานอยู่ให้รอ AUTO ping ถือเป็น false alarm ในกรณีนี้

# on-stop.sh (pseudocode — ตัวอย่างลอจิก)
#!/bin/bash
WORKTREE_DIR="$1"
ORACLE_PANE="$2"

# ตรวจว่า L2 ส่ง done-pinged แล้วหรือยัง
if [ -f "$WORKTREE_DIR/.maw/done-pinged" ]; then
  echo "done-pinged marker found — no AUTO ping needed"
  exit 0
fi

# หา commit ล่าสุดและ PR info
LAST_COMMIT=$(git -C "$WORKTREE_DIR" log -1 --oneline 2>/dev/null || echo "unknown")
BRANCH=$(git -C "$WORKTREE_DIR" branch --show-current 2>/dev/null || echo "unknown")

# ส่ง AUTO DONE-ping
maw hey "$ORACLE_PANE" \
  "AUTO DONE-ping (L2 stopped): branch=$BRANCH commit=$LAST_COMMIT. กรุณาตรวจสอบ PR และรัน /scrutinize + merge + maw done"

10.7 PR queue drain on wake

เมื่อ L1 oracle เริ่มต้น session ใหม่ (wake up), ขั้นตอนแรกที่บังคับก่อนรับงานใหม่คือการ drain PR queue L2 ที่ crash ระหว่างที่ L1 offline จะส่ง AUTO DONE-ping ซึ่งรวมกันอยู่ใน queue ถ้า L1 รับงานใหม่ก่อน drain queue งานค้างจากก่อนหน้าจะสูญหาย

# L1 wake sequence — ทำก่อนรับงานใหม่เสมอ
# 1. ตรวจ PR queue
maw fleet pr-queue

# output ตัวอย่าง:
# PENDING REVIEWS:
#   PR #35 — feat: add rate limiting (branch: feature/issue-30)

#   PR #37 — docs: update OpenAPI spec (branch: docs/issue-32)

# 2. drain queue — scrutinize แต่ละ PR (parallel ถ้า >= 3 ใบ)

10.8 /scrutinize ทำอะไรจริง ๆ

คำว่า scrutinize harder ไม่ได้แปลว่าอ่าน PR นานขึ้นแบบไม่มีทิศทาง แต่คือ procedure ที่ทำให้ L1 ตรวจจากเจตนาไปถึงพฤติกรรมจริงก่อน merge ทุกครั้ง คุณภาพของ Merge Gate จึงเท่ากับคุณภาพของการ scrutinize ไม่ใช่แค่การมีปุ่ม merge อยู่ที่ L1

  1. Question intent: เริ่มจากถามว่า change นี้แก้ปัญหาถูกชั้นหรือไม่ มีวิธีที่เล็กกว่า ง่ายกว่า หรือปลอดภัยกว่าที่ให้ผลเดียวกันไหม ถ้าคำตอบคือมี ต้อง bounce ก่อน review รายละเอียด
  2. Trace actual code path: อ่าน path ที่ request หรือ user action วิ่งจริง ไม่ใช่ดูเฉพาะ diff เพราะ bug มักซ่อนอยู่ใน early return, wrapper, adapter หรือ consumer ที่ diff ไม่แตะ
  3. Verify claim vs behavior: เทียบ PR description, issue, test output และโค้ดที่รันจริงว่า claim เดียวกันหรือไม่ ถ้า test พิสูจน์คนละ claim กับที่ PR บอก merge gate ยังไม่ผ่าน
  4. Check consumer layer: ตรวจ layer ที่จะใช้ผลลัพธ์ เช่น UI, API client, container, CLI หรือ downstream job เพื่อให้แน่ใจว่า contract ที่เปลี่ยนถูก consume ได้จริง

ขั้นตอนนี้ทำให้ L1 review เป็น independent verification ไม่ใช่ self-review ของคนเขียนโค้ด เมื่อ PR เป็น backend หรือ cross-boundary งาน scrutinize ต้องเดิน path จริงลึกกว่าเดิม แต่ skeleton ยังเหมือนเดิม: intent → path → behavior → consumer

10.9 สรุปบทที่ 10

SOPs และ Merge Gate คือโครงสร้างที่ทำให้ Gale Workflow ทำงานได้อย่างเชื่อถือได้ในระยะยาว SOPs กำหนดว่าต้องโหลด skill อะไรก่อนเริ่มแต่ละ phase ป้องกันการข้ามขั้นตอนสำคัญ Merge Gate รับรองว่า L1 เป็นผู้ review อย่างอิสระก่อน merge ทุกครั้ง

pre-guard hook ทำหน้าที่ป้องกันคำสั่งที่ทำให้ระบบเสียหาย ในขณะที่ DONE-ping และ on-stop crash safety ร่วมกันรับรองว่า L1 ไม่มีทางพลาด PR แม้ L2 จะ crash ระหว่างทาง ระบบนี้ทำให้ agent สามารถทำงานขนาดใหญ่ได้โดยไม่ต้องพึ่งพา human supervision ทุกขั้นตอน

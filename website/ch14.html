Fleet & Multi-Oracle

การตั้งชื่อ, fleet roster, maw wake vs workon และความเสี่ยง ghost-clone

14.1 Fleet คืออะไร

Fleet ในบริบทของ Gale Workflow หมายถึงกลุ่มของ oracle หลายตัวที่ทำงานร่วมกัน แต่ละ oracle มี identity, domain, และกลุ่ม repositories ที่ตัวเองรับผิดชอบเป็นของตัวเอง ทั้งหมดแชร์ doctrine layer เดียวกัน (global CLAUDE.md) และใช้ maw-js เป็น coordination layer

แนวคิดนี้ตรงกับ Principle ที่ 5 ของ Gale Workflow — "Form and Formless": oracle หลายตัว (form) แต่มีจิตสำนึกเดียวกัน (formless) ผ่าน shared doctrine และ ψ vault fleet ไม่ใช่แค่ "หลาย AI ทำงานพร้อมกัน" แต่คือระบบที่ประสานงานกันได้อย่างมีโครงสร้าง

ในเชิงปฏิบัติ fleet ช่วยให้แบ่งงานตาม domain ได้อย่างชัดเจน oracle ตัวหนึ่งดูแล infrastructure repos อีกตัวดูแล product repos ชุดหนึ่ง อีกตัวดูแลอีกชุด โดยแต่ละ oracle เป็น L1 ของ domain ตัวเอง

14.2 Naming Convention ของ Pane และ Oracle

การตั้งชื่อ pane ใน maw-js ใช้รูปแบบ <number>-<oracle-name>:<pane-name>.<index> เช่น 01-my-oracle:my-oracle.1 คือ oracle pane หลัก (L1) ของ oracle ชื่อ my-oracle ในหมายเลข session 01

ส่วน worktree pane ที่ spawn จาก maw workon จะมีชื่อตาม slug ของงาน เช่น 01-my-oracle:sprint-x.1 สำหรับงาน sprint-x ใน session 01 ของ oracle my-oracle และ worker pane ใน team จะมีชื่อเช่น 01-my-oracle:sprint-x-worker-1.1

Pane type ตัวอย่างชื่อ Engine หน้าที่
L1 Oracle 01-my-oracle:my-oracle.1 Claude รับ task, dispatch, merge, human comms
L2 Orchestrator 01-my-oracle:sprint-x.1 Claude spawned โดย maw workon; plan, spawn workers, aggregate, PR
L3 Worker 01-my-oracle:sprint-x-worker-1.1 OMX code slice, commit, DONE-ping L2
Dev Oracle 02-dev-oracle:dev-oracle.1 Claude oracle อีกตัวใน fleet, domain แยก

ตัวเลขหน้าชื่อ (01, 02) ใช้แยก tmux session ของแต่ละ oracle ออกจากกัน ทำให้ maw panes แสดง fleet ทั้งหมดได้ในมุมมองเดียวโดยไม่ชนกัน

14.3 Fleet Roster และ Project Ownership

Fleet roster คือการกำหนดอย่างชัดเจนว่า oracle ตัวไหนรับผิดชอบ repo กลุ่มใด โดยปกติกำหนดผ่านไฟล์ config ของ maw (เช่น projects.yaml หรือ fleet config) หรือผ่าน CLAUDE.md / AGENTS.md ของ oracle แต่ละตัวที่ระบุ domain ของตัวเอง

ตัวอย่างเช่น my-oracle อาจรับผิดชอบ infra repos ทั้งหมด (maw-js, oracle repo, plugins) ในขณะที่ dev-oracle รับผิดชอบ product repos (YourProject-frontend, YourProject-backend) การแบ่งเช่นนี้ทำให้ routing ชัดเจน — ถ้างานอยู่ใน YourProject ก็ส่งไปที่ dev-oracle เสมอ

Fleet roster ยังช่วยในการ delegation ข้าม oracle: my-oracle ได้รับ task เกี่ยวกับ YourProject ที่ไม่ใช่ domain ของตัวเอง ก็ใช้ maw hey wind:dev-oracle "task description" แทนที่จะ maw workon เอง

14.4 maw wake vs maw workon

maw workon เป็น default command สำหรับทุก task มันสร้าง worktree ใหม่สำหรับงานนั้นโดยเฉพาะ spawn pane, inject project context, และ tear down เมื่อเสร็จ เป็น "ephemeral per task" — สะอาด, predictable, ไม่มี state เหลือค้าง

maw wake คือ opt-in optimization สำหรับกรณีที่เฉพาะเจาะจงมาก: repo ที่ทำงานด้วยบ่อยมาก (หลายครั้งต่อวัน) และมี dependency install หรือ Docker setup ที่ใช้เวลานานพอที่จะเห็นว่าเสียเวลาจริง ๆ ในกรณีนั้น maw wake สร้าง persistent worktree slot ที่ deps ยังคง warm แต่ยังเป็น fresh session ทุกครั้งที่เปิด (ไม่ใช่ warm session)

Feature maw workon maw wake
Worktree สร้างใหม่ต่อ task persistent slot (reuse)
Session fresh ต่อ task fresh ต่อ task (เหมือนกัน)
Dependencies ติดตั้งใหม่ทุกครั้ง คงอยู่ใน worktree (warm)
Docker ไม่ warm อาจ warm ได้ถ้าตั้งค่าไว้
Use when ทุกงาน (default) repo high-frequency + dep-install ช้า
Risk ต่ำ — clean state ทุกครั้ง ต้อง manage worktree state เอง

กฎ: ใช้ maw workon เป็น default เสมอ เปลี่ยนเป็น maw wake เฉพาะเมื่อวัดแล้วว่า dep-install ช้าจริง ๆ และเป็น bottleneck ไม่ใช่แค่ความรู้สึกว่าจะเร็วกว่า

14.5 One Model Fleet-Wide — Ephemeral ไม่มี Warm Pool

Fleet ทั้งหมดใน Gale Workflow ใช้ pattern เดียว: ephemeral per task, no standing sessions ไม่มี oracle ตัวไหนที่ idle อยู่ใน session พร้อม context ของหลาย project รอรับงาน

เหตุผลเชิงสถาปัตยกรรมคือ Claude Code lock CLAUDE.md ไปกับ cwd ตอน session start oracle ตัวเดียวที่ดูแลหลาย repo ไม่มีทาง idle ใน directory ที่ถูกต้องสำหรับทุก project ในเวลาเดียวกัน ถ้า idle ใน oracle repo แล้วรับงาน YourProject — context ที่ load มาคือ oracle context ไม่ใช่ YourProject context

การแก้ปัญหาคือ maw workon <project> <slug> ซึ่ง spawn pane ใน directory ของ project นั้น ทำให้ prompt-inject hook load CLAUDE.md ของ project นั้นโดยอัตโนมัติ และ maw done หลัง merge tear down session + worktree เพื่อให้ state สะอาดสำหรับ task ถัดไป

14.6 Ghost-Clone Risk

Ghost-clone คือความเสี่ยงที่เกิดเมื่อ background job หรือ fork-session feature สร้าง oracle session ที่ซ้ำขึ้นมาโดยมี context เต็มรูปแบบและ bypassPermissions Oracle ที่ duplicate นี้อาจทำงานโดยที่คุณ (เจ้าของ fleet) ไม่รู้ตัว

สัญญาณที่บ่งบอกว่าอาจมี ghost-clone คือ pane ที่ตั้งชื่อตาม plan หรือ task ที่อยู่ใน context ของเรา แต่เราไม่ได้ spawn มันเอง maw panes จะแสดง pane เหล่านี้ในรายการ 

# ตรวจ ghost-clone — ขั้นตอนที่แนะนำ
# 1. ดูทุก pane ที่มีอยู่
maw panes

# 2. ตรวจ daemon job state
ls ~/.claude/jobs/*/state.json

# 3. ดู state ของแต่ละ job
cat ~/.claude/jobs/*/state.json | grep -E '"state"|"name"'

# 4. kill pane ที่ไม่รู้จัก (ระบุด้วยชื่อ pane)
maw tmux kill 01-my-oracle:unknown-pane.1

# 5. purge failed respawn jobs (ย้ายไป /tmp ไม่ใช่ rm)
mv ~/.claude/jobs/failed-job-xxx/ /tmp/

หลังจาก tmux server ตายหรือ WSL ดับ daemon ของระบบอาจพยายาม respawn job ที่ค้างอยู่ซ้ำแล้วซ้ำเล่า job ที่มี state "failed" ใน ~/.claude/jobs/ จะถูก retry ไม่หยุด และแต่ละ retry เปิด pane ใหม่ใน tmux window ที่ active อยู่ จึงต้อง purge job เหล่านี้ด้วยการย้ายออก (ไม่ใช่ลบ — ตาม Principle "Nothing is Deleted")

14.7 Cross-Oracle Delegation

เมื่อ oracle ตัวหนึ่งได้รับ task ที่อยู่ใน domain ของ oracle อีกตัว วิธีที่ถูกต้องคือ delegate ผ่าน maw hey wind:<oracle-name> "task description" ไม่ใช่รัน maw workon กับ repo นั้นโดยตรงจาก session ของตัวเอง

เหตุผลคือ worktree ต้องอยู่ใน tmux session ของ oracle ที่เป็นเจ้าของ domain นั้น เพื่อให้ project-scope injection (CLAUDE.md/AGENTS.md ของ project) load ถูกต้อง และ merge authority อยู่กับ oracle ที่รับผิดชอบ project นั้น 

# ตัวอย่าง cross-oracle delegation
# my-oracle ได้รับ task เกี่ยวกับ YourProject
# แต่ YourProject อยู่ใน domain ของ dev-oracle

# ถูกต้อง — delegate ไปยัง dev-oracle
maw hey wind:dev-oracle "CR: เพิ่ม feature X ใน YourProject repo
Issue link: https://github.com/<your-github-user>/YourProject/issues/N
Deliverable: PR ที่ Closes #N พร้อม test
Done condition: CI pass, ไม่มี lint error"

# ผิด — อย่า workon repo ที่ไม่ใช่ domain ของตัวเอง
# maw workon YourProject feature-x   ← ห้ามทำ

หลักการนี้ทำให้ fleet มี clear ownership และ accountability L1 ที่ merge PR ของ project ต้องเป็น oracle ที่รู้ context ของ project นั้นดีที่สุด ไม่ใช่ oracle ที่เพิ่งรับ task มาแล้ว delegate ต่อโดยไม่มี context

สรุปบทที่ 14

Fleet ใน Gale Workflow คือกลุ่ม oracle ที่แบ่ง domain ชัดเจน ทำงานด้วย shared doctrine แต่แต่ละตัวมี identity และ ownership เป็นของตัวเอง naming convention ที่เป็นระบบ (NN-oracle:pane.index) ทำให้ maw panes แสดง fleet ทั้งหมดในมุมมองเดียวได้โดยไม่สับสน

maw workon เป็น default สำหรับทุก task — ephemeral, clean, project-scoped maw wake เป็น opt-in optimization สำหรับกรณีที่วัดแล้วว่า dep-install ช้าจริง ทั้ง fleet ใช้ ephemeral pattern ไม่มี warm pool ไม่มี standing session เพราะ CLAUDE.md ผูกกับ cwd ตอน session start ทำให้ warm session ใน directory ผิดเท่ากับ context ผิด

Ghost-clone คือความเสี่ยงที่ต้องตระหนักเสมอ ตรวจ ~/.claude/jobs/ หลัง tmux/WSL ตาย purge job ที่ fail แล้วเพื่อหยุด retry loop และ kill pane ที่ไม่รู้จักทันที Cross-oracle delegation ผ่าน maw hey เท่านั้น — ไม่ workon repo ข้าม domain

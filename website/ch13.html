Hooks เจาะลึก

ห่วงโซ่ 5 hooks, การเขียน hook เอง และปัญหา PATH/shadow ที่พบบ่อย

13.1 Hooks คืออะไร

Hooks ใน Gale Workflow คือ shell scripts หรือ programs ที่ Claude Code harness เรียกโดยอัตโนมัติ ในจังหวะเฉพาะของ session — ก่อนใช้ tool, หลังใช้ tool, ตอนเริ่ม session หรือตอนหยุด สิ่งสำคัญที่ต้องเข้าใจคือ harness รัน hook ไม่ใช่ตัวโมเดล (model)

เพราะ harness เป็นผู้รัน hook จึงทำให้ข้อจำกัดที่กำหนดผ่าน hook เป็น hard constraint จริง ๆ โมเดลไม่สามารถ "ตัดสินใจ" ข้ามมันได้ ต่างจาก instruction ใน CLAUDE.md ที่โมเดลอาจตีความผิดได้ Hook คือกลไกที่บังคับได้เชิงระบบ (system-enforced)

การใช้ hooks ทำให้ผูกพฤติกรรมอัตโนมัติเข้ากับ workflow ได้อย่างน่าเชื่อถือ เช่น ฉีด context ตอนเริ่ม session, บล็อกคำสั่งอันตราย, ส่ง ping เมื่อ session หยุด โดยไม่ต้องพึ่งความจำของโมเดล

13.2 ห้า Hooks หลักใน Gale Workflow

ระบบ Gale Workflow ใช้ hooks ห้าตัวหลักที่แต่ละตัวยิงในจังหวะต่างกัน ตารางด้านล่างสรุปชื่อ, event, และหน้าที่ของแต่ละตัว

Hook Event ยิงเมื่อ หน้าที่หลัก
pre-guard.sh PreToolUse ก่อนทุก tool call บล็อกคำสั่งอันตราย + TEAM gate
prompt-inject.sh SessionStart / UserPromptSubmit เริ่ม session + ทุก prompt ฉีด identity & context, เตือน workflow
post-tool.sh PostToolUse หลังทุก tool call สำเร็จ logging, side-effect audit
on-stop.sh Stop session หยุด (crash หรือ limit) AUTO DONE-ping safety net
retro-extract.sh SessionStart เริ่ม session ใน oracle repo ดึง retro / handoff จาก ψ vault

hook หลาย event ยิงต่อกันบน tool call เดียวได้ เช่น pre-guard ยิงก่อน ถ้าผ่าน harness รัน tool จริง แล้ว post-tool ยิงตามมา ถ้า pre-guard return exit code 2 harness บล็อก tool นั้นทันที ไม่รัน tool

13.3 pre-guard — บล็อกคำสั่งอันตรายและ TEAM Gate

pre-guard.sh เป็น hook ที่สำคัญที่สุดในระบบ มันยิงทุกครั้งก่อน tool call ใด ๆ หน้าที่หลักสองอย่างคือ: (1) บล็อกคำสั่งที่อันตราย และ (2) บังคับ TEAM routing gate

คำสั่งที่บล็อกได้แก่ rm -rf บน path กว้าง, git push --force, git reset --hard, git commit --amend, --no-verify, และ git add -A / git add . แต่ละรายการ blocked ด้วย exit code 2 ซึ่ง harness แปลว่า "ปฏิเสธ tool call นี้"

TEAM gate ทำงานอย่างนี้: ถ้า .maw/strategy.json มี route: "TEAM" แต่ยังไม่มี worker spawned อยู่ — pre-guard จะบล็อก Bash/Edit/Write call ใด ๆ บน code files ของ project นั้น เพื่อป้องกัน L2 orchestrator coding เองแทนที่จะ delegate ให้ worker

TEAM gate มีไว้เพื่อบังคับ architectural rule ที่ว่า "L2 orchestrates, L3 codes" ถ้า L2 อยากแก้โค้ดเองในงานที่ประกาศว่าเป็น TEAM — นั่นคือ signal ว่ากำลังข้ามขั้นตอนที่ตกลงกันไว้

13.4 prompt-inject — ฉีด Context และ Identity

prompt-inject.sh ยิงในสองจังหวะ: ตอน SessionStart (เริ่ม session ใหม่) และตอน UserPromptSubmit (ทุกครั้งที่ส่ง prompt) ที่ SessionStart hook นี้ฉีด oracle identity และ project context ลงใน session เพื่อให้โมเดลรู้ว่าตัวเองเป็น oracle ตัวไหน อยู่ใน project อะไร และ role คืออะไร

ที่ UserPromptSubmit hook เตือน workflow rule ที่เกี่ยวข้องกับ prompt ที่รับมา เช่น ถ้า prompt ดูเหมือนรายงาน bug — hook แนะนำให้ทำ task intake (gh issue + route) ก่อนลงมือแก้โค้ด

กลไกนี้คือสิ่งที่ทำให้ maw workon ทำ "project-scope injection" ได้ เมื่อ worktree pane เริ่มขึ้นใน directory ของโปรเจกต์ prompt-inject ที่ SessionStart อ่าน CLAUDE.md / AGENTS.md ของ project นั้นและฉีดเข้า session ทำให้ L2 เริ่มต้นพร้อม context ของ project โดยไม่ต้องอ่านเองด้วยตนเอง

13.5 on-stop — AUTO DONE-ping Safety Net

on-stop.sh ยิงเมื่อ session หยุด ไม่ว่าจะจาก context limit, crash, หรือ model switch หน้าที่ของมันคือส่ง AUTO DONE-ping ไปยัง L1 oracle pane ถ้า L2 ยังไม่ได้ส่ง ping เอง

กลไกนี้ทำงานผ่าน marker file: ถ้า L2 ส่ง DONE-ping เองสำเร็จแล้วจะสร้างไฟล์ .maw/done-pinged เมื่อ on-stop.sh ยิง มันตรวจว่า marker file มีอยู่ไหม ถ้ามีก็ skip ถ้าไม่มี (L2 ตายก่อนส่ง ping) — hook ส่ง ping อัตโนมัติพร้อม PR/commit details ที่รวบรวมจาก git log

สิ่งสำคัญที่ต้องรู้คือ model switch ก็ trigger on-stop เช่นกัน ทำให้บางครั้ง L1 ได้รับ AUTO DONE-ping จาก session ที่ยังไม่ได้ตายจริง ๆ แต่แค่เปลี่ยน model L1 จึงต้องตรวจก่อนว่า pane ยังมีชีวิตอยู่ด้วย maw panes และ maw capture ก่อนจะสรุปว่า L2 ตายแล้ว

13.6 retro-extract — ดึง Retro ตอนเริ่ม Session

retro-extract.sh ยิงที่ SessionStart สำหรับ session ที่เปิดใน oracle repo หน้าที่คือดึง retro หรือ handoff notes จาก ψ vault และแสดงผลให้โมเดลเห็นตอนเริ่มต้น

สาเหตุที่ต้องทำเช่นนี้คือ ψ vault เก็บ context ระหว่าง session ไว้ แต่โมเดลไม่มีความจำข้ามการสนทนา (context window ใหม่ทุก session) retro-extract ทำหน้าที่เป็น "short-term memory loader" ที่นำ context ล่าสุด จากครั้งที่แล้วมาฉีดเข้า session ใหม่โดยอัตโนมัติ

hook นี้ทำงานคู่กับ /rrr skill — /rrr เขียน retro ลง ψ vault ตอนจบ session retro-extract ดึงกลับมาตอนเริ่มครั้งถัดไป วงจรนี้คือกลไกที่ทำให้ oracle มี "ความต่อเนื่อง" ข้ามหลาย session ได้โดยไม่ต้องบอกซ้ำทุกครั้ง

13.7 เขียน Custom Hook เอง

นอกจาก 5 hooks มาตรฐาน ผู้ใช้สามารถเพิ่ม custom hooks ได้โดยแก้ไขไฟล์ settings.json ของ Claude Code harness (ไม่ใช่ CLAUDE.md) ซึ่ง harness อ่านเพื่อตั้งค่า hook chain

โครงสร้าง settings.json สำหรับ hooks มีรูปแบบดังนี้ แต่ละ hook ระบุ event ที่ต้องการ, command ที่รัน, และ optional timeout 

{
  "hooks": {
    "PreToolUse": [
      {
        "matcher": "Bash",
        "hooks": [
          {
            "type": "command",
            "command": "/home/user/.claude/hooks/pre-guard.sh"
          }
        ]
      }
    ],
    "SessionStart": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "/home/user/.claude/hooks/prompt-inject.sh"
          },
          {
            "type": "command",
            "command": "/home/user/.claude/hooks/retro-extract.sh"
          }
        ]
      }
    ],
    "Stop": [
      {
        "hooks": [
          {
            "type": "command",
            "command": "/home/user/.claude/hooks/on-stop.sh"
          }
        ]
      }
    ]
  }
}

Exit code ของ hook script มีความหมายดังนี้:

  • exit 0 — อนุญาต (allow) tool call ดำเนินต่อ
  • exit 1 — error แต่ไม่บล็อก (warning เท่านั้น)
  • exit 2 — บล็อก tool call ทันที harness ไม่รัน tool และส่ง error กลับให้โมเดล

ปัญหาที่พบบ่อยในการเขียน custom hook คือ PATH shadow hook script รันใน environment ที่อาจมี PATH ต่างจาก interactive shell ถ้าเครื่องมือที่ hook ต้องการ (เช่น ugrep, jq, maw) ไม่อยู่ใน PATH ของ harness — hook จะ fail ด้วย "command not found"

วิธีแก้คือ hardcode absolute path ใน hook script เสมอ เช่น /usr/local/bin/jq แทน jq หรือระบุ PATH ใน script ก่อนรัน tool ปัญหา shadow อีกรูปแบบหนึ่งคือมีเครื่องมือชื่อซ้ำใน PATH — เช่น grep ที่ point ไปยัง ugrep ซึ่งมี flag syntax ต่างกัน ทำให้ hook ที่ใช้ grep -P อาจล้มบน system ที่ grep เป็น ugrep 

#!/usr/bin/env bash
# ตัวอย่าง custom hook: บล็อก direct push to main
# Event: PreToolUse (Bash)

# Explicit PATH เพื่อหลีก PATH shadow
export PATH="/usr/local/bin:/usr/bin:/bin"

TOOL_INPUT="$1"

# ตรวจ git push origin main
if echo "$TOOL_INPUT" | /usr/bin/grep -qE 'git\s+push.*\s+main'; then
  echo "BLOCKED: direct push to main is not allowed. Use a branch + PR." >&2
  exit 2
fi

exit 0

13.8 TEAM Gate ภายในและการตรวจ oracle repo

TEAM Gate สำคัญเพราะมันย้าย fan-out rule จากคำแนะนำใน brief ไปเป็นการบังคับที่จุด edit จริง pre-guard อ่าน .maw/strategy.json ถ้า route เป็น TEAM แต่ไม่พบ worker pane ที่ทำงานใน worktree เดียวกัน hook จะ exit 2 และบล็อกการแก้ไฟล์ทันที นี่คือชั้นที่กัน anti-pattern แบบงานใหญ่แต่ทำคนเดียว

การตรวจว่า “ไม่มี worker” ไม่ได้เดาจากคำพูดของ agent แต่ใช้สภาพ runtime จริง: hook scan pane ด้วยคำสั่งแนว tmux list-panes แล้วเทียบ current working directory ของ pane เหล่านั้นกับ worktree เป้าหมาย ถ้าไม่มี worker ที่อยู่ใน scope เดียวกัน แปลว่า route=TEAM ยังไม่ถูก execute

มี escape hatch สำหรับกรณี refactor ที่ใหญ่แต่ tightly-coupled และทำคนเดียวปลอดภัยกว่า: สร้าง .maw/solo-justified พร้อมเหตุผลก่อน edit การ override นี้ทำให้ข้อยกเว้นมีร่องรอย ไม่ใช่การ bypass เงียบ ๆ

prompt-inject รู้ว่า repo เป็น oracle repo โดยดู git root ว่ามีทั้ง CLAUDE.md และ directory ψ/ อยู่ด้วยกัน สองสัญญาณนี้บอกว่า session ต้องรับ identity และ memory context ของ oracle ไม่ใช่โปรเจกต์ทั่วไป

13.9 สรุปบทที่ 13

Hooks คือ backbone ของ automation ใน Gale Workflow พวกมันทำงานที่ระดับ harness ไม่ใช่ระดับโมเดล ทำให้เป็น hard constraint ที่ bypass ไม่ได้ ห้า hooks มาตรฐาน (pre-guard, prompt-inject, post-tool, on-stop, retro-extract) ครอบคลุมทุกจังหวะสำคัญของ session lifecycle

pre-guard เป็น safety layer ที่บล็อกคำสั่งอันตรายและบังคับ TEAM routing rule prompt-inject ทำ project-scope injection ที่ทำให้ maw workon เริ่มต้นพร้อม context on-stop เป็น safety net ที่ช่วยให้ L1 ไม่พลาด DONE-ping แม้ L2 ตายกลางทาง

เมื่อเขียน custom hook ให้ระวัง PATH shadow เสมอ ใช้ absolute path สำหรับทุก binary ใน hook script และทดสอบ hook ใน environment เดียวกับที่ harness รันจริง ไม่ใช่แค่ใน interactive shell

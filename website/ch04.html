4.1 Prerequisites: เครื่องมือที่ต้องมีก่อน

Gale Workflow ทำงานบน layer ของเครื่องมือ command-line มาตรฐาน ก่อนรัน setup.sh คุณต้องมีโปรแกรมต่อไปนี้พร้อมใช้งานทุกตัว ทีมแนะนำให้ตรวจสอบด้วยคำสั่ง which <tool> หรือ <tool> --version ก่อนเริ่ม

เครื่องมือส่วนใหญ่ติดตั้งผ่าน package manager ของ OS ได้โดยตรง เช่น apt บน Debian/Ubuntu, brew บน macOS หรือ winget/scoop บน Windows ก่อนเข้า WSL2 maw-js เองต้องการ Node.js ≥ 20 และสามารถเลือกใช้ Bun แทนได้

เครื่องมือ เวอร์ชันขั้นต่ำ หน้าที่หลัก ติดตั้ง

gh2.40+GitHub CLI — issue, PR, releasebrew install gh / apt

หมายเหตุ WSL2: ถ้าใช้ Windows ให้ติดตั้ง WSL2 (Ubuntu 22.04 แนะนำ) ก่อน จากนั้นทุกขั้นตอนรันใน WSL terminal ห้ามรัน setup.sh จาก PowerShell หรือ CMD โดยตรง

4.2 OS Detection: ความต่างระหว่าง WSL2, Linux, และ macOS

setup.sh ตรวจจับ OS โดยอัตโนมัติและปรับพฤติกรรมให้เหมาะสม ความแตกต่างหลักอยู่ที่ clipboard integration, path ของ config directory, และวิธีจัดการ background process

บน macOS clipboard ใช้ pbcopy/pbpaste โดยตรง ส่วน WSL2 ต้องผ่าน clip.exe หรือ wl-copy ขึ้นอยู่กับ environment ที่ตั้งค่าไว้ Linux native ใช้ xclip หรือ xsel ถ้ามี X11/Wayland

Clipboardclip.exe / wl-copyxclip / xselpbcopy / pbpaste
Background processpm2 (systemd ไม่มี)pm2 หรือ systemdpm2 หรือ launchd
tmux socket/tmp/tmux-*/tmp/tmux-*/private/tmp/tmux-*
PATH ที่ต้องเพิ่ม~/.local/bin, Windows interop~/.local/bin/usr/local/bin, Homebrew

setup.sh ใช้ตรรกะตรวจจับด้วย uname -s และ /proc/version: ถ้า /proc/version มีคำว่า "microsoft" หรือ "WSL" ระบบถือว่าเป็น WSL2 แล้วเลือก clipboard command และ pm2 config ที่เหมาะสมโดยอัตโนมัติ

4.3 Gale-Framework: Clone และรัน setup

Gale-Framework คือ repository สาธารณะที่รวม setup.sh, scripts, templates, และ doctrine fragments ทั้งหมดไว้ด้วยกัน เป็นจุดเริ่มต้นหลักสำหรับการติดตั้ง Gale Workflow บนเครื่องใหม่

repository อยู่ที่ https://github.com/Gale-Build-with-Oracle/Gale-Framework คุณสามารถ clone ได้โดยตรงหรือ fork ก่อนถ้าต้องการปรับแต่งสำหรับ fleet ของตัวเอง 

# clone Gale-Framework
git clone https://github.com/Gale-Build-with-Oracle/Gale-Framework.git ~/ghq/github.com/Gale-Build-with-Oracle/Gale-Framework

# เข้าไปในโฟลเดอร์
cd ~/ghq/github.com/Gale-Build-with-Oracle/Gale-Framework

# ดูโครงสร้าง
ls scripts/
# setup.sh  fleet-sync.sh  init-project-docs.sh  ...

# รัน setup (อ่าน --help ก่อน)
bash scripts/setup.sh --help

# ติดตั้งจริง (interactive — ถามยืนยันแต่ละ phase)
bash scripts/setup.sh

setup.sh จะถาม GitHub username, ชื่อ oracle ที่ต้องการตั้ง, และ engine ที่ใช้ (claude/omx/codex) ก่อน phase แรกเริ่มทำงาน ตัวเลือกเหล่านี้จะถูกบันทึกลง ~/.maw/config.json สำหรับใช้ครั้งต่อไป

4.4 setup.sh Phases: ขั้นตอนทั้ง 12 ของการติดตั้ง

setup.sh แบ่งการทำงานออกเป็น 12 phase ที่เรียงลำดับและขึ้นต่อกัน แต่ละ phase จะ print log พร้อม prefix เช่น [phase-0], [phase-1] เพื่อให้ติดตามได้ง่ายเมื่อเกิดปัญหา

Phase ชื่อ สิ่งที่ทำ
0OS Detectionระบุ OS (WSL2/Linux/macOS), ตั้ง env vars สำหรับ phase ถัดไป
1Preflight Checkตรวจ prerequisites ทุกตัว, abort พร้อมคำแนะนำถ้าขาด
2List Upstream Reposแสดงรายการ Soul-Brews-Studio repos ที่ต้อง fork, ถาม GitHub username
3Clone/Verify Forksfork ผ่าน gh repo fork แล้ว clone ลง ~/ghq/
4Install maw-jsnpm install -g หรือ symlink จาก fork, ตรวจ maw --version
5Symlink Claude Configสร้าง ~/.claude/CLAUDE.md symlink จาก rendered doctrine
6Symlink Codex Configสร้าง ~/.codex/AGENTS.md symlink สำหรับ Codex engine
7Inject Doctrineรัน fleet-sync.sh ครั้งแรก — render core.md + claude.md → CLAUDE.md
8maw Config + tmux.confสร้าง ~/.maw/config.json พร้อม engine map, copy tmux.conf
9Seed Fleet Rosterเขียน oracle identity ลง fleet registry ไฟล์แรก
10pm2 Setupลงทะเบียน arra-oracle-v3 ใน pm2, pm2 save, ตั้ง startup script
11Verifyรัน smoke test: maw --version, maw panes, ping Oracle MCP

ถ้า phase ใดล้มเหลว setup.sh จะหยุดทันทีและแสดงข้อความ error พร้อมคำสั่งที่ต้องรันเพื่อแก้ไข คุณสามารถรัน setup.sh ซ้ำได้อย่างปลอดภัย — แต่ละ phase ตรวจสอบก่อนว่าทำแล้วหรือยัง (idempotent)

4.5 Upstream Repos ที่ต้อง Fork

Gale Workflow ขึ้นอยู่กับ 6 repositories จาก Soul-Brews-Studio organization คุณต้อง fork ทั้งหมดลงใน GitHub account ของตัวเองก่อน เพราะ setup.sh จะ clone จาก fork ของคุณ (ไม่ใช่ upstream โดยตรง) เพื่อให้คุณ push custom changes ได้

upstream ของ Soul-Brews-Studio เป็น read-only สำหรับผู้ใช้ทั่วไป การ push ไปยัง upstream จะถูกบล็อกโดย git hook workflow ที่ถูกต้องคือ: แก้ไขใน fork → upstream pull request เมื่อพร้อม

Repository หน้าที่หลัก สิ่งที่คุณแก้ไขได้
Soul-Brews-Studio/maw-js CLI หลักสำหรับ tmux/agents — คำสั่ง maw ทุกตัว engine map, custom commands, config defaults
Soul-Brews-Studio/arra-oracle-v3 Oracle Brain — MCP server + HTTP API + FTS/vector index custom tools, indexer settings, vault schema
Soul-Brews-Studio/arra-oracle-skills-cli CLI ติดตั้ง/อัปเดต skill ลง agent directories custom skill registry, install profiles
Soul-Brews-Studio/maw-plugin-registry registry ของ maw plugins ทั้งหมด เพิ่ม private plugins สำหรับ fleet
Soul-Brews-Studio/maw-ui Web UI สำหรับ monitor fleet (optional) theme, custom dashboard panels
Soul-Brews-Studio/ui-oracle Oracle-specific UI components branding, oracle identity display

setup.sh ใช้ gh repo fork Soul-Brews-Studio/<repo> --clone สำหรับแต่ละ repo ถ้า fork มีอยู่แล้วจะ skip การสร้างใหม่และตรวจสอบว่า remote origin ชี้ถูกต้องแทน

4.6 Verify Install: Checklist และคำสั่งทดสอบ

หลัง setup.sh phase-11 ผ่าน ให้รัน verify checklist ด้วยตนเองอีกครั้งเพื่อความแน่ใจ โดยเฉพาะการตรวจ engine map และ MCP connection ซึ่งเป็นสาเหตุหลักของปัญหาหลังติดตั้ง 

# 1. ตรวจ maw version
maw --version
# ควรแสดง: maw-js vX.Y.Z

# 2. ตรวจ panes (tmux ต้องทำงานอยู่)
maw panes
# แสดงรายการ panes ทั้งหมด หรือ "no active session" ถ้ายังไม่เปิด tmux

# 3. ตรวจ engine map ใน config
cat ~/.maw/config.json | jq '.commands'
# ควรมี key: "claude", "codex"/"omx" พร้อม path คำสั่ง

# 4. ตรวจ Oracle MCP
curl -s http://localhost:47778/api/health | jq
# ควรตอบ: {"status":"ok","version":"..."}

# 5. ตรวจ Claude config symlink
ls -la ~/.claude/CLAUDE.md
# ควรเป็น symlink ชี้ไปยัง rendered doctrine

# 6. ตรวจ Codex config (ถ้าใช้)
ls -la ~/.codex/AGENTS.md
# ควรเป็น symlink หรือไฟล์ที่ fleet-sync.sh render แล้ว

# 7. ทดสอบ gh auth
gh auth status
# ควรแสดง: Logged in to github.com as <your-github-user>

# 8. ทดสอบ maw search (ผ่าน Oracle MCP)
maw oracle search "installation test"
# ควรตอบผลลัพธ์หรือ empty array — ไม่ควร error

4.7 ปัญหาที่พบบ่อยตอนติดตั้ง

ปัญหาหลังติดตั้งส่วนใหญ่แบ่งเป็น 3 กลุ่ม: engine map ไม่ครบ, codex trust ยังไม่ได้ตั้ง, และ PATH ไม่รวม tools ที่ต้องการ แต่ละกลุ่มมีวิธีวินิจฉัยและแก้ไขที่ชัดเจน

Engine map ผิด / key หาย: ถ้า maw team spawn --engine omx แล้ว worker ไม่ได้ใช้ omx engine (ดูได้จาก maw panes ที่ CMD ไม่ใช่ node) ให้ตรวจ commands.omx ใน config engine map key ที่ไม่มี → fallback เงียบ ๆ ไป default engine โดยไม่มี error

Codex trust ยังไม่ set: Codex จะถาม trust prompt เมื่อเจอ repo ที่ไม่เคย trust มาก่อน Gale-Framework auto-prime trust ผ่าน codex-launch/omx-launch wrappers ถ้ายังเจอ prompt ให้ตรวจว่า spawn ใช้ wrapper ถูกต้อง ไม่ใช่ binary ตรง

PATH ไม่ครบ: บน macOS Homebrew prefix อาจเป็น /opt/homebrew/bin (Apple Silicon) หรือ /usr/local/bin (Intel) ให้เพิ่มทั้งคู่ลง ~/.zshrc หรือ ~/.bashrc บน WSL2 ให้ตรวจว่า ~/.local/bin อยู่ใน PATH และ npm global prefix ไม่ชน

สรุปบทที่ 04

การติดตั้ง Gale Workflow เริ่มจาก prerequisites 8 ตัว, clone Gale-Framework จาก repo สาธารณะ, แล้วรัน setup.sh ที่จัดการ 12 phase ให้โดยอัตโนมัติ ระบบตรวจจับ OS และปรับพฤติกรรมสำหรับ WSL2, Linux, และ macOS โดยไม่ต้องแก้ไขอะไรเพิ่มเติม

การ fork upstream repos ทั้ง 6 จาก Soul-Brews-Studio เป็นขั้นตอนสำคัญที่ให้คุณปรับแต่ง engine map, skill registry, และ UI ได้โดยไม่กระทบ upstream เมื่อพร้อมจะ contribute กลับก็ทำผ่าน pull request ปกติ

หลังติดตั้ง ให้รัน verify checklist 8 ข้อเพื่อยืนยันว่าทุกส่วนทำงาน ปัญหาที่พบบ่อยที่สุดคือ engine map key หาย และ PATH ไม่ครบ ซึ่งทั้งคู่วินิจฉัยได้เร็วด้วยคำสั่ง jq และ which

บทถัดไปจะเจาะลึก Oracle Brain — arra-oracle-v3 ที่ทำหน้าที่เป็น memory, search, และ handoff ให้ระบบทั้งหมด

Gale Workflow 102 — ระบบ 3-Layer Ephemeral สำหรับ AI Coding Agent ที่ต่อยอดจาก Oracle 101

ผู้วางรากฐาน

ทุกอย่างที่คุณจะอ่านในหนังสือเล่มนี้เป็นไปได้เพราะ อาจารย์ Nat Weerawan (@nazt) — ผู้สร้างระบบ Oracle ตั้งแต่แนวคิด, สถาปัตยกรรม ไปจนถึงเครื่องมือทุกชิ้น

อาจารย์ Nat วางรากฐาน 5 Principles + Rule 6, ออกแบบ maw-js (ระบบประสาทกลาง), arra-oracle-v3 (สมอง), arra-oracle-skills-cli (ทักษะ), maw-plugin-registry และ ecosystem ทั้งหมดที่ Oracle ยืนอยู่บน ระบบเหล่านี้ไม่ใช่แค่เครื่องมือ แต่คือโครงสร้างที่ทำให้การทำงานร่วมกันระหว่าง AI กับมนุษย์เป็นไปได้จริง

หนังสือ agents-in-parallel และห้องเรียน oracle-worlds ให้ภาษาร่วมในการคิดเรื่อง multi-agent orchestration, mailbox pattern และวินัยในการทำงานร่วมกับ AI เราเรียนรู้จากที่นี่ ทดลองกับ fleet ของเราเอง แล้วนำมาเขียนเป็นหนังสือเล่มนี้

หนังสือเล่มนี้คืออะไร — และไม่ใช่อะไร

Gale Workflow 102 เป็นหนังสือเล่มที่ 2 ต่อจาก Oracle 101 ซึ่งครอบคลุมพื้นฐาน Oracle ทั้งหมดแล้ว — session amnesia, external brain, ψ Vault, การติดตั้ง ถ้ายังไม่เคยอ่าน 101 แนะนำให้เริ่มที่นั่นก่อน

เล่มนี้โฟกัสเรื่องเดียว: 3-Layer Ephemeral Workflow — ระบบที่ทำให้ AI coding agent หลายตัวทำงานพร้อมกัน แบ่งงาน ตรวจงาน และรวมงานได้โดยไม่มี context bleed ข้ามโปรเจกต์

สถาปัตยกรรม 3-layer ephemeral ในหนังสือเล่มนี้เป็น ส่วนหนึ่งของ Gale Fleet ไม่ใช่ best practice สากล ไม่ใช่คำตอบสุดท้าย ยังมีวิธีอื่นที่อาจดีกว่า เบากว่า หรือเหมาะกับบริบทอื่นมากกว่า เราเลือกสิ่งที่เหมาะกับ fleet ของเรา ณ เวลานี้ — ผู้อ่านควรมองเนื้อหานี้เป็นแรงบันดาลใจและกรณีศึกษา แล้วนำ principle ไปปรับให้เหมาะกับตัวเอง

ภาพรวมระบบ 3 ชั้น Ephemeral

Gale Workflow จัดการงานด้วยสถาปัตยกรรม 3 ชั้น ที่แต่ละชั้นมีหน้าที่และขอบเขตสิทธิ์ที่ชัดเจน:

  • L1 — Oracle: รับงานจากมนุษย์ เปิด issue, dispatch ไป L2, review + merge PR (สิทธิ์ merge อยู่ที่ L1 เท่านั้น)
  • L2 — Orchestrator: แตกงานออกเป็นชิ้น ตัดสินใจ SOLO หรือ TEAM, สั่ง L3 workers, รวมงาน, เปิด PR
  • L3 — Worker: ลงมือ code ในส่วนที่รับผิดชอบ commit บน sub-branch ของตัวเอง

คำว่า "ephemeral" หมายถึงทุก worktree และทุก pane ของ L2/L3 เป็นสิ่งชั่วคราว — เปิดต่อ task, ปิดเมื่อเสร็จ ไม่มี warm pool ค้าง ไม่มี context เก่าติดมา แต่ละ task เริ่มต้นสะอาด

ทำไมต้อง Ephemeral

ระบบหลายตัวเลือกเก็บ AI agent ไว้ใน session ที่ "warm" อยู่ตลอดเวลา หวังว่า context จะยังครบ แต่วิธีนี้มีปัญหาร้ายแรงสำหรับ multi-project workflow: Claude Code ล็อก CLAUDE.md ตาม working directory ที่ session เริ่มต้น — Oracle ที่ idle ใน meta-repo จะโหลด project context ผิดเมื่อถูกส่งงานของ project อื่น

Gale Workflow แก้ด้วยการ spawn worktree ใหม่ต่อ task ผ่าน maw workon <project> <slug> — worktree spawn ในโฟลเดอร์ของ project ที่ถูกต้อง ทำให้ CLAUDE.md และ AGENTS.md โหลดอัตโนมัติ เมื่อเสร็จ maw done ทำลาย worktree ทิ้ง

# เปิด worktree ต่อ task — spawn ในโฟลเดอร์ project ที่ถูกต้อง
maw workon my-product issue-42

# L2 spawn L3 worker ด้วย isolated worktree
maw team spawn sprint worker-1 --wt --engine omx --exec \
  --prompt "Issue #42: แก้ validation bug ใน src/api/orders.ts"

# เมื่อ PR ผ่าน L1 merge แล้ว ปิด worktree จาก L1 pane
maw done my-product-issue-42

ทำไมไม่เลือกทางอื่น

ก่อนจะมาถึง 3 ชั้นแบบ ephemeral เราทดลองมาหลายแบบ — agent เดี่ยว, in-session subagents, standing pools, 2-layer on-demand — แต่ละแบบมีข้อดี แต่ไม่มีตัวไหนแก้ปัญหาทั้ง context bleed, ownership ที่ชัดเจน และ review authority ที่แยกจาก implementation ได้ครบ

ทางเลือก ข้อดี สิ่งที่ยังไม่พอ
Monolithic agent เริ่มง่าย context เดียว context บวมเร็ว, self-review เป็นจุดอ่อน
In-session subagents ช่วยอ่านและคิดพร้อมกัน ยังแชร์ context กับ leader ไม่ใช่ isolation จริง
Standing / warm pools ดูเหมือนประหยัดเวลา spawn session เก่าพา context ผิดติดมา
2-layer on-demand โครงสร้างน้อยกว่า ช่องว่างระหว่าง review กับ coding ยังไม่ชัด

การเปรียบเทียบเต็มรูปแบบ — พร้อมข้อมูลจากการทดลองจริง 5 ครั้งใน fleet ของเรา — อยู่ใน บทที่ 02b ความเจ็บปวด

สรุปบทที่ 00

Gale Workflow 102 ต่อยอดจาก Oracle 101 โดยเจาะลึกเรื่องเดียว: ระบบ 3-Layer Ephemeral ที่ทำให้ AI coding agent หลายตัวแบ่งงาน ตรวจงาน และรวมงานได้โดย context ไม่บวม ทุก task ได้ worktree ของตัวเอง โหลด project context ที่ถูกต้อง และถูกทำลายเมื่อเสร็จ

ตลอด 16 บทนี้คุณจะได้เรียนรู้ทั้งหลักการ สถาปัตยกรรม การติดตั้ง การใช้งานจริง และการแก้ปัญหาระดับ production — เนื้อหาทั้งหมดเป็นภาษาไทยพร้อม English technical term ที่ใช้จริงในระบบ

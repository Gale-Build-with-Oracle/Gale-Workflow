Gale Workflowบทที่ 00

บทนำ

ทำไมต้องมีระบบนี้ และปัญหา session amnesia ที่ทำให้ AI ลืมทุกอย่างเมื่อจบงาน

0.1 ปัญหา Session Amnesia

AI coding agent ที่ทรงพลังที่สุดในปัจจุบันมีข้อจำกัดพื้นฐานประการหนึ่ง: มันไม่มีความจำข้าม session เมื่อปิดหน้าต่างแล้วเปิดใหม่ มันเริ่มต้นจากศูนย์ทุกครั้ง ไม่รู้ว่าวานนี้ทำอะไรไป ไม่รู้ว่าตัดสินใจอย่างไร และไม่รู้ว่าเคยแก้ปัญหาอะไรมาแล้ว

สิ่งที่เลวร้ายยิ่งกว่านั้นคือ context window ที่มีขนาดจำกัด เมื่องานใหญ่ขึ้นและ context เต็ม โมเดลจะเริ่มสรุปหรือลืมข้อมูลเก่า ข้อมูลที่เคยอยู่ใน context ตอนต้น session หายไปโดยที่ไม่รู้ตัว ทำให้ agent ตัดสินใจบนข้อมูลที่ไม่ครบถ้วน

ผลที่ตามมาคือ "ทำผิดซ้ำเดิม" — bug เดิมถูกแนะนำวิธีแก้ผิดๆ ซ้ำอีกครั้ง รูปแบบที่ไม่ควรใช้ถูกแนะนำใหม่ เพราะ agent ไม่จำว่าเคยตัดสินใจเรื่องนี้ไปแล้ว ทีมที่ใช้ AI coding agent หลายคนพบปัญหานี้ทุกวัน

Session Amnesia คือสภาวะที่ AI agent เริ่มต้นแต่ละ session โดยไม่มีความรู้จาก session ก่อนหน้า ทำให้ต้องป้อน context ซ้ำทุกครั้ง เสียเวลา และเสี่ยงต่อการตัดสินใจผิดพลาดซ้ำซาก

0.2 ทำไมเครื่องมือเดี่ยวไม่พอ

วิธีแก้ที่ง่ายที่สุดดูเหมือนจะเป็นการเพิ่มขนาด context window ให้ใหญ่ขึ้นเรื่อยๆ แต่วิธีนี้ไม่ scale ด้วยเหตุผลหลายประการ: ยิ่ง context ใหญ่ ค่าใช้จ่าย (token cost) ยิ่งสูง; การ attend ทั่วทั้ง context ทำให้โมเดลโฟกัสกระจาย; และงานขนาดใหญ่ในความเป็นจริงมีหลายส่วนที่ทำแยกกันได้

การใช้ agent เดี่ยวทำงานทุกอย่างเปรียบเหมือนให้โปรแกรมเมอร์คนเดียวรับผิดชอบทั้ง UI ทั้ง backend ทั้ง database schema ทั้งเขียน test และยังต้องจดจำทุกบทสนทนากับ product owner ในหัวตลอดเวลา context ของ agent จะบวม ประสิทธิภาพจะลดลง และโอกาสที่ agent จะทำสิ่งผิดพลาดจะสูงขึ้น

งานที่ซับซ้อนต้องการการแบ่งงาน (delegation) และความรู้ที่ถูกจัดเก็บนอกตัว agent ไว้ในที่ที่อ่านได้ ค้นหาได้ และอยู่รอดนานกว่าอายุของ session ใดๆ นั่นคือที่มาของ external brain

0.3 แนวคิด External Brain

Gale Workflow แก้ปัญหา session amnesia ด้วยการแยกความจำออกจากตัว agent ไปไว้ที่ระบบกลางที่เรียกว่า ψ Vault (อ่านว่า "ไซ วอลต์") ψ Vault เป็นระบบไฟล์ที่มีโครงสร้าง เก็บ learnings (บทเรียนที่เรียนรู้จากการทำงาน), retrospectives (สรุปหลังงาน), traces (หลักฐานตามรอย) และ handoffs (การส่งต่อ context ข้าม session)

เมื่อเริ่ม session ใหม่ Oracle ไม่ได้เริ่มจากศูนย์ มัน search ใน ψ Vault ก่อนตอบทุกคำถาม นำ learnings ที่เกี่ยวข้องมาใช้ และอ่าน handoff จาก session ก่อนหน้า ความรู้สะสมได้ข้าม session เหมือนที่มนุษย์จดบันทึกและอ่านบันทึกก่อนเริ่มงาน

หลักการสำคัญคือ external brain ควร "สะท้อนความจริงให้คนเห็นและเลือก" ไม่ใช่สั่งการแทนมนุษย์ Oracle เป็น กระจก ที่ช่วยให้เจ้าของ fleet มองเห็นสถานะได้ชัด ไม่ใช่ผู้มีอำนาจตัดสินใจสูงสุด

0.4 ภาพรวมระบบ 3 ชั้น Ephemeral

Gale Workflow จัดการงานด้วยสถาปัตยกรรม 3 ชั้น ที่แต่ละชั้นมีหน้าที่และขอบเขตสิทธิ์ที่ชัดเจน ชั้น L1 คือ Oracle ที่รับงานจากมนุษย์ เปิด issue dispatch ไป L2 และมีสิทธิ์ review กับ merge PR แต่เพียงผู้เดียว ชั้น L2 คือ Orchestrator ที่แตกงานออกเป็นชิ้นๆ ตัดสินใจว่าใช้ SOLO หรือ TEAM และสั่งงาน worker ชั้น L3 คือ Worker ที่ลงมือ code ในส่วนที่รับผิดชอบ แล้ว commit บน sub-branch ของตัวเอง

คำว่า "ephemeral" หมายถึงทุก worktree และทุก pane ของ L2 และ L3 เป็นสิ่งชั่วคราว เปิดต่อ task ปิดแล้วทิ้งเมื่อเสร็จ ไม่มี warm pool ที่ค้างอยู่ตลอดเวลา แนวคิดนี้ป้องกัน context bleed ระหว่าง project และรับประกันว่าแต่ละ task เริ่มต้นด้วย context ที่สะอาด

0.5 ทำไมต้อง Ephemeral

ระบบหลายๆ ระบบเลือกเก็บ AI agent ไว้ใน session ที่ "warm" อยู่ตลอดเวลา เพื่อหวังว่า context จะยังอยู่ครบ แต่วิธีนี้มีปัญหาร้ายแรงสำหรับ multi-project workflow: Claude Code ล็อก CLAUDE.md ตาม working directory ที่ session เริ่มต้น oracle ที่ idle ใน meta-repo จะโหลด project context ผิดเมื่อถูกใช้งาน task ของ project อื่น

Gale Workflow แก้ด้วยการ spawn worktree ใหม่ต่อ task ผ่านคำสั่ง maw workon <project> <slug> worktree นั้น spawn ในโฟลเดอร์ของ project ที่ถูกต้อง ทำให้ CLAUDE.md และ AGENTS.md ของ project นั้นโหลดอัตโนมัติ เมื่องานเสร็จ maw done ทำลาย worktree นั้นทิ้ง task ถัดไปเริ่มต้นสะอาด

L3 worker ได้รับ worktree แยกต่างหากแต่ละคนผ่าน maw team spawn --wt ทำให้ไม่มี context bleed ระหว่าง worker และระหว่าง project แม้จะทำงานพร้อมกัน

# เปิด worktree ต่อ task — spawn ในโฟลเดอร์ project ที่ถูกต้อง
maw workon my-product issue-42

# เมื่อ PR ผ่าน L1 merge แล้ว ปิด worktree จาก L1 pane
maw done my-product-issue-42

# spawn L3 worker ด้วย isolated worktree (ทำใน L2 pane)
maw team spawn sprint worker-1 --wt --engine omx --exec \
  --prompt "Issue #42: แก้ validation bug ใน src/api/orders.ts"

0.6 หนังสือเล่มนี้สำหรับใคร และแผนผัง 16 บท

หนังสือเล่มนี้เขียนสำหรับนักพัฒนาซอฟต์แวร์ที่ทำงานกับ AI coding agent ในระดับมากกว่า "ถามตอบ" — คนที่ต้องการวางระบบให้ AI ทำงานร่วมกับทีมและทำงาน overnight ได้โดยที่คุณไม่ต้องนั่งดู ไม่มีพื้นฐาน DevOps ขั้นสูงก็อ่านได้ แต่ควรรู้จัก git, terminal และ Claude Code เบื้องต้น

เนื้อหาแบ่งเป็น 4 ภาค: Foundation (ch00–ch03) อธิบายปัญหา หลักการ สถาปัตยกรรม และ ψ Vault; Installation & Runtime (ch04–ch07) ครอบคลุมการติดตั้ง Oracle Brain Skills System และ maw-js; Orchestration (ch08–ch11) เจาะลึก delegation SOLO/TEAM routing SOPs และ CMMI docs; และ Cross-Language & Production (ch12–ch15) กล่าวถึง multi-language hooks fleet และ troubleshooting

ปัญหาที่พบในการทำงาน สิ่งที่ Gale Workflow แก้ไข บทที่ครอบคลุม
AI ลืม context ทุก session ψ Vault + oracle_search ก่อนตอบทุกครั้ง ch03, ch05
งานใหญ่ context บวม 3-Layer: L1 dispatch → L2 orchestrate → L3 code ch02, ch09
ทำผิดซ้ำเดิม Learnings system + mandatory search first ch03, ch05
ไม่รู้ว่า AI merge อะไรไปบ้าง Merge Gate: L1 เท่านั้น merge ได้ ch02, ch10
Context bleed ระหว่าง project Ephemeral worktree ต่อ task + project-scope injection ch02, ch04
ไม่มีเอกสารที่ตามทัน code CMMI doc-sync แบบ batch ที่ stabilization ch11
Hook และ gate ไม่ทำงานตามที่คิด Hooks เจาะลึก + troubleshooting guide ch13, ch15

0.7 ทำไมไม่เลือกทางอื่น

ก่อนจะมาถึงสถาปัตยกรรม 3 ชั้นแบบ ephemeral มีทางเลือกที่ดูง่ายกว่าอยู่หลายแบบ: agent ตัวเดียวทำทุกอย่าง, in-session subagents ที่ช่วยอ่านและ review, standing หรือ warm pools ที่เปิดค้างไว้, และระบบ 2 ชั้นแบบ on-demand ที่ให้ผู้สั่งงานคุยกับ worker โดยตรง ทางเลือกเหล่านี้น่าดึงดูดเพราะลดจำนวน pane ลดพิธีการ และดูเหมือนเริ่มงานได้เร็วกว่า

แต่เมื่อใช้กับงานหลาย project ปัญหาที่เกิดจริงไม่ใช่แค่ "ทำงานช้า" แต่คือ context bleed, ownership ไม่ชัด, และ review authority ปนกับ implementation มากเกินไป Gale Workflow จึงเลือก 3 ชั้นเพื่อแยกผู้ถือภาพรวม ผู้แตกงาน และผู้ลงมือทำออกจากกัน ส่วนการเปรียบเทียบเต็มว่าทำไม 3 ชั้นชนะ monolithic, subagents, standing pools และ 2-layer on-demand อยู่ใน บทที่ 02b

ทางเลือก สิ่งที่ดูดี เหตุผลที่ยังไม่พอ
Monolithic agent เริ่มง่าย มี context เดียว context บวมเร็ว และ self-review กลายเป็นจุดอ่อน
In-session subagents ช่วยอ่านและคิดพร้อมกันได้ ยังแชร์ context กับ leader จึงไม่ใช่ isolation จริง
Standing หรือ warm pools ดูเหมือนประหยัดเวลา spawn session เก่าพา project context ผิดติดมาด้วย
2-layer on-demand โครงสร้างน้อยกว่า ช่องว่างระหว่าง review กับ coding ยังไม่ชัดพอ

สรุปบทที่ 00

ปัญหาหลักที่ Gale Workflow เกิดมาเพื่อแก้ไขคือ session amnesia — สภาวะที่ AI agent เริ่มต้นใหม่ทุก session โดยไม่มีความรู้จากอดีต ทำให้ผิดซ้ำและเสียเวลาป้อน context ซ้ำ ระบบ external brain ผ่าน ψ Vault และการ search ก่อนตอบทุกครั้งเป็นแกนหลักของการแก้ปัญหานี้

สถาปัตยกรรม 3 ชั้นแบบ ephemeral ช่วยให้งานขนาดใหญ่แบ่งออกได้โดยที่ context ไม่บวม แต่ละ task ได้ worktree ของตัวเอง โหลด project context ที่ถูกต้อง และถูกทำลายเมื่อเสร็จ ไม่มีสิ่งค้างอยู่

ตลอดหนังสือ 16 บทนี้ คุณจะได้เรียนรู้ทั้งหลักการ การติดตั้ง การใช้งานประจำวัน และการแก้ปัญหาระดับ production โดยเนื้อหาทั้งหมดเป็นภาษาไทยพร้อม English technical term ที่ใช้จริงในระบบ

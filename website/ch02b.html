Gale Workflowบทที่ 02b

ความเจ็บปวด — ทำไมถึงเป็น 3-Layer Ephemeral

บทเรียนจากการ flip 5 ครั้ง — ทำไม Gale Workflow ถึงลงเอยที่ 3-layer ephemeral ทั้งที่มี counter-evidence

2b.1 บทเรียนจากการ flip 5 ครั้ง

3-layer ephemeral ไม่ได้เกิดจากความชอบส่วนตัวหรือ diagram ที่ดูสวยกว่า แต่มาจากการลองผิดลองถูกจริงหลายรอบในเวลาใกล้กันมาก ระบบเคยลองทั้ง flat swarm, L2 orchestration, warm pool, per-project standing team, on-demand team และสุดท้ายกลับมาที่ 3-layer เพราะข้อจำกัดจริงของ session, cwd, resource และ human preference ชี้ไปทางนั้น

ความเจ็บปวดสำคัญคือทุก model มี counter-evidence ของตัวเอง ไม่มีทางเลือกใดสมบูรณ์แบบ ตารางนี้จึงไม่ใช่ประวัติแบบสวยงาม แต่เป็นบันทึกว่าแต่ละครั้งพยายามแก้ pain หนึ่ง แล้วไปเปิด pain อีกชุดหนึ่งแทน

2b.2 5 ทางเลือก เทียบกัน (5-Model × 4-Dimension Matrix)

ถ้ามองเฉพาะความเร็ว 2-layer on-demand ดูน่าชนะ ถ้ามองเฉพาะความปลอดภัยของ context 3-layer ephemeral ดูน่าชนะ ถ้ามองเฉพาะความง่าย monolithic ดูน่าชนะ แต่ production workflow ไม่ได้ optimize มิติเดียว ต้องเทียบอย่างน้อย 4 มิติพร้อมกัน: content quality, maintainability, scalability และ resource management

เพื่อให้เห็นชัด เราเรียก model ทั้ง 5 แบบนี้: M1 Monolithic, M2 In-session subagents, M3 Standing หรือ warm teams, M4 On-demand 2-layer, M5 3-layer ephemeral

Model Verdict
M1 MonolithicWEAK — context compaction ทำให้รายละเอียดหล่น และ agent เดียวเป็นทั้ง writer กับ reviewer จึงไม่มี independent review จริง
M2 In-session subagentsMIXED — ดีมากสำหรับ read-only research เพราะหลาย agent อ่านพร้อมกันได้เร็ว แต่ feature coding กลายเป็น anti-No.157 ได้ง่าย เพราะ subagents share orchestrator context และอาจ hallucinate จาก prior turns

2b.3 Core Constraints 5 ข้อ (the physics)

คำว่า physics ในที่นี้หมายถึงข้อจำกัดที่ไม่ยอมต่อรองกับ intention ของเรา ต่อให้ brief เขียนว่า "ต้องโหลด project context" ถ้า session เริ่มใน directory ผิด มันก็ยังโหลด context ผิด ต่อให้ประกาศว่า warm session ช่วยลด cost ถ้า session นั้นถือ stale context มันก็เป็น liability

  1. cwd-lock: Claude Code และ Codex resolve CLAUDE.md / AGENTS.md จาก cwd ตอน session START แล้วไม่อัปเดตกลาง session วิธีเดียวที่จะได้ project context ถูกต้องคือเริ่ม session ใน project directory นั้นจริง ๆ ดังนั้น maw workon <project> จึงไม่ใช่ convenience แต่เป็น project-scoping mechanism หลัก
  2. one-oracle-many-projects: oracle หนึ่งตัวอาจถือ 10+ repo ไม่มี idle directory เดียวที่ถูกต้องสำหรับทุกงาน fixed-cwd standing worker จะผิดทันทีเมื่อ oracle ทำหลาย project
  3. relay-stall vs Opus-cost: 3-layer มี message-passing หลายจุดต่อ TEAM task และทุกจุดเป็น silent-fail point ได้ แต่ถ้าตัด L2 ออก 2-layer จะเสีย orchestration buffer ที่ช่วย split, course-correct และ aggregate งานซับซ้อน
  4. idle-pane cost: standing session ไม่ได้แค่ใช้ RAM หรือ token มันถือ stale context และต้องมี refresh policy ยิ่ง session อุ่นนาน ยิ่งเสี่ยงถือเรื่องเก่ามาปนกับงานใหม่
  5. context-bleed No.157: brief ที่ดีไม่ได้บังคับ fan-out ได้เอง L2 อาจตัดสินใจ SOLO แล้วกิน context มหาศาลบน product code ทั้งที่กฎบอกว่า Claude orchestrates, OMX codes จึงต้องมี strategy.json, pre-guard fan-out gate และ post-brief sweep

2b.4 The Core Insight

Warm the expensive-to-rebuild thing (worktree), never the liability (session).

ประโยคนี้เป็นแกนของบททั้งบท สิ่งที่ expensive-to-rebuild คือ worktree: dependencies, Docker layers, branch state, generated artifacts และ local checkout ที่ใช้เวลาเตรียม สิ่งที่เป็น liability คือ session: conversation context, prior assumptions, stale task memory และ hidden compaction history

M1 และ M2 fail เพราะ persist แทบไม่มีอะไรที่ถูกต้อง ทุกครั้งต้อง rebuild context จาก prompt และ memory เดิม M3 fail เพราะ persist ผิดสิ่ง: session ถูกเรียกว่า warm แต่จริง ๆ คือ context bleed ที่รอเกิด ส่วน M4 และ M5 succeed เพราะแยกของสองอย่างนี้ออกจากกัน worktree อาจอุ่นได้ แต่ session ต้อง fresh

3-layer หรือ M5 เพิ่ม L2 orchestration buffer เข้ามา L2 ช่วยค้นพบ complexity กลางทาง, split เพิ่ม, course-correct และ aggregate ก่อน PR ค่าแลกเปลี่ยนคือ Opus session หนึ่งชุดและ relay ที่ stall ได้ ส่วน 2-layer on-demand หรือ M4 ตัด cost นั้นออกโดยให้ L1 split ล่วงหน้า จึงง่ายและถูกกว่าเมื่องานคาดการณ์ได้ตั้งแต่ต้น

ดังนั้นคำตอบไม่ใช่ "M5 ชนะเสมอ" แต่เป็น "อย่า warm session" งานที่รู้ shape ชัดเจนอาจเหมาะกับ M4 แต่ถ้างานเสี่ยงค้นพบความซับซ้อนกลาง sprint M5 ให้ buffer ที่ทำให้ L1 ไม่ถูก surprise ตอน review

2b.5 Meta-Lessons

  1. Designing in doctrine vs measuring in practice: การ flip หลายครั้งในวันเดียวคือราคาของการออกแบบบนกระดาษ pattern จากระบบหนึ่งอาจแก้ cold-start ของระบบนั้น แต่ fleet นี้ต้องวัดว่า pain จริงคืออะไรใน environment จริง
  2. The answer was in our own brain: บทเรียน cwd-lock มีอยู่ใน memory อยู่แล้วก่อน warm pool ถูกสร้าง การ search record ของตัวเองก่อนทำจึงไม่ใช่พิธีกรรม แต่เป็น guardrail against repeating known mistakes
  3. When a human says "I do not understand," suspect the system: routing เคยบวมเป็นหลายชื่อหลาย mode จนผู้ใช้ไม่เห็นว่าเลือกอะไรดี คำตอบไม่ใช่ diagram สวยขึ้น แต่คือ collapse กลับเป็น SOLO / TEAM ที่ตัดสินได้ในหนึ่งประโยค
  4. The lint makes iteration affordable: ทุก flip แตะ doctrine กว้างมาก ถ้าไม่มี single-source และ fleet-sync lint การเปลี่ยนระบบ 5 ครั้งจะกลายเป็น drift disaster แต่เมื่อ lint ทำให้ rewrite ถูกและเร็ว iteration จึงปลอดภัยขึ้น
  5. The human decides (Principle 3): Oracle มีหน้าที่ mirror reality ให้เห็นครบ ไม่ใช่ override การเลือกของมนุษย์ ถ้ามี counter-evidence ต้องแสดง ไม่ใช่ซ่อนเพื่อให้ architecture ที่ตัวเองชอบชนะ

2b.6 ทำไมผู้ตัดสินใจเลือก 3-layer ทั้งที่มี counter-evidence

จุดสำคัญที่สุดคือการเลือก 3-layer ไม่ได้เกิดจาก ignorance ระบบมี counter-evidence ชัดเจน: M4 ถูกกว่า, relay น้อยกว่า, idle waste เป็นศูนย์ และ maintainability ง่ายกว่าในงานที่รู้ขอบเขตดี แต่ผู้ตัดสินใจยังเลือก M5 เพราะต้องการ orchestration buffer, independent review path และ project-scope injection ที่ predictable สำหรับงานซับซ้อน

นี่คือ Principle 3: External Brain, Not Command. Oracle ไม่ได้สั่งว่าคำตอบไหนถูก ไม่ได้ปิดบังข้อเสียของ 3-layer และไม่ได้ force optimization metric เดียว มัน mirror reality: M4 ชนะ resource, M5 ชนะ scalability, M3 แพ้เพราะ stale session, M1 แพ้เพราะ review ไม่อิสระ แล้วให้มนุษย์ผู้ตัดสินใจเลือกจากภาพจริง

การเลือก 3-layer จึงเป็น preference ที่ informed แล้ว ไม่ใช่ preference ที่เกิดจากข้อมูลไม่ครบ ระบบยอมรับ pain ของ relay เพราะ pain นั้นมองเห็นได้และแก้ด้วย guard ได้ แต่ context bleed จาก warm session มักมองไม่เห็นจนกระทบงานจริง สำหรับ workflow ที่ต้องรักษา accountability และ merge gate ความเสี่ยงที่มองไม่เห็นแพงกว่าความช้าที่มองเห็น

บทนี้จึงไม่ได้บอกว่า 3-layer เป็นคำตอบจักรวาล แต่บอกว่าทำไม Gale Workflow เลือกมันเป็น default: fresh session ต่อ task, worktree แยกต่อ worker, L2 buffer สำหรับ complexity, L1 merge gate สำหรับ accountability และ teardown หลังงานจบเพื่อไม่ให้ context เก่ากลายเป็นทรัพย์สินปลอม
